## Strategy: Determinism First

> "Determinism first — remove all `test.skip(true, …)`, fix the 19
> data-loading race failures, then add new coverage on top of a stable
> base."  
> — operator, 2026-05-10

Adding new tests against an unstable suite hides regressions: a new
test failing for a real reason looks like just-another-flake. v2.9
starts by reducing the suite to a hard pass/fail signal:

1. **Remove `test.skip(true, …)` modifiers across the suite.** Phase
   64 cleared `voter-results.spec.ts`; the rest of the suite has not
   been swept. Existing todo:
   `.planning/todos/pending/2026-04-27-remove-e2e-skip-modifiers.md`
   has the inventory + classification protocol.
2. **Resolve the 19 known data-loading race failures.** Captured in
   PROJECT.md "Future" as a long-standing tech-debt item; needs
   investigation per failure type (race in initial data fetch?
   subscription not flushed before assertion? auth-cookie not set in
   time?) and a deterministic fix per case.
3. **Sweep the 98 pre-existing `playwright/*` warnings in `tests/`.**
   `no-conditional-in-test`, `no-raw-locators`, `no-networkidle` —
   captured in
   `.planning/todos/pending/2026-05-10-tests-playwright-hygiene-sweep.md`
   from the v2.8 OOS triage. Fixing these forces the rewrite of the
   conditional / `networkidle` patterns that drive flakiness.

Only after (1)+(2)+(3) land does v2.9 add the **new coverage** below.

## New Coverage to Add

Each gap below maps to a candidate v2.9 plan or set of plans. Strategy
hints in italics.

### Translation flow (multilocale candidate UX)

- [ ] **Multilingual translation surface in candidate questions.** When
  there are multiple locales configured, candidates can add
  translations to questions where localization is *not* disabled. When
  there is only 1 locale, the translation surface is hidden.
  *Strategy: parameterized variant template — multilocale fixture +
  single-locale fixture; assert via aria-label / role on the
  translation tab/dialog.*

### Browse-without-match results

- [ ] **Browse results when located but not enough answers for
  matching.** Voters who haven't answered enough opinion questions but
  have completed location should still be able to browse entities.
  Results list should not show match scores; intro text should not
  reference matches. *Strategy: variant template with low
  `minimumAnswers`; voter completes location + skips opinion questions;
  navigate to results — assert match score absent + alternative intro.*

### Feedback dialog persistence

- [ ] **Feedback typed text is remembered across dismiss; reset on
  send.** *Strategy: open feedback → type text → close → reopen —
  expect text retained. Type text → send — expect text reset on next
  open.*

### Election + constituency selector matrix

- [ ] **Always bypass selectors when election + constituency can be
  implied (not just 1+1).** *Strategy: variant matrix — 1e×1c (already
  covered), 1e×Nc, Ne×1c, Ne×Nc, plus startFromConstituency setups.
  Assert URL and selector visibility per cell.*
- [ ] **Constituency selector shows minimal options + disallows
  conflicts.** *Strategy: same matrix; specifically assert that the
  const dropdown options for selected election filter to the
  constituencies of that election only.*

### Voter answer in entity details

- [ ] **Voter's answer rendered next to entity's answer.** All four
  cases: agree, disagree, voter missing, entity missing, both missing.
  *Strategy: parameterized fixture with each case; voter-detail spec
  asserts both rows present + appropriate visual state.*

### Skip / delete / back navigation

- [ ] **Voter skips a question, deletes their answer, navigates back
  with predictable result-availability.** Specifically: deleting an
  answer that brings count below `minimumAnswers` should hide the
  results CTA. *Strategy: build a sequence test — answer N questions →
  delete one → assert results-CTA disabled; re-answer → assert
  re-enabled.*

### Match SubMatches by category

- [ ] **Per-category match breakdown.** Voter-detail tests verify
  info+opinions tabs; the per-category match-by-subdimensional-pillar
  rendering is uncovered. *Strategy: extend voter-detail.spec.ts with
  a per-category assertion block; reuse existing fixture's category
  metadata.*

### Locale switching

- [ ] **Voter switches locale and verifies translated UI.** *Strategy:
  visit page in en → assert key strings → switch to fi → assert
  translated key strings → switch back. Tests both the route-prefixed
  form (`/fi/...`) and the locale-switcher widget if any.* (i18n is a
  v1.2 milestone capability; no E2E covers this end-to-end yet.)

### Opinion question rendering — focused user stories

- [ ] **Boolean question rendering (voter flow).** v2.6 added
  BooleanQuestion voter-flow rendering; the smoke was Phase 61
  verification (single-shot). No permanent E2E gate.
- [ ] **Categorical (single/multi-choice) question rendering.** Covered
  indirectly by matching tests; no focused user-story spec.

*Strategy: small dedicated specs that walk through one of each
question-type, assert input-shape correct, voter answers, navigates,
sees their answer reflected. Deduplicate against matching tests.*

### Accessibility sweep

- [ ] **WCAG 2.1 AA via axe / keyboard / screen-reader smoke.** The
  project commits to AA but no automated a11y E2E sweep is wired into
  the suite. *Strategy: integrate `@axe-core/playwright`. Wire as a
  smoke against the home + selector + questions + results + voter-
  detail routes initially. Cite-and-fix any WCAG violations as
  separate plans.*

### Profile (candidate)

- [ ] **Profile field validation rejection paths.** Happy paths are
  covered; rejection paths (invalid email format, name length, image
  type/size) are uncovered. *Strategy: parameterize candidate-profile
  spec with bad-input cells; assert validation error UI + unsaved
  state.*
- [ ] **Profile field persistence beyond the image.** CAND-12 covers
  image + answers + comment text reload-persistence; other profile
  fields (name, bio, social links — note that these depend on the info
  questions) are not covered by reload-persistence E2E. *Strategy:
  extend CAND-12 to cover the full profile-field set per fixture.*

### Question visibility / answer-required logic

- [ ] **Per-question visibility flags + "must answer" enforcement.**
  hideHero is covered; per-question visibility flags or "must answer"
  enforcement is not. *Strategy: variant fixture with mixed
  visibility/required flags; assert hidden questions don't render and
  required-but-unanswered blocks navigation.*

### Question custom-data

- [ ] **`customData.allowOpen`** — flagged as a known gap in v2.0
  milestone notes; still uncovered. *Strategy: variant fixture
  enabling allowOpen on a subset of questions; assert open-comment UI
  surfaces and persists.*

### appCustomization toggle matrix

- [ ] **Individual `appSettings` / `appCustomization` toggles.** The
  candidate-settings.spec.ts covers app-mode + notifications +
  hideHero + help/privacy. The matrix of all toggles is far larger.
  *Strategy: enumerate toggles + assertion-per-toggle; consider a
  per-toggle dedicated spec or a parameterized matrix-driven spec.*
  Likely paired with the
  `2026-04-27-extend-e2e-filter-type-coverage.md` todo.

### Skip-modifier & flake debt (determinism prerequisites)

- [ ] **`test.skip(true, …)` audit.** Phase 64 cleared
  voter-results.spec.ts; the rest of the suite is untouched. Captured
  in `.planning/todos/pending/2026-04-27-remove-e2e-skip-modifiers.md`.
- [ ] **Test rerun stability / 19 known data-loading race failures.**
  Captured in PROJECT.md Future as "resolve 19 pre-existing data-
  loading race E2E failures."

These two are the **determinism prerequisites** above; called out here
so the gap inventory is exhaustive.

## Suggested v2.9 phase structure (operator to confirm via /gsd-new-milestone)

The "determinism first, coverage on top" plan suggests this shape:

| Phase (proposed) | Scope | Why this order |
|------------------|-------|----------------|
| Phase A — Determinism baseline | Sweep `test.skip(true, …)`; resolve 19 data-loading races; sweep 98 playwright/* warnings. Goal: 100% green / 0 skipped on the existing suite. | Without this, every new test cell adds noise rather than signal. |
| Phase B — High-leverage coverage | Add specs for: multilocale translation surface, browse-without-match, full election/const matrix, voter answer in entity details, skip/delete/back, locale switching. | Highest UX impact + spans most user flows. |
| Phase C — Question-rendering specs | Boolean + categorical focused specs; per-category match breakdown. | Targeted; can run in parallel with B. |
| Phase D — Profile + a11y | Profile validation + persistence + WCAG axe sweep. | Self-contained; depends on neither B nor C. |
| Phase E — Settings matrix | appSettings / appCustomization toggle coverage. | Parameterized; benefits from determinism work in A. |
| Phase F — Cleanup todos | DX rename (`dev:* → db:*`), unlocated-voter redirect, the post-71 carry-forward 3-pack. Same shape as v2.7 Phase 68 / v2.8 Phase 72 "Hygiene Trio". | Closes residual debt before v3.0 starts. |

This is a starting point, not a commitment — `/gsd-new-milestone` will
refine and resequence based on dependency analysis.
