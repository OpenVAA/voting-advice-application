# Tighten i18n wrapper

`apps/frontend/src/lib/i18n/wrapper.ts` exports a `t()` function that:

1. Accepts `key: string` — should accept `key: TranslationKey` (the auto-generated union type at `apps/frontend/src/lib/types/generated/translationKey.ts`) so callers get compile-time validation that the key exists. Today, missing keys silently fall through to `return key`, which is what surfaced the v2.8 Phase 69 Plan 01 Task 7 fix-up bug (a key showing as raw text in the running app).

2. Defines `t.get = t` as a non-reactive convenience alias. Since Paraglide messages are already non-reactive, `t.get` is functionally identical to `t`. If no consumer relies on the alias, the assignment is dead code — should be removed. If consumers exist, evaluate whether they can switch to plain `t()`.

## What to do

1. **Type the key parameter:** change `t(key: string, params?: Record<string, unknown>): string` → `t(key: TranslationKey, params?: Record<string, unknown>): string`. The `TranslationKey` is auto-generated by `apps/frontend/tools/translationKey/generateTranslationKeyType.ts`. Any callsites passing dynamic strings (e.g., the existing `t(\`entityDetails.tabs.\${tab}\`)` pattern in EntityDetails.svelte:82) need either:
   - A stricter typed pre-check (`tab as 'children' | 'info' | 'opinions'`), or
   - An explicit cast at the boundary, or
   - A documented `as TranslationKey` escape hatch with reason

2. **Audit `t.get` consumers:**
   ```
   git grep -nE "\\bt\\.get\\b" apps/frontend/
   ```
   If zero matches: delete `t.get = t`. If matches: rewrite each to plain `t()` (or, if a non-reactive guarantee is wanted at the type level, keep but document why it differs from `t`).

3. **Update tests:** `apps/frontend/src/lib/i18n/tests/translations.test.ts` already validates locale key parity but does not type-check `t()` callsites. Optionally add a small TS-level test that asserts `t('nonexistent.key')` is a type error.

## Acceptance

- `t()` signature uses `TranslationKey` (or stricter); compile-time validation catches missing keys.
- `t.get = t` either deleted or documented inline (reason for keeping).
- `yarn workspace @openvaa/frontend check` baseline does not regress beyond v2.7-close (160 err / 12 warn).
- All `t.get` callsites either rewritten or, if zero remain, the alias removed.
- No runtime regressions; `yarn test:unit` + Playwright parity baseline continue to pass.

## Why now (low priority)

The smoke-bug surfaced during v2.8 Phase 69 Plan 01 Task 7 (`results.alliance.summary` leaked as raw key path) would have been caught at compile time if `t()` were typed against `TranslationKey`. Fixing this prevents future "wrong i18n key" regressions from sneaking into production. Low priority because:

- The bug class is rare (Phase 69's was specifically a wrong-target-directory + Paraglide-compile-bug compound).
- The current `apps/frontend/src/lib/types/generated/translationKey.ts` regen step is already part of the i18n workflow.
- Minor refactor; no behavioral change.

Suggested phase placement: a v2.9+ frontend-hygiene-followup phase, OR fold into the broader v2.8 Phase 71 strict-typing cleanup if scope permits (Phase 71's `no-explicit-any` sweep includes `apps/frontend/src/lib/api/utils/auth/` and the wrapper layer).
