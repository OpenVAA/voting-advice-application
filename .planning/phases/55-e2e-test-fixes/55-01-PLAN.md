    **For each failing test from Task 1's results, follow this diagnosis loop:**

    1. **Identify root failures** — look at the FIRST failure in each Playwright project dependency chain. "Did not run" tests are cascade failures; fix the root cause upstream.

    2. **Trace to app code** — The fix should almost always be in application code (components, layouts, contexts), NOT in test code. Common regression sources post-migration:
       - `$app/state` page object used without `$` prefix (correct) vs old `$page` auto-subscription (broken)
       - `$state` mutations not propagating through `$derived` chains in layout components
       - `$effect` timing differences vs old `$:` reactive declarations (async vs sync)
       - Context initialization order changes after root layout migration
       - `getXxxContext()` calls failing because `initXxxContext()` timing changed

    3. **Fix in app code** — Preserve the Svelte 5 rune patterns established in Phases 50-54. Do NOT revert to Svelte 4 patterns.

    4. **Re-run affected project only** using `npx playwright test -c tests/playwright.config.ts --project=<name>` to verify the fix works before running the full suite.

    **Known likely issues to check (from RESEARCH.md):**

    - **voter-detail party tab test**: If `activeEntityType` reactivity doesn't propagate after clicking the organizations/parties tab, check the results `+layout.svelte` file. The `handleEntityTabChange` function should mutate a `$state` variable, and the template should read it reactively. If the party section (`voter-results-party-section` test-id) doesn't become visible, the `$state` -> template binding is broken.

    - **results-sections candidates-only/organizations-only**: If after `updateAppSettings` + `reload`, the settings don't take effect, check that `entityTypes` in the voter context is a `$derived` that correctly reads from `$state`-backed `appSettings`. The `reload()` should re-initialize the context with fresh server settings.

    - **voter-settings showResultsLink hidden**: If the results link is still visible after setting `showResultsLink: false` + reload, check the `questions/+layout.svelte` file. The `topBarSettings.push()` call should read `appSettings.questions.showResultsLink` at initialization time, and after reload the component reinitializes with fresh settings.

    **For timing-related failures:**
    - If a test fails intermittently or with timeout errors, add explicit waits (`await expect(...).toBeVisible({ timeout: 10000 })`) in the TEST code rather than making app code synchronous.
    - Use `page.waitForSelector` for elements that appear after async `$effect` runs.

    **IMPORTANT: Do NOT modify test logic.** Only:
    - Fix app code (components, layouts, contexts) for reactivity regressions
    - Add/increase timeouts in test code for timing-sensitive assertions
    - Add explicit `waitFor` calls in test code if `$effect` async timing causes flakiness
