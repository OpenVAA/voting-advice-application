**# Matching Algorithm Reference** (title)

**## Paradigm Overview**
Point Claude to the authoritative description:
- "For the full paradigm, process description, and design principles, read `packages/matching/README.md`."
- Briefly state: Treat voter and candidate as positions in N-dimensional space. Measure distance. Normalize to [0, COORDINATE.Extent]. Return sorted Match objects.
- Note the optional projection step for 2D political compass via `MatchingSpaceProjector`.

**## Matching Pipeline** (~25 lines)
Numbered step-by-step of what `MatchingAlgorithm.match()` does internally. Reference the actual methods for each step:

1. Validate inputs: no duplicate question IDs, non-empty questions and targets arrays
2. Filter questions to those the reference (voter) has answered (non-MISSING_VALUE, non-undefined)
3. Call `projectToNormalizedSpace()`:
   a. Create `MatchingSpace` via `MatchingSpace.fromQuestions()` with shape from `normalizedDimensions` and optional `questionWeights`
   b. For each entity, call `question.normalizeValue(answer.value)` for each question
   c. Create `Position` objects with the normalized coordinates in the space
4. Optionally apply `this.projector.project(positions)` for lower-dimensional projection
5. Separate reference position from target positions
6. Create subspaces via `createSubspace()` if `questionGroups` are provided
7. For each target, call `measureDistance()`:
   a. `imputeMissingPosition()` imputes missing coordinates in target based on reference
   b. Metric function computes distance in global space and all subspaces
8. Create `Match<TTarget>` objects with distance and optional `SubMatch` array
9. Sort by ascending distance (best match first)

**## Distance Metrics** (~40 lines)
For each of the three metrics, document:
- The `kernel`, `sum`, and `subdimWeight` functions used
- The semantic meaning
- When to use it

| Metric | Kernel | Sum | SubdimWeight | Semantics |
|--------|--------|-----|--------------|-----------|
| Manhattan | `absoluteKernel`: `\|a - b\|` | `basicSum`: `Σ values` | `basicDivision`: `1/n` | Standard proximity; most common in VAAs |
| Directional | `directionalKernel`: `0.5*Ext - 2*(a-N)*(b-N)/Ext` | `basicSum`: `Σ values` | `basicDivision`: `1/n` | Treats neutral as uncertain; from Mendez 2017 |
| Euclidean | `absoluteKernel`: `\|a - b\|` | `euclideanSum`: `√(Σ v²)` | `euclideanSubdimWeight`: `1/√n` | Geometric distance; penalizes large single-dimension differences |

Then describe the generic `distance()` function pipeline:
1. Flatten positions (subdimensions become flat arrays)
2. Compute weights: `space.weights[i] * subdimWeight(shape[i])` for subdimensions, just `weights[i]` for single dimensions
3. For each dimension: if either coordinate is missing and `allowMissing`, skip; else compute `weight * kernel(a, b)`
4. Sum distances and maximum weights via `sum()`
5. Normalize: `COORDINATE.Extent * sum(distances) / sum(maxima)`
6. If all dimensions are missing, return `COORDINATE.Extent / 2`

**## MatchingSpace and Position** (~25 lines)
Document:
- `MatchingSpace`: defined by `shape` (Array<number> of subdimension counts) and `weights` (Array<number> for each dimension)
- `MatchingSpace.fromQuestions()`: shape[i] = question.normalizedDimensions ?? 1; weights[i] = questionWeights[question.id] ?? 1
- `MatchingSpace.isCompatible()`: checks shape equality (weights may differ)
- `Position`: holds `coordinates` (PositionCoordinates) and a reference to its `space`
- `PositionCoordinates`: `Array<CoordinateOrMissing | Array<CoordinateOrMissing>>` -- one level of nesting for subdimensions
- `Shape`: `Array<number>` -- each element is the subdimension count for that dimension
- `flatten()`: converts nested coordinates to flat array
- `reshape()`: reverses flatten with a target shape
- `createSubspace()`: creates a MatchingSpace where excluded questions have weight 0

**## Missing Value Handling** (~20 lines)
Document:
- `MISSING_VALUE_METHOD.Neutral`: impute `COORDINATE.Neutral` (0)
- `MISSING_VALUE_METHOD.RelativeMaximum`: impute the coordinate furthest from the reference value
  - reference < Neutral -> impute Max
  - reference > Neutral -> impute Min
  - reference == Neutral -> use `MISSING_VALUE_BIAS` (Positive -> Max, Negative -> Min)
- `imputeMissingValue()`: single coordinate imputation
- `imputeMissingPosition()`: iterates over flattened coordinates; imputes target missing from reference; if reference also missing, imputes Neutral
- Creates new Position -- NEVER mutates original
- `allowMissingReference` in `DistanceMeasurementOptions`: if true, reference missing coordinates are skipped in distance calculation

**## Match Objects** (~25 lines)
Document the class hierarchy and conversions:
- `MatchBase` (abstract-ish base):
  - `distance: NormalizedDistance` (0 = perfect, COORDINATE.Extent = worst)
  - `matchFraction: number` = `(COORDINATE.Extent - distance) / COORDINATE.Extent` (0 = worst, 1 = best)
  - `score: number` = `Math.round(matchFraction * MatchBase.multiplier)` (0 = worst, 100 = best by default)
  - `toString()` = `score + MatchBase.unitString` (e.g., "85%")
  - Static `multiplier` (default 100) and `unitString` (default "%") are configurable
- `Match<TTarget, TGroup>`:
  - `readonly target: TTarget` -- reference to the matched entity
  - `readonly subMatches?: Array<SubMatch<TGroup>>` -- per-category submatches
  - `readonly matchType = MATCH_TYPE.Match`
- `SubMatch<TGroup>`:
  - `readonly questionGroup: TGroup` -- reference to the question group
  - `readonly matchType = MATCH_TYPE.SubMatch`
- `MATCH_TYPE`: `{ Match: 'match', MatchBase: 'matchBase', SubMatch: 'subMatch' }`

**## Question Types (in matching package)** (~20 lines)
These are example implementations; `@openvaa/data` provides the production versions:
- `OrdinalQuestion`: Likert-scale. `normalizedDimensions = 1`. Maps answer ID to choice value, normalizes via `normalizeCoordinate()`. Factory: `OrdinalQuestion.fromLikert({ id, scale })`.
- `CategoricalQuestion`: Non-orderable choices. Binary (N=2): `normalizedDimensions = 1`, first choice = Min, second = Max. N>2: `normalizedDimensions = N`, selected = Max, others = Min. Max disagreement = `2/n`.
- `MatchableQuestionGroup`: `{ questions: Array<MatchableQuestion> }`. Used for SubMatch computation via `createSubspace()`.

**# Extension Patterns for @openvaa/matching** (title)

**## Adding a New Distance Metric** (~50 lines)
Step-by-step guide listing every file that must be created or modified. Use Manhattan as the reference implementation. Number each step. For each step specify: the file path, what to do, and what to look at for the pattern.

Steps (all paths relative to `packages/matching/src/`):

1. **Implement the metric function** in `distance/metric.ts`
   - Write a function matching the `MetricFunction` signature: `({ a: Position, b: Position, space?: MatchingSpace, allowMissing?: boolean }) => NormalizedDistance`
   - Choose or write: a `kernel` function `(a: Coordinate, b: Coordinate) => number`, a `sum` function `(values: Array<number>) => number`, and a `subdimWeight` function `(numDimensions: number) => number`
   - Call the generic `distance()` function with these three components
   - Pattern: copy `manhattanDistance()` and replace kernel/sum/subdimWeight as needed
   - For a hybrid metric (e.g., Manhattan-Directional): compute two distances and combine

2. **Add to DISTANCE_METRIC const** in `distance/metric.ts`
   - Add `{Name}: {name}Distance` to the `DISTANCE_METRIC` const object
   - The `DistanceMetric` type auto-derives from the const

3. **Export from module barrel** `distance/index.ts`
   - The metric function is already exported from `metric.ts`, which is re-exported via `distance/index.ts`
   - If you created a separate file (not recommended for simple metrics), add it to `distance/index.ts`

4. **Write tests** `tests/distance.test.ts`
   - Add to the `metric: distance` describe block
   - Test extreme distance (min vs max = COORDINATE.Extent)
   - Test zero distance (same position = 0)
   - Test with subdimensions (shape like [1, 1, 1, 3, 1]) -- verify subdimWeight correctness
   - Test with custom weights
   - Test commutativity: `metric({ a, b }) === metric({ b: a, a: b })`
   - Pattern: see existing `manhattanDistance` and `euclideanDistance` tests

5. **Verify** all existing tests still pass: `cd packages/matching && yarn test:unit`

**## Adding a New Question Type (for Matching)** (~60 lines)

### Scenario A: In @openvaa/matching (example implementation)

   - For multi-dimension questions: `normalizedDimensions = N`, return `Array<CoordinateOrMissing>` of length N
   - Handle missing values: if `value == null`, return `MISSING_VALUE` (or array of MISSING_VALUE)

**## Adding SubMatch Support** (~20 lines)
SubMatches provide per-category match scores. They use zero-weight masking (not question filtering) via `createSubspace()`.

To use SubMatches:
1. Define `MatchableQuestionGroup` objects: `{ questions: Array<MatchableQuestion> }`
2. Pass as `options.questionGroups` to `algorithm.match()`
3. Each Match result will contain `subMatches: Array<SubMatch>` with one SubMatch per group
4. SubMatch uses the same MatchBase base class -- access `distance`, `score`, `matchFraction`
5. SubMatch's `questionGroup` property references back to the group definition

Implementation detail: `createSubspace()` creates a new MatchingSpace with the same shape but weights set to 0 for excluded questions. This reuses the existing distance measurement infrastructure without recomputing positions.

**## Implementing a MatchingSpaceProjector** (~15 lines)
For projecting to lower-dimensional spaces (e.g., 2D political compass):

1. Implement the `MatchingSpaceProjector` interface in `algorithms/matchingSpaceProjector.ts`:
   - `project(positions: ReadonlyArray<Position>): Array<Position>`
   - Input: positions in the original N-dimensional space
   - Output: positions in the target lower-dimensional space
2. Pass the projector to `MatchingAlgorithm` constructor via `projector` option
3. The projector runs after `projectToNormalizedSpace()` and before `measureDistance()`
4. Note: no concrete implementation exists yet. The interface is 3 lines. A typical implementation would use a weight matrix to map dimensions.

**## Verification After Extension** (~15 lines)
After completing any extension, verify:
1. `cd packages/matching && yarn test:unit` -- all existing tests pass
2. New test file passes with expected distance values
3. For new metrics: test extreme distances equal `COORDINATE.Extent`, zero distances equal `0`
4. For new question types: test `normalizedDimensions` matches `normalizeValue()` array length
5. For multi-dimension questions: verify weight compensation is documented if `max disagreement < 1`
6. If metric is added to `DISTANCE_METRIC`, verify it participates in the `test.each(metrics)` tests
7. `yarn build:shared` -- builds without errors (important if `@openvaa/core` interfaces changed)
