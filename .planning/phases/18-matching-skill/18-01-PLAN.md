**## Conventions** (~40 lines)
Write each convention as an imperative rule (DO/NEVER/ALWAYS). Include these 8 conventions:

1. **Coordinate range**: All normalized values use `COORDINATE` from `@openvaa/core`: `Min=-0.5`, `Max=0.5`, `Neutral=0`, `Extent=1`. Distances range `[0, COORDINATE.Extent]`. Convert to percentage: `score = Math.round((1 - distance / COORDINATE.Extent) * 100)`. Use `normalizeCoordinate()` from `@openvaa/core` for value-to-coordinate mapping.

2. **MatchingSpace from questions**: ALWAYS create `MatchingSpace` via `MatchingSpace.fromQuestions({ questions, questionWeights })`. The factory reads `normalizedDimensions` from each question to build the shape array and applies optional weights. NEVER manually construct shape arrays from question lists.

3. **normalizedDimensions consistency**: The `normalizedDimensions` property of a `MatchableQuestion` MUST match the length of the array returned by `normalizeValue()`. If `normalizeValue()` returns a single `CoordinateOrMissing`, set `normalizedDimensions` to `1` or `undefined`. Mismatch causes "shape incompatible" errors.

4. **Missing value asymmetry**: Reference (voter) missing values cause the question to be SKIPPED from matching. Target (candidate) missing values are IMPUTED via `MISSING_VALUE_METHOD.Neutral` (impute 0) or `.RelativeMaximum` (impute furthest from reference). The `imputeMissingPosition()` function creates a NEW Position -- NEVER mutate the original.

5. **Distance metric decomposition**: All metrics use the `distance()` function with three pluggable components: `kernel(a, b)` for per-coordinate distance, `sum(values)` for aggregation, `subdimWeight(n)` for subdimension weight factor. When adding a new metric, implement all three and add to `DISTANCE_METRIC` const in `distance/metric.ts`.

6. **Type guards over instanceof**: Use `isMatch()`, `isSubMatch()`, `isMatchBase()`, `isMatchType()` from `utils/typeGuards.ts`. NEVER use `instanceof Match` or `instanceof SubMatch`. Uses `MATCH_TYPE` discriminator (same pattern as `@openvaa/data`).

7. **Blob re-exports**: The matching package uses `export *` in module-level `index.ts` files, re-exported from `src/index.ts`. No `internal.ts` barrel needed (package is small, no circular dependencies). When adding files, export from the relevant module `index.ts`.

8. **Test location**: Tests live in `packages/matching/tests/` (NOT co-located with source). Shared test utilities are in `tests/utils.ts`. Import source via `../src/` paths.

**## Mathematical Nuances** (~25 lines)
Document these three mathematical subtleties with brief explanations:

1. **CategoricalQuestion subdimension model**: For N>2 choices, creates N subdimensions. Selected choice gets `COORDINATE.Max`, others get `COORDINATE.Min`. Maximum disagreement is `2/n` (not 1), because non-selected choices "agree on absence." Binary questions (N=2) use 1 dimension with full [0,1] range. Weight compensation: multiply question weight by `n/2` to correct relative weighting vs ordinal questions. Note: weight compensation corrects inter-question weighting but does NOT expand isolated categorical range beyond `2/n`.

2. **Directional distance semantics**: Formula adapted from Mendez (2017): `0.5 * Extent - (2 * (a - Neutral) * (b - Neutral)) / Extent`. Treats neutral as "uncertain" -- neutral-neutral pairs yield 50% distance (not 0%). Neutral with any other value also yields 50%. Only use when "uncertain neutral" semantics are desired. Manhattan is standard for typical VAA matching.

3. **Euclidean subdimension weighting**: Euclidean metric uses `1/sqrt(n)` as subdimWeight (not `1/n` like Manhattan). This ensures maximum possible distance is equal for spaces with different subdimension shapes (e.g., shape [1,1] and shape [1,3]).
