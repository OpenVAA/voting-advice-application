<objective>
Convert all test dataset JSON files from Strapi format to Supabase format. The SupabaseAdminClient's `bulkImport` → `bulk_import` RPC expects specific field naming, M:N relationship syntax, and project scoping that differs from Strapi's import-data endpoint.
</objective>

<tasks>

<task id="02.1">
<title>Convert default-dataset.json to Supabase format</title>
<read_first>
- `tests/tests/data/default-dataset.json` (current dataset)
- `git show feat-gsd-supabase-migration:tests/tests/data/default-dataset.json` (target format)
- `tests/tests/utils/supabaseAdminClient.ts` (after Plan 01 creates it — understand field mapping)
</read_first>
<action>
Update `tests/tests/data/default-dataset.json` with these changes:

1. **Remove `questionTypes` array entirely** — question types are seed data in Supabase, not imported via bulk_import.

2. **Add to ALL entity records** (constituencies, constituencyGroups, elections, questionCategories, questions, candidates, nominations):
   - `"projectId": "00000000-0000-0000-0000-000000000001"`
   - `"published": true`

3. **In `questionCategories`:**
   - Rename `"type"` to `"categoryType"` (e.g., `"categoryType": "opinion"`)
   - Rename `"elections"` to `"_elections"` (M:N join table syntax for linkJoinTables)
   - Remove `customData.video` URLs pointing to LocalStack (`http://localhost:4566/static.openvaa.org/...`) — replace with empty object or remove the customData.video section entirely (video assets will be handled separately if needed)

4. **In `constituencyGroups`:**
   - Rename `"constituencies"` to `"_constituencies"` (M:N join table syntax)

5. **In `elections`:**
   - Rename `"constituencyGroups"` to `"_constituencyGroups"` (M:N join table syntax)

6. **In `questions`:**
   - Remove `"questionType": { "externalId": "test-qt-..." }` reference (question type settings are denormalized)
   - Add inline `"settings"` field with the question type settings. For Likert-5 questions, add:
     ```json
     "settings": {
       "type": "singleChoiceOrdinal",
       "choices": [
         {"id": "1", "key": "1", "normalizableValue": "1", "label": {"en": "Fully disagree"}},
         {"id": "2", "key": "2", "normalizableValue": "2", "label": {"en": "Somewhat disagree"}},
         {"id": "3", "key": "3", "normalizableValue": "3", "label": {"en": "Neutral"}},
         {"id": "4", "key": "4", "normalizableValue": "4", "label": {"en": "Somewhat agree"}},
         {"id": "5", "key": "5", "normalizableValue": "5", "label": {"en": "Fully agree"}}
       ]
     }
     ```
   - For date questions: `"settings": {"type": "date"}`
   - For number questions: `"settings": {"type": "number"}`
   - For text questions: `"settings": {"type": "text"}`
   - For boolean questions: `"settings": {"type": "boolean"}`

7. **In `nominations`:**
   - Rename `"party"` to `"organization"` (e.g., `"organization": {"externalId": "test-party-a"}`)

8. **In `parties` (top-level key):**
   - Rename the key from `"parties"` to `"organizations"`

9. **Keep camelCase property names** (externalId, firstName, lastName, etc.) — SupabaseAdminClient handles snake_case conversion.

Use the parallel branch's `git show feat-gsd-supabase-migration:tests/tests/data/default-dataset.json` as the reference for the target format.
</action>
<acceptance_criteria>
- `default-dataset.json` does NOT contain `"questionTypes"` key at root level
- All entity records contain `"projectId": "00000000-0000-0000-0000-000000000001"`
- All entity records contain `"published": true`
- `questionCategories` entries use `"categoryType"` not `"type"`
- `questionCategories` entries use `"_elections"` not `"elections"` for M:N refs
- `constituencyGroups` entries use `"_constituencies"` not `"constituencies"` for M:N refs
- `elections` entries use `"_constituencyGroups"` not `"constituencyGroups"` for M:N refs
- Questions do NOT contain `"questionType"` reference — use inline `"settings"` instead
- Nominations use `"organization"` not `"party"`
- Root-level key is `"organizations"` not `"parties"`
- File is valid JSON (no trailing commas, proper encoding)
</acceptance_criteria>
</task>
