# Plan 01: Verify Schema Reorganization Conventions

<objective>
Verify that the 4 success criteria for schema reorganization are met in the files copied from the parallel branch in Phase 30. This is a verification-only plan — no code changes expected unless a gap is found.
</objective>

## Tasks

<task id="1">
<title>Verify numbered migration file naming convention</title>
<read_first>
- apps/supabase/supabase/schema/ (directory listing)
- apps/supabase/supabase/migrations/ (directory listing)
</read_first>
<action>
Verify that ALL schema files in apps/supabase/supabase/schema/ follow the numbered naming convention:
- 000-enums.sql
- 010-utility-functions.sql, 011-validation-functions.sql
- 100-tenancy.sql through 108-admin-jobs.sql
- 200-indexes.sql
- 300-auth-tables.sql through 303-column-grants.sql
- 400-storage.sql
- 500-external-id.sql through 504-admin-rpcs.sql

Check that no un-numbered files exist. Check that migration files in apps/supabase/supabase/migrations/ also use numbered convention (00001_initial_schema.sql).

</action>
<acceptance_criteria>
- `ls apps/supabase/supabase/schema/ | grep -v '^[0-9]'` returns empty output (0 lines)
- `ls apps/supabase/supabase/schema/ | wc -l` returns at least 20 files
- Every .sql file in schema/ starts with a 3-digit number prefix
- `ls apps/supabase/supabase/migrations/ | grep -v '^[0-9]'` returns empty output
</acceptance_criteria>
</task>

<task id="2">
<title>Verify p_ parameter prefix convention in function signatures</title>
<read_first>
- apps/supabase/supabase/schema/011-validation-functions.sql
- apps/supabase/supabase/schema/301-auth-functions.sql
- apps/supabase/supabase/schema/400-storage.sql
- apps/supabase/supabase/schema/500-external-id.sql
- apps/supabase/supabase/schema/501-bulk-operations.sql
- apps/supabase/supabase/schema/503-entity-rpcs.sql
- apps/supabase/supabase/schema/504-admin-rpcs.sql
</read_first>
<action>
Verify that function parameters across all schema files use the p_ prefix convention. Check every CREATE OR REPLACE FUNCTION statement for parameter names.

Run: `grep -n "CREATE OR REPLACE FUNCTION\|CREATE FUNCTION" apps/supabase/supabase/schema/*.sql` to list all functions.

For each function that takes explicit parameters (not trigger functions or no-arg functions), verify parameters use p_ prefix.

Exceptions that are acceptable:
- Trigger functions (no user-facing parameters)
- Functions with no parameters
- The p_event parameter in custom_access_token_hook (already uses p_)
- Internal variable declarations (DECLARE blocks) do NOT need p_ prefix

Run: `grep -A5 "CREATE OR REPLACE FUNCTION" apps/supabase/supabase/schema/*.sql | grep -E "^\s+\w+\s+(uuid|text|jsonb|int|boolean|bigint)" | grep -v "p_" | grep -v "RETURNS\|DECLARE\|BEGIN"` — should return empty or only trigger/internal variables.
</action>
<acceptance_criteria>
- All functions with explicit parameters in schema/*.sql use p_ prefix for parameter names
- `grep -rn "FUNCTION public\." apps/supabase/supabase/schema/*.sql | wc -l` returns at least 15 (confirming functions exist to verify)
- Functions confirmed with p_ prefix: custom_access_token_hook(p_event), can_access_project(p_project_id), is_candidate_self(p_row_auth_user_id), is_storage_entity_published(p_entity_type_segment, p_entity_id_segment), delete_storage_object(p_bucket, p_file_path), is_localized_string(p_val)
</acceptance_criteria>
</task>

<task id="3">
<title>Verify public. schema qualifiers on table and view references</title>
<read_first>
- apps/supabase/supabase/schema/100-tenancy.sql
- apps/supabase/supabase/schema/101-elections.sql
- apps/supabase/supabase/schema/102-entities.sql
- apps/supabase/supabase/schema/302-rls.sql
- apps/supabase/supabase/schema/301-auth-functions.sql
</read_first>
<action>
Verify that table and view references throughout schema files use explicit public. schema qualifiers.

Check CREATE TABLE statements: `grep "CREATE TABLE" apps/supabase/supabase/schema/*.sql` — all should be `CREATE TABLE public.tablename`.

Check CREATE OR REPLACE FUNCTION statements: `grep "CREATE OR REPLACE FUNCTION" apps/supabase/supabase/schema/*.sql` — all should be `CREATE OR REPLACE FUNCTION public.functionname`.

Check RLS POLICY statements reference public. tables.

Check foreign key references and JOIN clauses for public. qualifiers where applicable.

Run: `grep -n "CREATE TABLE " apps/supabase/supabase/schema/*.sql | grep -v "public\." | grep -v "auth\.\|storage\.\|CREATE TABLE IF"` — should return empty (all tables qualified with public.).

Run: `grep -n "CREATE OR REPLACE FUNCTION " apps/supabase/supabase/schema/*.sql | grep -v "public\."` — should return empty.
</action>
<acceptance_criteria>
- `grep "CREATE TABLE " apps/supabase/supabase/schema/*.sql | grep -v "public\." | grep -v "auth\.\|storage\.\|CREATE TABLE IF" | wc -l` returns 0
- `grep "CREATE OR REPLACE FUNCTION " apps/supabase/supabase/schema/*.sql | grep -v "public\." | wc -l` returns 0 or only contains trigger functions from auth/storage schemas
- All CREATE TABLE statements in 100-108 files use `CREATE TABLE public.`
- All CREATE OR REPLACE FUNCTION statements use `CREATE OR REPLACE FUNCTION public.`
</acceptance_criteria>
</task>
