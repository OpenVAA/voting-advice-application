**Shared validation function (place in migration 00005, before JSONB triggers):**

Create `validate_answer_value(answer_val jsonb, question_type text, valid_choices jsonb DEFAULT NULL)` function. This is used by BOTH answer alternatives. Implementation from RESEARCH.md:
- Extract `answer_val -> 'value'` as the actual answer value
- NULL/missing value is always valid (unanswered)
- CASE on question_type:
  - 'text': must be string
  - 'number': must be number
  - 'boolean': must be true/false
  - 'date': must be string
  - 'singleChoiceOrdinal', 'singleChoiceCategorical': must be string, validate against valid_choices if provided
  - 'multipleChoiceCategorical': must be array
  - 'multipleText': must be array
  - 'image': must be object
  - else: RAISE EXCEPTION

**CRITICAL pitfall avoidance for validation triggers:**
- Use `TG_OP` to distinguish INSERT from UPDATE. On INSERT there is no OLD record.
- `IF TG_OP = 'UPDATE' AND NEW.answers IS NOT DISTINCT FROM OLD.answers THEN RETURN NEW; END IF;`

**Migration 00005 -- JSONB answer storage:**
- ALTER TABLE candidates ADD COLUMN answers jsonb DEFAULT '{}'::jsonb;
- ALTER TABLE organizations ADD COLUMN answers jsonb DEFAULT '{}'::jsonb;
- Create `validate_answers_jsonb()` trigger function that:
  - On UPDATE: skips if answers unchanged (using TG_OP check)
  - Iterates over jsonb_each(NEW.answers)
  - For each entry: looks up question and template, calls validate_answer_value
  - Validates question belongs to same project_id
- Create BEFORE INSERT OR UPDATE triggers on candidates and organizations

**Migration 00006 -- Relational answer storage:**
```sql
CREATE TABLE answers (
  id           uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  project_id   uuid NOT NULL REFERENCES projects(id),
  entity_id    uuid NOT NULL,
  entity_type  text NOT NULL CHECK (entity_type IN ('candidate', 'organization')),
  question_id  uuid NOT NULL REFERENCES questions(id),
  value        jsonb,
  open_answer  text,
  created_at   timestamptz DEFAULT now(),
  updated_at   timestamptz DEFAULT now(),
  UNIQUE (entity_id, question_id)
);
```
- Create `validate_answer_relational()` trigger function that:
  - Looks up the question by question_id within same project_id
  - Gets template via question's template_id
  - Calls validate_answer_value with the answer value, type, and choices
- Apply BEFORE INSERT OR UPDATE trigger
- Apply update_updated_at trigger

**IMPORTANT:** Both answer alternatives MUST be independently appliable/rollbackable. Migration 00005 must NOT depend on 00006 and vice versa. Both depend on the shared validate_answer_value function which lives in 00005 (since it is created first). If 00006 is applied without 00005, it should still work -- so place validate_answer_value in a separate migration or use CREATE OR REPLACE in both. Best approach: put validate_answer_value in 00005 and also have 00006 use CREATE OR REPLACE for the same function, making either migration self-contained.

**For load testing (Phase 11):** Both alternatives will be tested by applying one, running benchmarks, then rolling back and applying the other. Both must produce the same validation behavior.

**Migration 00008 -- Enable RLS + placeholder policies:**
For EVERY table (including join tables -- constituency_group_constituencies, election_constituency_groups):
```sql
ALTER TABLE {table_name} ENABLE ROW LEVEL SECURITY;

-- Placeholder deny-all policy. Real role-based policies added in Phase 10.
-- service_role bypasses RLS automatically in Supabase.
CREATE POLICY "{table_name}_deny_all" ON {table_name}
  FOR ALL
  USING (false);
```

This ensures:
- `supabase db lint` shows no RLS warnings
- All tables are locked down by default
- service_role (used by server-side operations) still has full access
- Phase 10 will DROP these placeholder policies and CREATE real ones
