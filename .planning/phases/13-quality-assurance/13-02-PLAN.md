- No INSERT/DELETE for party role (admin-only)

- candidates SELECT: `has_role('party', 'party', organization_id)` clause

Test sections:

1. **Anon can SELECT published records** (per-table explicit assertions):
   - Switch to anon via `set_test_user('anon')`
   - For each voter-facing table with published flag (elections, constituency_groups, constituencies, organizations, candidates, factions, alliances, question_categories, questions, nominations):
     - SELECT count(*) WHERE published = true -> assert >= 1 (Project A data is published)
     - SELECT count(*) WHERE published = false -> assert = 0 (unpublished data invisible)
   - app_settings: SELECT count(*) -> assert >= 1 (always readable, no published flag)
   - Join tables (constituency_group_constituencies, election_constituency_groups): SELECT count(*) -> assert >= 1 (always readable)

2. **Anon cannot read admin-only tables**:
   - Switch to anon
   - question_templates: SELECT count(*) -> assert = 0 (no anon policy)
   - accounts: SELECT count(*) -> assert = 0 (no anon policy)
   - projects: SELECT count(*) -> assert = 0 (no anon policy)

3. **Anon write denial on ALL tables** (loop-based per CONTEXT.md decision):
   - Use a PL/pgSQL DO block or helper function
   - Define array of ALL content tables: `['accounts', 'projects', 'elections', 'constituency_groups', 'constituencies', 'organizations', 'candidates', 'factions', 'alliances', 'question_templates', 'question_categories', 'questions', 'nominations', 'app_settings']`
   - Switch to anon
   - For each table, test INSERT denial using throws_ok with error code '42501' (insufficient_privilege) or '42501' pattern
   - For each table, test UPDATE denial: attempt `UPDATE tablename SET updated_at = now()` or similar neutral column -- for tables with RLS, this should return 0 rows affected (not an error). Use `is((SELECT count(*) affected), 0, ...)` pattern or check that the UPDATE statement affects 0 rows
   - For each table, test DELETE denial: `DELETE FROM tablename` should affect 0 rows

**Important:** Anon INSERT denial manifests as error 42501 (new row violates RLS). Anon UPDATE/DELETE manifests as 0 rows affected (RLS WHERE clause filters everything out). Use the appropriate assertion for each.

For INSERT denial in a loop, use a DO block with `PERFORM throws_ok(...)` inside the loop. The INSERT statement needs to be a valid-ish statement for the table. Use `format('INSERT INTO %I (id) VALUES (gen_random_uuid())', tbl)` as a minimal insert. Some tables require NOT NULL columns, so the INSERT may fail for constraint reasons too -- but the key is that RLS blocks it first with 42501. Test this carefully.

Alternatively, for tables where a minimal INSERT is hard to construct, use `format('INSERT INTO %I DEFAULT VALUES', tbl)` which will either fail on RLS (42501) or on NOT NULL constraints. Either failure proves anon cannot write. Use `throws_ok` without specifying error code to catch any exception.

4. **Anon cannot access user_roles or storage_config**:
   - These tables have REVOKE ALL FROM anon -- even SELECT should fail with 42501
   - Test with throws_ok on SELECT
