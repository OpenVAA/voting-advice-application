2. **set_test_user(p_role text, p_user_id uuid, p_user_roles jsonb)** function:
   - For `p_role = 'anon'`: set role to 'anon', clear request.jwt.claims and request.jwt.claim.sub
   - For `p_role = 'authenticated'`: set role to 'authenticated', set request.jwt.claims to a JSON object with `sub`, `role`, and `user_roles` fields, also set request.jwt.claim.sub separately (some auth functions read this)
   - Use `set_config(..., true)` for all settings (transaction-local)

3. **reset_role()** function: resets role back to postgres superuser via `set_config('role', 'postgres', true)` -- needed for test fixture insertion between role-switching assertions.

4. **Predictable UUID constants** as SQL variables or a helper function `test_uuid(name text)` that returns predictable UUIDs:
   - Account A: `'11111111-1111-1111-1111-111111111111'`
   - Account B: `'22222222-2222-2222-2222-222222222222'`
   - Project A: `'aaaaaaaa-aaaa-aaaa-aaaa-aaaaaaaaaaaa'`
   - Project B: `'bbbbbbbb-bbbb-bbbb-bbbb-bbbbbbbbbbbb'`
   - User IDs for: admin_a, admin_b, candidate_a, candidate_b, party_a, super_admin, account_admin_a (use distinct predictable UUIDs like `'cccccccc-cccc-cccc-cccc-000000000001'` through `'cccccccc-cccc-cccc-cccc-000000000007'`)

   - Creates user_roles entries linking users to roles:
     - admin_a: project_admin scoped to Project A
     - admin_b: project_admin scoped to Project B
     - candidate_a: candidate scoped to cand_a_id (in Project A)
     - candidate_b: candidate scoped to cand_b_id (in Project B)
     - party_a: party scoped to org_a (in Project A)
     - super_admin_user: super_admin with scope_type='global', scope_id=NULL
     - account_admin_a: account_admin scoped to Account A
   - Creates accounts (A, B), projects (A in account A, B in account B)
   - Creates elections (one per project, published=true for A, published=false for B)
   - Creates constituency_groups (one per project, published=true for A)
   - Creates constituencies (one per project, published=true for A)
   - Creates constituency_group_constituencies join entries
   - Creates election_constituency_groups join entries
   - Creates organizations (one per project with auth_user_id for party user, published=true for A)
   - Creates candidates (one per project with auth_user_id, published=true for A, published=false for B)
   - Creates factions, alliances (one per project, published=true for A)
   - Creates question_templates (one per project)
   - Creates question_categories (one per project, published=true for A)
   - Creates questions (one per project, type='singleChoiceOrdinal', published=true for A)
   - Creates nominations (candidate nomination under org nomination, in correct hierarchy -- org nomination first with election/constituency, then candidate nomination under it)
   - Creates app_settings (one per project)
   - Ensures all data satisfies trigger constraints (nomination hierarchy, valid answers if any)
   - Does NOT set answers on candidates/organizations (avoids trigger complications -- answer tests are in triggers file)

6. **Helper functions for building JWT claims**:

**File 1: 01-tenant-isolation.test.sql (QUAL-01)**

Structure: BEGIN, SET search_path, no_plan(), create_test_data(), then test sections, finish(), ROLLBACK.

Test sections:

1. **Cross-project SELECT isolation for all project_id tables** (loop-based):
   - Switch to user admin_b (project_admin for Project B)
   - Loop over all 12 tables with project_id: `['elections', 'constituency_groups', 'constituencies', 'organizations', 'candidates', 'factions', 'alliances', 'question_templates', 'question_categories', 'questions', 'nominations', 'app_settings']`
   - For each table: `EXECUTE format('SELECT count(*) FROM %I WHERE project_id = $1', tbl) INTO cnt USING project_a_id`
   - Assert `ok(cnt = 0, format('project_admin_b cannot see Project A data in %s', tbl))`

2. **Cross-project INSERT isolation** (loop-based):
   - Switch to user admin_b
   - For each key table (elections, candidates, organizations, questions, app_settings), attempt INSERT with project_id = Project A
   - Use throws_ok or verify that RLS blocks the insert (WITH CHECK violation results in error 42501 or new.row_security_violation)
   - Note: The INSERT will fail because can_access_project(project_a_id) returns false for admin_b

3. **Cross-project UPDATE isolation** (explicit):
   - Switch to admin_b, attempt UPDATE on elections in Project A
   - Assert 0 rows affected (RLS USING clause filters out non-project rows)

4. **Cross-project DELETE isolation** (explicit):
   - Switch to admin_b, attempt DELETE on elections in Project A
   - Assert 0 rows affected

5. **Candidate cross-project isolation**:
   - Switch to candidate_a (in Project A)
   - Verify candidate_a can see their own record
   - Verify candidate_a CANNOT see candidate_b's record (different project)
   - Verify candidate_a cannot update candidate_b's record

6. **Join table isolation via parent**:
   - Switch to admin_b
   - Verify constituency_group_constituencies and election_constituency_groups are always readable (USING true for SELECT)
   - But verify admin_b cannot INSERT into join tables referencing Project A parents (the WITH CHECK uses EXISTS on parent table project_id)
