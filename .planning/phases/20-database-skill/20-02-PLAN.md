**# RLS Policy Map** (title)

**## Policy Naming Convention** (~5 lines)
Document the naming pattern: `{role}_{operation}_{table}`. Operations: select, insert, update, delete. Roles in names: anon, authenticated, admin, candidate, party, service_role, auth_admin.

**## Role Hierarchy** (~10 lines)
Visual hierarchy showing role escalation:
- super_admin -> can do everything
- account_admin -> manages all projects in their account
- project_admin -> manages one project (via can_access_project)
- party -> self-edits their organization record
- candidate -> self-edits their candidate record
- anon -> reads published data only

**## Content Table Policies** (~80 lines)
For each content table, list all policies with their USING/WITH CHECK logic in compact form. Group by table. Use this format:

```
### elections (5 policies)
| Policy | Operation | Role | Condition |
|--------|-----------|------|-----------|
| anon_select_elections | SELECT | anon | published = true |
| authenticated_select_elections | SELECT | authenticated | can_access_project OR published |
| admin_insert_elections | INSERT | authenticated | can_access_project |
| admin_update_elections | UPDATE | authenticated | can_access_project |
| admin_delete_elections | DELETE | authenticated | can_access_project |
```

Cover all tables: accounts (4 policies, special), projects (4 policies, special), elections (5), constituency_groups (5), constituencies (5), constituency_group_constituencies (4, join), election_constituency_groups (4, join), organizations (6, has party self-edit), candidates (6, has candidate self-edit), factions (5), alliances (5), question_categories (5), questions (5), nominations (5), app_settings (5, anon always readable).

**## Auth Table Policies** (~10 lines)
Document user_roles special policies:
- auth_admin_read_user_roles (SELECT, supabase_auth_admin, USING true)
- service_role_manage_user_roles (ALL, service_role, USING true WITH CHECK true)
- REVOKE ALL from authenticated, anon, public

**## Storage Policies** (~25 lines)
List the 15 storage.objects policies organized by bucket and operation:
- public-assets: anon SELECT, authenticated SELECT, candidate INSERT/UPDATE/DELETE, admin INSERT/UPDATE/DELETE
- private-assets: authenticated SELECT, candidate INSERT/UPDATE/DELETE, admin INSERT/UPDATE/DELETE
Note the path-based authorization pattern using storage.foldername().

**## Role Access Matrix** (~25 lines)
A summary table showing what each role can do on each table category:

| Role | Content (own project) | Content (other project published) | Content (other project unpublished) | Own record | Storage |
|------|----------------------|----------------------------------|--------------------------------------|-----------|---------|
| super_admin | full CRUD | full CRUD | full CRUD | n/a | full CRUD |
| account_admin | full CRUD (own account projects) | read | none | n/a | own project |
| project_admin | full CRUD | read | none | n/a | own project |
| party | read published | read published | none | update own org | own org files |
| candidate | read published | read published | none | update own record | own files |
| anon | read published | read published | none | none | read published |

**## Column-Level Restrictions** (~10 lines)
Document the REVOKE/GRANT pattern for candidates and organizations:
- Protected columns: project_id, auth_user_id, organization_id (candidates only), published, id, is_generated
- Allowed columns for self-edit: name, short_name, info, color, image, sort_order, subtype, custom_data, answers, first_name/last_name (candidates), created_at, updated_at
- Admin operations use service_role which bypasses column-level grants

**Target:** ~200 lines total.
