**## Schema Conventions** (~50 lines)
Write each convention as an imperative rule (DO/NEVER/ALWAYS). Include these conventions:

1. **Common column set**: ALWAYS include on new content tables: id (uuid PK gen_random_uuid()), project_id (uuid NOT NULL FK projects ON DELETE CASCADE), name/short_name/info (jsonb localized), color/image (jsonb), sort_order (integer), subtype (text), custom_data (jsonb), is_generated (boolean DEFAULT false), created_at/updated_at (timestamptz), published (boolean NOT NULL DEFAULT false), external_id (text nullable with composite unique index).

2. **JSONB localization**: Store user-facing text as `{"en": "English", "fi": "Finnish"}`. Use `get_localized(val, locale, default_locale)` for 3-tier fallback. Mark functions using localization as IMMUTABLE where possible.

3. **JSONB answer storage**: Answers stored as JSONB column on candidates/organizations: `Record<QuestionId, {value, info?}>`. NEVER create a separate answers table (relational alternative benchmarked 2-10x slower). Smart validation trigger only validates changed keys on UPDATE.

4. **Enums**: Use PostgreSQL enums for constrained values: question_type (9 values), entity_type (4), category_type (3), user_role_type (5). NEVER use text with CHECK constraints -- enums provide type safety in generated TypeScript types.

5. **Nomination hierarchy**: Nominations use separate FK columns (candidate_id, organization_id, faction_id, alliance_id) with CHECK exactly-one constraint. Generated entity_type column. Validated by trigger: alliance=no parent, organization=parent alliance, faction=parent organization (required), candidate=parent organization/faction.

6. **Triggers**: ALWAYS add update_updated_at trigger on new tables. ALWAYS add cleanup_storage_on_delete and cleanup_image_on_update triggers on new entity tables. Add validate_answers_jsonb trigger if table stores answers. Add enforce_external_id_immutability trigger.

7. **Indexes**: ALWAYS create B-tree on project_id for new content tables. Create indexes on FK columns. Create partial indexes on published WHERE published = true for tables with anon SELECT policies. Composite unique partial index on (project_id, external_id) WHERE external_id IS NOT NULL.

8. **Schema file ordering**: Schema files are numbered 000-017. New schema files extend the sequence. Dependencies flow forward (higher numbers can reference lower). Schema concatenated into single migration file.

**## RLS and Auth Conventions** (~50 lines)
Write as imperative rules:

1. **5 role hierarchy**: super_admin (global) > account_admin (account scope) > project_admin (project scope) > party (organization scope) > candidate (self scope). Roles stored in user_roles table, injected into JWT via Custom Access Token Hook.

2. **Policy naming**: Use pattern `{role}_select_{table}`, `admin_insert_{table}`, etc. Specify TO anon or TO authenticated. SELECT: USING only. INSERT: WITH CHECK only. UPDATE: USING + WITH CHECK. DELETE: USING only.

3. **Optimizer caching**: ALWAYS use `(SELECT auth.uid())` and `(SELECT auth.jwt())` -- scalar subquery evaluated once per query, not per row. NEVER use bare `auth.uid()` or `auth.jwt()` in policy expressions.

4. **Standard content table**: 5 policies per table with published column: (1) anon_select USING published=true TO anon, (2) authenticated_select USING can_access_project OR published=true TO authenticated, (3) admin_insert WITH CHECK can_access_project, (4) admin_update USING+WITH CHECK can_access_project, (5) admin_delete USING can_access_project.

5. **Self-edit patterns**: Candidates use `auth_user_id = (SELECT auth.uid())` in USING/WITH CHECK. Organizations use `auth_user_id = (SELECT auth.uid()) OR has_role('party', 'party', id)`. Protected columns (project_id, auth_user_id, published, id) enforced via column-level REVOKE/GRANT.

6. **Join tables**: SELECT USING (true) for both anon and authenticated (structural data). INSERT/DELETE via EXISTS subquery checking parent table's can_access_project.

7. **user_roles table isolation**: NEVER expose user_roles to authenticated/anon roles. Only supabase_auth_admin (for Custom Access Token Hook) and service_role (for Edge Functions) can access. This prevents circular RLS.

8. **SECURITY DEFINER functions**: has_role(), can_access_project(), is_candidate_self() are SECURITY DEFINER with `SET search_path = ''` to prevent search_path attacks. These read JWT claims, not the database directly.

**## Service Conventions** (~40 lines)
Write as imperative rules:

1. **Edge Function pattern**: All Edge Functions verify admin role by decoding JWT claims from Authorization header. Create caller client (anon key) for auth verification, service_role client for admin operations. Handle CORS preflight. Return JSON with consistent error format.

2. **Bulk import**: Use bulk_import() RPC for batch data ingestion. Items MUST include external_id and project_id. Relationship fields expressed as `{"external_id": "..."}` objects. Processing order follows dependency chain. SECURITY INVOKER enforces admin RLS.

3. **Bulk delete**: Use bulk_delete() RPC. Three modes: prefix (LIKE prefix%), ids (UUID array), external_ids (text array). Processes in reverse dependency order. SECURITY INVOKER enforces admin RLS.

4. **Storage paths**: Format: `{project_id}/{entity_type}/{entity_id}/filename.ext`. Two buckets: public-assets (public=true), private-assets (public=false). Storage cleanup is automatic via pg_net async HTTP triggers.

5. **Email system**: send-email Edge Function uses resolve_email_variables() RPC for per-recipient variable resolution. Templates are locale-keyed: `{locale: {subject, body}}`. Variables use `{{path.to.field}}` mustache syntax. Supports dry_run mode.

**## pgTAP Testing Conventions** (~40 lines)
Write as imperative rules:

1. **File structure**: Tests in `apps/supabase/supabase/tests/database/`. Files numbered NN-name.test.sql. File 00-helpers.test.sql creates persistent helper functions (outside transaction). All other files run in BEGIN/ROLLBACK transactions.

2. **Test template**: Every test file follows: BEGIN; SET search_path = public, extensions; DROP TABLE IF EXISTS __tcache__; SELECT plan(N); SELECT create_test_data(); [tests]; SELECT * FROM finish(); ROLLBACK.

3. **User impersonation**: Use `set_test_user('authenticated', test_user_id('admin_a'), test_user_roles('admin_a'))` to simulate a user. Use `set_test_user('anon')` for anonymous. Use `reset_role()` to return to postgres superuser for fixture verification.

4. **Predictable UUIDs**: Use test_user_id(name) and test_id(entity_name) for predictable test entity references. NEVER use gen_random_uuid() for test entities that need to be referenced.

5. **RLS testing patterns**: Positive: `ok((SELECT count(*) FROM table WHERE condition)::integer >= 1, 'msg')`. Negative silent: `lives_ok($$UPDATE...$$, 'no error')` then `reset_role(); is((SELECT col FROM table WHERE id = ...), 'original', 'no effect')`. Error: `throws_ok($$INSERT...$$, '42501', NULL, 'msg')`.

6. **Multi-tenant testing**: create_test_data() provides two projects (A=published, B=unpublished) with 8 users across all 5 role types. Cross-project tests verify isolation. Same-project tests verify access levels.

7. **Adding new tests**: Create new numbered file after existing ones. Call create_test_data() for fixtures. If new test entities needed, add them to create_test_data() in 00-helpers.test.sql (add predictable UUID mappings to test_id and test_user_id).
