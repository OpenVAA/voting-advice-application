  <name>Task 1: Fix voter-settings.spec.ts settings mutation and navigation flow</name>
  <files>tests/tests/specs/voter/voter-settings.spec.ts</files>
  <action>
The voter-settings.spec.ts fails because `beforeAll` changes `questionsIntro.show` to `true`, which adds an extra questionsIntro page to the navigation flow. The tests then assume a Home -> Intro -> first question navigation but actually get Home -> Intro -> QuestionsIntro -> first question.

The root cause is twofold:
1. The `updateAppSettings` call in `beforeAll` may partially overwrite sibling settings (Pitfall 2: Strapi content-manager PUT replaces the entire `questions` component, not just the specified fields). The spec already sends complete siblings, but confirm every `beforeAll` sends ALL required sibling fields.
2. After `updateAppSettings` changes settings via API, the frontend SvelteKit app caches the old settings. The test navigates immediately without giving the frontend a chance to pick up new settings. Fix: After `updateAppSettings`, the first `page.goto()` in the test triggers a fresh server-side load which picks up the new settings. This should be fine as long as the navigation flow accounts for the changed settings.

For each describe block, audit the navigation flow to match the settings being set:

**Describe 1 (category selection)**: Sets `questionsIntro.show: true, allowCategorySelection: true`. Navigation should be: Home -> start -> Intro -> start -> QuestionsIntro (with category list). The test already expects to see `categoryList` after clicking intro start, which is correct. Verify the flow: `home.startButton` -> `intro.startButton` -> should see `questions.categoryList`. The `questionsIntro` page IS the page with the category list, so clicking intro start should navigate to the questionsIntro page which shows the category list. This flow should work IF the settings are applied. The issue might be that the intro page's start button navigates to questions, not questionsIntro. Check: with `questionsIntro.show: true`, after clicking intro start, the app navigates to `/questions` which then renders the questionsIntro page (with category list) as the first step.

If the navigation flow is actually correct per the settings, then the issue is that `updateAppSettings` is racing with the test. Fix by adding `await page.goto(buildRoute({ route: 'Home', locale: 'en' }))` at the START of each test (which already happens) and ensuring the page is fully loaded before clicking.

Add a `page.waitForLoadState('networkidle')` or equivalent wait after `page.goto()` in the first test of each describe block to ensure settings are fully loaded.

Additionally, ensure that if the `updateAppSettings` call in `beforeAll` takes effect mid-test-navigation, the first test gets a fresh page.goto. Add `{ timeout: 15000 }` to the `waitFor` calls for the first navigation step to give the app time to load updated settings.

For describe blocks where `questionsIntro.show: true`:
- After Home -> start -> Intro -> start, the next page IS the questionsIntro page
- Do NOT expect to land on a question page directly
- The questionsIntro page shows category list (when allowCategorySelection is true) or just a start button

For describe blocks where `questionsIntro.show: false`:
- After Home -> start -> Intro -> start, the next page IS the first question
- The test should see answer options directly

Review and fix the navigation expectations in each describe block to correctly match the settings being tested.

IMPORTANT: Do NOT change the describe block structure or settings values. Only fix the navigation expectations and timing.

<task type="auto">
  <name>Task 2: Fix voter-popups.spec.ts fixture interaction with settings mutation</name>
  <files>tests/tests/specs/voter/voter-popups.spec.ts</files>
  <action>
The voter-popups.spec.ts fails because `beforeAll` runs `updateAppSettings` to configure popup settings BEFORE the `answeredVoterPage` fixture navigates the voter journey. The fixture assumes default data.setup.ts settings (questionsIntro.show=false, categoryIntros.show=false) to navigate Home -> Intro -> Questions -> Results.

The `beforeAll` changes `notifications.voterApp.show: false` and `analytics.trackEvents: false` which are popup-related settings and should NOT affect the navigation flow. However, the `beforeAll` also sets `results.showFeedbackPopup` and `survey.*` settings, which also should not affect navigation.

The REAL issue is: the `updateAppSettings` PUT request replaces fields at the top level. If `beforeAll` sends `{ results: { showFeedbackPopup: 2, showSurveyPopup: null }, ... }`, the Strapi content-manager PUT may overwrite the entire `results` component, losing sibling fields like `results.cardContents` (which was the null crash already fixed in voterContext.ts).

The fix: Ensure every `updateAppSettings` call includes ALL sibling fields for each component being modified. Specifically:

1. For `results` component: always include `cardContents` alongside `showFeedbackPopup`/`showSurveyPopup` if needed. But more importantly, check what the data.setup.ts sets for `results` -- it sets nothing explicitly, meaning it's the Strapi default.

2. The more likely cause: The fixture navigation fails at `voter-home-start` click. This suggests the page itself is not loading. Check if `notifications.voterApp.show: false` setting somehow prevents the voter app from loading. Looking at the voter `+layout.svelte`, the notification check is `if ($appSettings.notifications.voterApp?.show)` which only affects popup queuing, not page rendering. The `access.voterApp` check gates page rendering, but that's not being modified.

3. The ACTUAL fix needed: The `answeredVoterPage` fixture from voter.fixture.ts creates a fresh `page` and navigates. But the voter-popups.spec.ts uses `voterTest` (from voter.fixture.ts) which extends base test with the `answeredVoterPage` fixture. The issue is that `beforeAll` runs ONCE for the describe block, while the `answeredVoterPage` fixture runs PER TEST. If `beforeAll` corrupts settings, every test's fixture will fail.

The fix approach: Move the settings mutation INTO each test, AFTER the fixture has navigated to results. Restructure like this:

```typescript
test('should show feedback popup after delay on results page', async ({ answeredVoterPage }) => {
  const page = answeredVoterPage;
  // NOW apply popup settings via API (after fixture has navigated to results)
  const client = new StrapiAdminClient();
  await client.login();
  await client.updateAppSettings({
    results: { showFeedbackPopup: 2, showSurveyPopup: null },
    survey: { showIn: [], linkTemplate: '' },
    ...suppressInterferingPopups
  });
  await client.dispose();
  // Reload the page to pick up new settings
  await page.reload();
  await page.getByTestId(testIds.voter.results.list).waitFor({ state: 'visible', timeout: 10000 });
  // Now test popup behavior...
});
```

HOWEVER, this approach has a problem: the feedback popup appears after a delay on results page load. By reloading, we trigger the popup timer fresh, which should work.

ALTERNATIVE (simpler): Keep `beforeAll` for settings BUT also ensure the `questions` settings are included to preserve the navigation flow. The issue might be that `beforeAll` settings call overwrites `questions` settings to their Strapi defaults. Add `questions` to the `beforeAll` call:

```typescript
test.beforeAll(async () => {
  await client.login();
  await client.updateAppSettings({
    results: { showFeedbackPopup: 2, showSurveyPopup: null },
    survey: { showIn: [], linkTemplate: '' },
    questions: {
      questionsIntro: { show: false, allowCategorySelection: false },
      categoryIntros: { show: false, allowSkip: true },
      showResultsLink: true
    },
    ...suppressInterferingPopups
  });
});
```

This is the safer fix: ensure `beforeAll` preserves ALL settings the fixture depends on by including `questions` settings explicitly. Per the project decision "Complete sibling settings in every updateAppSettings call to avoid Pitfall 2", this is the correct approach.

Apply this fix to ALL three describe blocks in voter-popups.spec.ts:
1. **feedback popup**: Add `questions` settings to `beforeAll` updateAppSettings
2. **survey popup**: Add `questions` settings to `beforeAll` updateAppSettings
3. **popups disabled**: Add `questions` settings to `beforeAll` updateAppSettings

Also update `defaultPopupSettings` in `afterAll` to include `questions` settings to restore the data.setup.ts defaults.

Also add `entities: { hideIfMissingAnswers: { candidate: false }, showAllNominations: true }` to preserve the entity settings the fixture depends on (the fixture navigates through questions to results, and needs all candidates visible).
