<objective>
Add the SupabaseDataAdapter type to the settings type system and wire the dynamic adapter switch in dataProvider.ts, dataWriter.ts, and feedbackWriter.ts so that setting `staticSettings.dataAdapter.type` to `'supabase'` loads the Supabase adapter. The default remains `'strapi'` — no change to runtime behavior unless config is changed.
</objective>

<must_haves>
- SupabaseDataAdapter type added to staticSettings.type.ts
- dataAdapter union type includes SupabaseDataAdapter
- dataProvider.ts switch handles 'supabase' case
- dataWriter.ts switch handles 'supabase' case
- feedbackWriter.ts switch handles 'supabase' case
- Default adapter remains 'strapi' in staticSettings.ts
- Existing Strapi adapter code is NOT modified
- The switch is designed so Phase 38 can simply remove the 'strapi' case and Strapi directory
</must_haves>

<action>
In `apps/frontend/src/lib/api/dataProvider.ts`, add a `'supabase'` case to the switch:

Replace the current switch block:
```typescript
switch (type) {
  case 'strapi':
    module = import('./adapters/strapi/dataProvider');

  case 'local':

    break;
  default:
    throw new Error(`Unsupported data provider: ${type}`);
}
```

With:
```typescript
switch (type) {
  case 'strapi':
    module = import('./adapters/strapi/dataProvider');
    break;
  case 'supabase':
    module = import('./adapters/supabase/dataProvider');
    break;
  case 'local':
    module = import('./adapters/apiRoute/dataProvider');
    break;
  default:
    throw new Error(`Unsupported data provider: ${type}`);
}
```

Note: The `./adapters/supabase/dataProvider` module does not exist yet — it will be created in Phase 35. This is safe because dynamic `import()` is only evaluated at runtime when `type === 'supabase'`, and the default config uses `'strapi'`.
</action>
