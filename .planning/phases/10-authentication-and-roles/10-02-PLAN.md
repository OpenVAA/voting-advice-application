**Answer data RLS coverage:**
- **JSONB approach (default/active):** Answers are stored as a `jsonb` column on `candidates` and `organizations` tables. No separate answers table exists. Answer data is inherently protected by the candidates/organizations RLS policies -- a candidate who can UPDATE their own record can update their answers column, and anon users who can SELECT published candidates see the answers column. No additional RLS work needed for JSONB answers.
- **Relational approach (alternative, NOT active by default):** If the relational `answers` table is active (schema/alternatives/answers-relational.sql copied to schema/006-answers-relational.sql), it has its own deny-all policy that would also need replacement. Add a commented-out section at the end of 010-rls.sql showing the RLS policies for the relational answers table, with a note: "Uncomment if using relational answer storage instead of JSONB." The relational answers table has `project_id` and `entity_id` columns, so policies follow the same pattern as other content tables, with the addition that a candidate can only read/write their own answers (WHERE entity_id matches their candidate id and entity_type = 'candidate').

**Policy pattern for content tables with project_id and published flag** (elections, candidates, organizations, questions, question_categories, nominations, constituencies, constituency_groups, factions, alliances):
```sql
-- Anon read (published only)
CREATE POLICY "anon_select_{table}" ON {table} FOR SELECT TO anon
  USING (published = true);

-- Authenticated read (own project or own record)
CREATE POLICY "authenticated_select_{table}" ON {table} FOR SELECT TO authenticated
  USING (
    (SELECT can_access_project(project_id))
    OR ... -- table-specific (e.g., auth_user_id = (SELECT auth.uid()) for candidates)
  );

-- Admin insert
CREATE POLICY "admin_insert_{table}" ON {table} FOR INSERT TO authenticated
  WITH CHECK ((SELECT can_access_project(project_id)));

-- Admin update
CREATE POLICY "admin_update_{table}" ON {table} FOR UPDATE TO authenticated
  USING ((SELECT can_access_project(project_id)))
  WITH CHECK ((SELECT can_access_project(project_id)));

-- Admin delete
CREATE POLICY "admin_delete_{table}" ON {table} FOR DELETE TO authenticated
  USING ((SELECT can_access_project(project_id)));
```

**Table-specific variations:**

1. **accounts** -- no project_id, no published flag:
   - SELECT: authenticated users with `has_role('account_admin', 'account', id)` or `has_role('super_admin')`
   - INSERT/UPDATE/DELETE: super_admin only

2. **projects** -- has account_id, no published flag:
   - SELECT: authenticated users with `can_access_project(id)` (project itself) or `has_role('account_admin', 'account', account_id)` or `has_role('super_admin')`
   - INSERT: `has_role('account_admin', 'account', account_id)` or super_admin
   - UPDATE/DELETE: `can_access_project(id)`

3. **candidates** -- has auth_user_id for self-edit:
   - Authenticated SELECT: `can_access_project(project_id)` OR `auth_user_id = (SELECT auth.uid())`
   - Add candidate self-update: `auth_user_id = (SELECT auth.uid())` in USING and WITH CHECK
   - Party admin can read candidates in their party: add `has_role('party', 'party', organization_id)` to SELECT

4. **organizations** -- has auth_user_id for party admin:
   - Authenticated SELECT: `can_access_project(project_id)` OR `auth_user_id = (SELECT auth.uid())`
   - Party admin update: `has_role('party', 'party', id)` for party self-edit

5. **question_templates** -- no published flag (admin-only):
   - No anon access. Authenticated SELECT/CRUD: `can_access_project(project_id)` only

6. **app_settings** -- no published flag, anon needs read for voter app:
   - Anon SELECT: `true` (app settings are always readable, no published filter needed)
   - Authenticated CRUD: `can_access_project(project_id)`

7. **Join tables** (constituency_group_constituencies, election_constituency_groups) -- no project_id:
   - Anon SELECT: `true` (structural data, always readable)
   - Authenticated INSERT/DELETE: check access via parent table FK. For constituency_group_constituencies: EXISTS query checking constituency_groups.project_id via can_access_project. Same pattern for election_constituency_groups.

**Critical implementation rules (from RESEARCH.md anti-patterns):**
- Always use `(SELECT auth.uid())` and `(SELECT auth.jwt())` with SELECT wrapper for optimizer caching
- Always specify `TO anon` or `TO authenticated` on every policy
- INSERT = WITH CHECK only (no USING)
- DELETE = USING only (no WITH CHECK)
- UPDATE = both USING and WITH CHECK
- SELECT = USING only

Per the RESEARCH.md recommendation and user decision that "candidates can edit all their own fields except structural ones (project_id, role assignments)":

```sql
-- Prevent authenticated users from updating structural columns on candidates
-- Admins (project_admin, account_admin, super_admin) use service_role which bypasses these
REVOKE UPDATE (project_id, auth_user_id, organization_id) ON candidates FROM authenticated;

-- Prevent authenticated users from updating structural columns on organizations
REVOKE UPDATE (project_id, auth_user_id) ON organizations FROM authenticated;
```

Note: service_role bypasses column-level grants, so admin Edge Functions can still update these columns. The REVOKE only affects the `authenticated` role used by PostgREST for logged-in users.
