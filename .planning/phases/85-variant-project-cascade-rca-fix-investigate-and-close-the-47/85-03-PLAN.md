## Root-cause RCA (post-close)

Two coupled bugs, both surfaced via the Phase 87 Path A manual e2e re-run:

**RC-1 — `--likert-only` semantic gap in the variant Playwright setup project.** The base e2e dev-seed template (`packages/dev-seed/src/templates/e2e.ts`) seeds 16 singleChoiceOrdinal opinion questions + 1 boolean (test-question-boolean-1, QSPEC-01) + 1 categorical (test-question-directional-1). The variant template extends via `...baseFixed('questions')`, inheriting all 18. The `multi-election.spec.ts` helper `answerAllQuestions()` calls `getByTestId('question-choice').nth(2).click()` on every opinion question — assumes 5-choice Likert. On the boolean question (2 radios), `.nth(2)` times out.

The `--likert-only` CLI flag (Phase 78 CLEAN-05) drops non-ordinal opinion questions but is CLI-only — `variant-multi-election.setup.ts` invokes `runPipeline()` programmatically, bypassing the CLI plumbing. The `applyLikertOnlyFilter()` function exists in `packages/dev-seed/src/cli/likert-only.ts` but was not exported from the package barrel.

**RC-2 — sharedPage + SvelteKit silent-nav race in the spec.** The Multi-election describe used `let sharedPage = await browser.newPage()` at `beforeAll`, then walked tests serially. The freshly-newPage'd context's `(located)/+layout.ts` data-promise doesn't reliably settle before the elections-Continue's client-side `goto()` fires, so the URL never updates. Phase 78 CLEAN-05 WR-03 patched around this with a manual `page.goto(/questions?electionId=...&constituencyId=...)` fallback, but the inherited URL state then leaked into the next serial test, producing the "/questions page empty" failure we observed after RC-1 was fixed.

The same pattern forced `test.use({ trace: 'off' })` because Playwright 1.58.2's trace writer hits ENOENT when a shared `Page` spans multiple serial tests in one worker. Per-test `page` fixtures eliminate both bugs simultaneously — the matrix-cell test at the bottom of the same file (`matrix cell: Ne × 1c`) already used `async ({ page })` successfully.

## Tasks

### Task 1 — Export `applyLikertOnlyFilter` + apply in variant setup

Files:
- `packages/dev-seed/src/index.ts` — add `applyLikertOnlyFilter` runtime export + `LikertOnlyFilterStats` type export to the package barrel.
- `tests/tests/setup/variant-multi-election.setup.ts` — import + call `applyLikertOnlyFilter(template)` between `runTeardown` and `runPipeline`. Drops non-`singleChoiceOrdinal` opinion questions, keeps all info questions (per filter's existing semantic: category_type === 'info' bypasses the type check).

Idempotence: filter is no-op on a template with no `questions.fixed[]`; mutates in place but doesn't change the pipeline behavior for templates that already match the predicate.

Side benefit: `results-sections.spec.ts` reuses the multi-election dataset and uses the same `.nth(2)` pattern (line 234) — it inherits the fix transparently.
