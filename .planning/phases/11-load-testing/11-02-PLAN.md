  <files>apps/supabase/benchmarks/results/</files>
  <action>
Execute the full benchmark suite using the scripts created in Plan 01. This task requires Supabase local stack running and k6 installed.

**Prerequisites check:**
1. Verify Supabase is running: `supabase status` from `apps/supabase/`
2. Verify k6 is installed: `which k6` (if not: `brew install k6`)
3. Verify pgbench is available: either locally (`which pgbench`) or via the Supabase Docker container (`docker exec -it supabase_db_openvaa-local pgbench --version`)

**Execution sequence:**

If the run-benchmarks.sh script works end-to-end:
```bash
cd apps/supabase/benchmarks
bash scripts/run-benchmarks.sh --full
```

If the orchestration script has issues, run each step manually:

1. **JSONB schema (default):**
   - Ensure JSONB schema is active (should be default)
   - `supabase db reset` from `apps/supabase/`
   - For each scale (1K, 5K, 10K):
     a. Generate data: `psql postgresql://postgres:postgres@127.0.0.1:54322/postgres -v scale=1000 -f data/generate-shared-data.sql && psql ... -v scale=1000 -f data/generate-candidates-jsonb.sql`
     b. Run pgbench for each pattern (30s each, or 60s for full accuracy):
        - `PGPASSWORD=postgres pgbench -h 127.0.0.1 -p 54322 -U postgres -c 1 -T 30 --log --log-prefix=results/jsonb-1k-voter-read -f pgbench/voter-bulk-read-jsonb.sql postgres`
        - Repeat for candidate-write (c=1), candidate-write-concurrent (c=100 j=4), full-save (c=1), aggregation (c=1), concurrent-read (c=100 j=4)
     c. Parse logs: `python3 scripts/parse-pgbench-log.py --json < results/jsonb-1k-voter-read.* > results/jsonb-1k-voter-read.json`
     d. Clean up data for next scale tier

2. **Relational schema:**
   - Run swap-schema.sh: `bash scripts/swap-schema.sh relational`
   - Repeat all steps from (1) using relational variants
   - After completion: `bash scripts/swap-schema.sh restore`

3. **k6 HTTP tests (once per schema):**
   - With JSONB active: `k6 run k6/voter-bulk-read.js`
   - Swap to relational, run again
   - Capture k6 summary output to results/

**Troubleshooting:**
- If pgbench is not available on host, use Docker: `docker exec -it supabase_db_openvaa-local pgbench ...`
- If psql connection fails, check that Supabase is running on port 54322
- If k6 PostgREST query returns errors, test the query manually with curl first and adjust the URL syntax
- If 500-VU concurrent test fails with connection errors, skip it and note in results (local Postgres connection limits)
- If validation triggers cause excessive slowdown on writes, run a quick test with triggers disabled as a comparison data point, but use trigger-enabled results as the primary data

**Output:** All raw results in `apps/supabase/benchmarks/results/` with parsed JSON files for each scenario. Capture any script output/errors for debugging in the decision document if needed.

**Critical:** After all benchmarks complete, restore the JSONB schema as the default (`bash scripts/swap-schema.sh restore`).
