## Rule 1 Deviation Applied (Phase 84 Plan 01, mid-execution)

After the initial post-fix smoke (commit 752967d, captured 2026-05-14T06:31:30Z) revealed that
`candidate-app-settings > should show read-only warning when answers are locked` regressed from
PASS (Phase 83 baseline) to FAIL, deeper RCA against the smoke.json timeline + the test
trace's error-context.md identified the actual root cause:

### Root cause (confirmed empirically)

The DETERM-08 dependency repoint changed `re-auth-setup.dependencies` from
`['candidate-app-mutation']` to `['candidate-app']`. Under `--workers=1`, sibling projects
depending on the same parent execute in DECLARATION ORDER. Declaration order in
playwright.config.ts: `candidate-app-mutation` (L122) BEFORE `re-auth-setup` (L149), BUT
Playwright's actual scheduler resolved `re-auth-setup` FIRST in the smoke (06:30:32 vs.
06:30:34) — likely because `re-auth-setup` has no `testDir` override and minimal setup
cost compared to mutation's larger spec scan.

Consequence: re-auth-setup wrote a fresh STORAGE_STATE at 06:30:32. Then mutation ran
`candidate-registration.spec.ts > should complete forgot-password and reset flow via
Inbucket email` at 06:31:06, which uses `TEST_CANDIDATE_ALPHA_EMAIL` (NOT a fresh
candidate) and calls `client.setPassword(Alpha, originalPassword)` →
`auth.admin.updateUserById({ password })` — which **REVOKES ALL** of Alpha's existing
refresh tokens, including the one just written by re-auth-setup. Settings then started
at 06:31:09 with a revoked storageState; safeGetSession() returned no session and the
protected layout's load() 307'd to /candidate/login. The trace's error-context.md
confirmed: the failure-time page showed the candidate-app LOGIN form (with the
answersLocked "Registration or editing answers is no longer possible..." copy), NOT the
protected home — proving the test was on the login page, not at a timing race.

The research agent missed this because `candidate-profile.spec.ts` uses
`E2E_ADDENDUM_CANDIDATES[1]` (a fresh candidate). But `candidate-registration.spec.ts`
has TWO describes: ADDENDUM[0] for fresh-candidate registration AND Alpha for the
password-reset block (L108-163). The research agent's "mutation does NOT touch Alpha"
claim was based on inspecting candidate-profile.spec.ts only.

### Chosen fix (minimal-surface Rule 1 deviation)

Add a file-scoped `test.beforeAll` to `candidate-settings.spec.ts` that opens a temporary
fresh browser context, signs in via the UI as Alpha, and overwrites STORAGE_STATE. The
per-test page fixtures then read the freshly-written storageState at context-creation
time. The hook runs BEFORE the first test of the candidate-app-settings project, so it
re-establishes valid Alpha auth regardless of whether mutation's password-reset ran
before or after re-auth-setup.

### Why this fix preserves DETERM-08 maximally

The `re-auth-setup` dependency stays on `candidate-app` (NOT mutation). The imgproxy-502
cascade-break that DETERM-08 was designed to provide is retained: if CAND-03 502s in
mutation, settings still runs (it depends on `re-auth-setup`, not `mutation`), and its
own beforeAll establishes fresh auth even if mutation's password-reset never ran.

### Rejected alternatives

- (a) Repoint re-auth-setup back to mutation: re-introduces imgproxy-502 cascade. Defeats
  DETERM-08.
- (b) Add candidate-app-mutation to candidate-app-settings.dependencies: same problem.
- (c) Move ToU acceptance into setup: not the issue. Alpha is seeded with
  `terms_of_use_accepted: '2025-01-01T00:00:00.000Z'` at packages/dev-seed/src/templates/e2e.ts:783.
- (d) Add a retry/wait in the read-only test: masks the issue rather than fixing it.

### Cost of the fix

~15 lines (plus ~45 lines of documenting comments) in candidate-settings.spec.ts. Adds
~3-4 seconds to the candidate-app-settings project setup (one UI login + storageState
serialize). No config-graph change, no test-infrastructure change, no UI change.
