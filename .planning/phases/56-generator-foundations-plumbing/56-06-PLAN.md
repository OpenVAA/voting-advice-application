1. **Exactly one of** `candidate_id` / `organization_id` / `faction_id` / `alliance_id` is non-null per row (CHECK constraint, migration line 741).

   - alliance nominations: no parent allowed
   - organization nominations: parent must be an alliance (or none)

```ts
import type { TablesInsert } from '@openvaa/supabase-types';
import type { Ctx, Fragment } from '../types';

export type NominationsFragment = Fragment<TablesInsert<'nominations'>>;

// Polymorphic ref shapes — bulk_import's resolve_external_ref converts them to FK IDs.
// Exactly ONE of candidate/organization/faction/alliance must be set per row
// (CHECK constraint, migration line 741).
type CandidateRef = { candidate: { external_id: string } };
type OrganizationRef = { organization: { external_id: string } };
type FactionRef = { faction: { external_id: string } };
type AllianceRef = { alliance: { external_id: string } };
type PolymorphicRef = CandidateRef | OrganizationRef | FactionRef | AllianceRef;

type NominationRow = TablesInsert<'nominations'> & PolymorphicRef & {
  election?: { external_id: string };
  constituency?: { external_id: string };
  parent_nomination?: { external_id: string };
};

/**
 * NominationsGenerator — polymorphic, wires candidates/parties/factions/alliances
 * to elections × constituencies with full in-memory referential integrity.
 *
 * GEN-08 enforcement: BEFORE emitting any row, validates that the refs it needs
 * are populated in ctx.refs. Throws client-side with a descriptive error. This
 * catches configuration bugs that would otherwise abort the bulk_import RPC
 * deep in PL/pgSQL (harder to diagnose).
 *
 * Polymorphism rules (migration line 741 CHECK):
 *   Exactly ONE of candidate/organization/faction/alliance per row.
 *
 * Hierarchy rules (validate_nomination trigger, migration lines 298–378):
 *   - alliance: NO parent
 *   - organization: parent = alliance (optional)
 *   - faction: parent = organization (REQUIRED — faction can't be top-level)
 *   - candidate: parent = organization or faction (optional)
 *
 * Parent consistency (migration lines 360–373):
 *   If parent_nomination_id set, child.election_id / constituency_id / election_round
 *   MUST match parent's. Generator emits parent refs with matching election+con only.
 *
 * Phase 56 emission strategy (simple, deterministic): for each candidate, emit ONE
 * candidate-type nomination for the FIRST election × FIRST constituency. This
 * exercises the polymorphism + ref-resolution path end-to-end without the
 * combinatorial explosion of cross-election × cross-constituency wiring (Phase 58
 * handles richer topologies via templates).
 *
 * Does NOT emit:
 *   - `id` (auto-generated UUID)
 *   - `entity_type` (GENERATED column — would fail "cannot write to generated column")
 *   - Redundant `organization` ref on candidate-type rows (RESEARCH §9: emit clean
 *     refs — the party-candidate relationship is expressed via candidates.organization_id)
 */
export class NominationsGenerator {
  constructor(private ctx: Ctx) {}

  defaults(_ctx: Ctx): NominationsFragment {
    return { count: 0 };  // Default 0 — Phase 56 emits `fixed[]` or explicit count; Phase 58 templates drive richer counts
  }

  generate(fragment: NominationsFragment): Array<TablesInsert<'nominations'>> {
    const { projectId, externalIdPrefix, refs } = this.ctx;
    const rows: Array<NominationRow> = [];

    // fixed[] pass-through — user-authored nominations, prefix applied
    for (const fx of fragment.fixed ?? []) {
      rows.push({
        ...(fx as NominationRow),
        external_id: `${externalIdPrefix}${fx.external_id}`,
        project_id: fx.project_id ?? projectId
      });
    }

    // Generated rows: one candidate-type nomination per candidate in ctx.refs.candidates.
    // This is a minimal Phase 56 strategy — enough to exercise the code path. Phase 58
    // templates can override with `nominations: (fragment, ctx) => ...` for richer wiring.
    const n = fragment.count ?? 0;
    if (n > 0) {
      // GEN-08: client-side ref validation. Throw with a descriptive error that points
      // at the missing ref — catching this here is far more debuggable than letting
      // bulk_import fail deep in PL/pgSQL.
      this.assertRefsPopulated();

      const electionExtId = refs.elections[0].external_id;
      const constituencyExtId = refs.constituencies[0].external_id;

      // Cap count at the number of candidates available (can't nominate who doesn't exist)
      const effectiveN = Math.min(n, refs.candidates.length);
      if (n > effectiveN) {
        this.ctx.logger(
          `[dev-seed] NominationsGenerator: requested ${n} candidate nominations but only ${refs.candidates.length} candidates in ctx.refs. ` +
          `Clamped to ${effectiveN}.`
        );
      }

      for (let i = 0; i < effectiveN; i++) {
        const candidateExtId = refs.candidates[i].external_id;
        rows.push({
          external_id: `${externalIdPrefix}nom_cand_${String(i).padStart(4, '0')}`,
          project_id: projectId,
          candidate: { external_id: candidateExtId },
          election: { external_id: electionExtId },
          constituency: { external_id: constituencyExtId },
          election_round: 1
          // No parent_nomination — Phase 56 candidate nominations are top-level for simplicity
          // No organization ref — that redundancy was a legacy tests/ workaround (RESEARCH §9)
        });
      }
    }

    // Cast back to TablesInsert<'nominations'>[] — the polymorphic refs + sentinels are
    // part of the shape bulk_import consumes (per migration line 2625–2634 relationship map)
    return rows as Array<TablesInsert<'nominations'>>;
  }

  /**
   * GEN-08 in-memory ref validation per D-07.3.
   *
   * Runs before any candidate-type nomination is emitted. Throws a descriptive
   * error pointing at the missing ref category, which surfaces in the pipeline's
   * exception path with clear diagnostics — far better than a PL/pgSQL
   * `RAISE EXCEPTION 'External reference not found: external_id "%" in table "%"'`
   * trace from bulk_import.
   */
  private assertRefsPopulated(): void {
    const { refs } = this.ctx;
    const missing: Array<string> = [];
    if (refs.candidates.length === 0) missing.push('candidates');
    if (refs.elections.length === 0) missing.push('elections');
    if (refs.constituencies.length === 0) missing.push('constituencies');
    if (missing.length > 0) {
      throw new Error(
        `[dev-seed] NominationsGenerator: cannot emit candidate nominations — ` +
        `ctx.refs is empty for: ${missing.join(', ')}. ` +
        `Ensure the pipeline runs in D-06 topo order (candidates/elections/constituencies BEFORE nominations) ` +
        `and that the template requests non-zero counts for these entities. Required refs must be populated ` +
        `before NominationsGenerator.generate() is called.`
      );
    }
  }
}
```

Notes:
- The Phase 56 strategy is DELIBERATELY simple: one candidate-type nomination per candidate, all wired to the first election × first constituency. This exercises the polymorphism + ref-resolution code path end-to-end. Phase 58 templates override with richer topologies via `(fragment, ctx) => rows[]`.
- **Polymorphism cleanliness:** NO redundant `organization` ref on candidate nominations. This differs from the legacy tests/ workaround which emitted both, then stripped the redundant one. RESEARCH §9 explicitly recommends dropping the workaround in dev-seed since we control emission.
- **Client-side validation** via `assertRefsPopulated` is the GEN-08 + D-07.3 guarantee in code. Tests in Plan 08 assert the specific error message for each missing ref case.
- **No `entity_type` emitted** — it's GENERATED, would fail DB-side.
- **No organization nominations, no faction nominations, no alliance nominations in Phase 56 generation path.** These are polymorphism variants the generator SHAPE supports (via `PolymorphicRef` union type + `fixed[]` pass-through), but Phase 56's generator only PRODUCES candidate nominations. Users who want party/faction/alliance nominations supply them via `fixed[]`. Phase 58 templates extend to generate them.
