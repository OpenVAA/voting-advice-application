# Plan 37-01: Fix auth-setup cascade — Supabase login API adaptation

<objective>
Fix the auth-setup test failure that cascades to 8 candidate tests. The root cause is that the login API route (`/api/auth/login`) expects `authToken` from the DataWriter's `login()` response, but the Supabase adapter returns `{ type: 'success' }` without an `authToken` because Supabase uses cookie-based sessions. This single fix should unblock all 9 failing tests (1 auth-setup + 8 candidate tests).
</objective>

<context>
The login flow works as follows:
1. `auth.setup.ts` fills the login form and clicks submit
2. The form action in `+page.server.ts` POSTs to `/api/auth/login`
3. The login API route calls `dataWriter.login()`, then checks for `authToken` in the response
4. The Supabase `_login()` calls `signInWithPassword` which sets session cookies via the server client's cookie handler
5. But the API route rejects the response because `authToken` is undefined

The fix: Adapt the login API route to work with Supabase's cookie-based auth instead of requiring a JWT `authToken`.
</context>

<action>
Modify `apps/frontend/src/routes/api/auth/login/+server.ts`:

1. **Replace the authToken check** on line 25:
   - OLD: `if (!loginResponse?.authToken) return apiFail(400);`
   - NEW: `if (loginResponse?.type !== 'success') return apiFail(400);`

2. **Get session from Supabase instead of authToken** — after confirming login succeeded, get the session from the Supabase server client. Replace the authToken-based cookie with session-based verification:
   - After the login() call succeeds, call `locals.supabase.auth.getSession()` to verify the session was established
   - If no session, return `apiFail(500)` (login succeeded but session not set)

3. **Remove the legacy AUTH_TOKEN_KEY cookie** — the `cookies.set(AUTH_TOKEN_KEY, authToken, {...})` line (around line 47-51) should be removed because:
   - Supabase server client manages its own session cookies via `createServerClient`'s cookie handler
   - The `AUTH_TOKEN_KEY = 'token'` cookie is a Strapi-era artifact
   - The `candidateAuthHandle` in hooks.server.ts uses `safeGetSession()` which reads Supabase cookies, not the legacy token cookie

4. **Update getBasicUserData call** — change from `{ authToken }` to `{ authToken: '' }` since the Supabase adapter ignores authToken:
   - OLD: `const userData = await dataWriter.getBasicUserData({ authToken }).catch(...)`
   - NEW: `const userData = await dataWriter.getBasicUserData({ authToken: '' }).catch(...)`

5. **Update backendLogout call** — similarly change from `{ authToken }` to `{ authToken: '' }`:
   - OLD: `await dataWriter.backendLogout({ authToken })`
   - NEW: `await dataWriter.backendLogout({ authToken: '' })`

6. **Keep the redirect behavior** — the form action in `+page.server.ts` already handles the redirect to candidate home after success. The login page's `use:enhance` handler calls `applyAction(result)` which processes the redirect.

The final POST function should:
- Call `dataWriter.login()` and check `type === 'success'`
- Verify session via `locals.supabase.auth.getSession()`
- Get basic user data with `{ authToken: '' }`
- Check role (existing logic)
- Return success JSON (no cookie setting needed)
</action>
