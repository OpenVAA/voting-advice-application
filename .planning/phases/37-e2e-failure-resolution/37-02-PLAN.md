# Plan 37-02: Fix voter test failures — detail, matching, results

<objective>
Fix the 5 voter test failures (1 in voter-detail, N in voter-matching, 3 in voter-results) that are independent of the auth-setup cascade. These tests run in the `voter-app` project which depends only on `data-setup`, not `auth-setup`. Each failure must be individually debugged, fixed, and verified.
</objective>

<context>
The voter test failures come from three spec files:
- `voter-detail.spec.ts` — specifically the party detail test (test 4: "should open party detail drawer")
- `voter-matching.spec.ts` — matching algorithm ranking verification
- `voter-results.spec.ts` — results page card counts and section switching

All use the `answeredVoterPage` fixture or manual navigation to reach results.

Key test data: combined default-dataset.json + voter-dataset.json loaded by data.setup.ts.
- 5 candidates from default dataset (all have termsOfUseAccepted)
- 7 candidates from voter dataset (6 visible + 1 hidden without termsOfUseAccepted)
- 2 organizations from default + voter datasets
- 16 opinion questions total (8 from each dataset)
- Answer values are raw numbers ("1"-"5") mapped to choice IDs
</context>

Run the voter-results tests in isolation to capture the exact failure messages:

```bash
cd tests && npx playwright test --project=voter-app tests/specs/voter/voter-results.spec.ts --reporter=line
```

**Expected failure scenarios and fixes:**

1. **Card count mismatch** — Test expects `visibleCandidateCount` candidates. The computation:
   ```typescript
   const visibleCandidateCount = [...defaultDataset.candidates, ...voterDataset.candidates]
     .filter(c => 'termsOfUseAccepted' in c && c.termsOfUseAccepted).length;
   ```
   If the Supabase adapter filters candidates differently (e.g., using `published` field, or nomination confirmation status, or `entities.hideIfMissingAnswers`), the count may differ. Fix by:
   - Checking `data.setup.ts` settings: `hideIfMissingAnswers.candidate: false` should show all
   - Checking `showAllNominations: true` should include all nominated candidates
   - Verifying the `termsOfUseAccepted` filter in the dataset matches frontend logic
   - If needed, update `visibleCandidateCount` computation or the dataset

2. **Entity tabs not appearing** — Test expects entity tabs to be visible. This requires `results.sections` to have 2+ sections. `data.setup.ts` sets `['candidate', 'organization']`. If the setting isn't being applied or the component doesn't render tabs for 2 sections, investigate the EntityCard and results page components.

3. **Party section not appearing** — Test clicks "parties" tab and expects `voter-results-party-section` to be visible. If no organizations are loaded or the section isn't rendered, check:
   - Whether organizations from both datasets are being imported correctly
   - Whether the party/organization section requires organizations to have answers
   - The testId `voter-results-party-section` exists in the results page component

4. **h3 count mismatch** — Test checks `partySection.locator('h3').first().toContainText('${totalPartyCount} parties')`. The `totalPartyCount` is `defaultDataset.organizations.length + voterDataset.organizations.length`. Verify the heading format matches.

Fix each issue found. If the data import is correct but the frontend renders differently, update the test assertions. If the data import is wrong, fix data.setup.ts or the dataset.
</action>

Run the voter-matching tests in isolation:

```bash
cd tests && npx playwright test --project=voter-app tests/specs/voter/voter-matching.spec.ts --reporter=line
```

**Expected failure scenarios and fixes:**

1. **Ranking order mismatch** — The test independently computes matching rankings using `@openvaa/matching` and compares against UI display order. If the answer format in the dataset differs from what the frontend loads (e.g., the frontend expects `choice_5` but gets raw `5`, or vice versa), rankings will diverge.

   Check:
   - How the test builds `candidateEntities` — maps `c.answersByExternalId` entries, converting raw values ("5") to `choice_5`
   - How the frontend data adapter reads answers from Supabase — does it use the `answers` JSONB column which stores UUID-keyed answers with `{ value: "5" }` or `{ value: "choice_5" }`?
   - If the frontend receives answers as `{ value: 5 }` (numeric), it may compute differently than the test which uses `choice_5` (string)

2. **Card count mismatch** — Same as voter-results: `expectedRanking.length` may not match displayed cards if visibility filtering differs.

3. **First/last card mismatch** — If rankings compute correctly but candidate display names don't match (e.g., name field format differences), the first/last assertions fail.

4. **Hidden candidate showing** — If the hidden candidate (without termsOfUseAccepted) appears in results, the exclusion assertion fails.

Fix by aligning the test's independent computation with how the frontend actually receives and processes the data. The answer value format must match exactly between test computation and frontend matching.
</action>
