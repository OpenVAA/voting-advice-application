  <files>apps/supabase/scripts/lint-schema.mjs</files>
  <action>
Create `apps/supabase/scripts/lint-schema.mjs` -- a Node.js script that runs Splinter-derived SQL queries against the local Supabase Postgres instance to check for schema-level issues that `supabase db lint` does not cover.

The script must:

1. Connect to the local Supabase Postgres at `postgresql://postgres:postgres@localhost:54322/postgres` using `psql` via `child_process.execSync`.

2. Implement these Splinter-derived checks as SQL queries:

**Check 1: RLS disabled on public tables (ERROR severity)**
Query: Find all tables in the `public` schema that have `relrowsecurity = false` in `pg_class`. Exclude known system/extension tables (`schema_migrations`, `supabase_migrations`, tables starting with `_`). If any table is found, report as ERROR.

SQL approach:
```sql
SELECT schemaname, tablename
FROM pg_tables t
JOIN pg_class c ON c.relname = t.tablename AND c.relnamespace = (SELECT oid FROM pg_namespace WHERE nspname = t.schemaname)
WHERE t.schemaname = 'public'
  AND c.relrowsecurity = false
  AND t.tablename NOT LIKE '\_%'
  AND t.tablename NOT IN ('schema_migrations');
```

**Check 2: Unindexed foreign keys (WARNING severity)**
Query: Find foreign key constraints where the referencing columns do not have a corresponding index. Uses `pg_constraint` joined with `pg_index` to find foreign keys without matching indexes.

SQL approach:
```sql
SELECT
  c.conrelid::regclass AS table_name,
  c.conname AS constraint_name,
  array_agg(a.attname ORDER BY x.n) AS columns
FROM pg_constraint c
CROSS JOIN LATERAL unnest(c.conkey) WITH ORDINALITY AS x(attnum, n)
JOIN pg_attribute a ON a.attrelid = c.conrelid AND a.attnum = x.attnum
WHERE c.contype = 'f'
  AND c.conrelid::regclass::text NOT LIKE 'pg_%'
  AND NOT EXISTS (
    SELECT 1
    FROM pg_index i
    WHERE i.indrelid = c.conrelid
      AND c.conkey[1:array_length(c.conkey, 1)] = i.indkey[1:array_length(c.conkey, 1)]
  )
GROUP BY c.conrelid, c.conname;
```

3. For each check, run the SQL via `psql` and parse the output:
   - Use `psql` with `--tuples-only --no-align --field-separator='|'` for machine-readable output
   - Parse rows from stdout
   - If rows exist for ERROR severity checks, set exit code to 1
   - If rows exist for WARNING severity checks, print warnings but don't fail (unless --strict flag passed)

4. Output format should be clear and actionable:
```
=== Schema Lint (Splinter-derived) ===

[ERROR] RLS disabled on public tables:
  - public.some_table
  - public.another_table

[WARNING] Foreign keys without indexes:
  - public.candidates.party_id (constraint: candidates_party_id_fkey)

Summary: 2 errors, 1 warning
```

5. Exit with code 1 if any ERROR-level issues found, 0 otherwise.

6. Handle the case where psql is not available (it should be, since Supabase runs Postgres in Docker, but check `supabase status` first and provide a helpful error message if the stack is not running).

7. Accept an optional `--strict` flag that treats warnings as errors too.

IMPORTANT: Fetch the actual Splinter SQL from the Splinter GitHub repo if possible during implementation (`https://github.com/supabase/splinter/blob/main/src/` or similar). If rate-limited, use the SQL approaches described above which are derived from Splinter's documented behavior. The key rules to cover are 0013 (RLS disabled) and 0001 (unindexed foreign keys). Additional rules (0002, 0007, 0008) are nice-to-have but not required for INFRA-05.
