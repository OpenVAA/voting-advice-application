**Replace the _insertJobResult stub** (currently `protected _insertJobResult() { throw new Error('SupabaseDataWriter._insertJobResult not implemented'); }`) with:

```typescript
protected async _insertJobResult({
  data
}: InsertJobResultOptions): DWReturnType<DataApiActionResult> {
  // Resolve project_id from election_id (AdminJobRecord doesn't include project_id
  // but the admin_jobs table requires it for RLS)
  const { data: election, error: electionError } = await this.supabase
    .from('elections')
    .select('project_id')
    .eq('id', data.electionId)
    .single();
  if (electionError || !election)
    throw new Error(`Failed to resolve project for election: ${electionError?.message ?? 'not found'}`);

  const { error } = await this.supabase.from('admin_jobs').insert({
    project_id: election.project_id,
    job_id: data.jobId,
    job_type: data.jobType,
    election_id: data.electionId,
    author: data.author,
    end_status: data.endStatus,
    start_time: data.startTime ?? null,
    end_time: data.endTime ?? null,
    input: data.input ?? null,
    output: data.output ?? null,
    messages: data.messages ?? null,
    metadata: data.metadata ?? null
  });
  if (error) throw new Error(`insertJobResult: ${error.message}`);
  return { type: 'success' as const };
}
```

Key design decisions:
- `_updateQuestion` ignores `authToken` (Supabase uses cookie-based sessions, per Phase 24 decision)
- `_updateQuestion` validates customData is a non-null object before calling RPC (matches Strapi adapter pattern)
- `_insertJobResult` resolves project_id from election_id via a SELECT on elections table (AdminJobRecord type doesn't include project_id but admin_jobs table requires it for RLS)
- `_insertJobResult` uses `?? null` for optional fields to ensure explicit NULL values in the INSERT
- Both methods return `{ type: 'success' as const }` on success and throw Error on failure (matching the established pattern in other Supabase adapter methods like _setAnswers, _updateEntityProperties)
- Do NOT change `supportsAdminApp` flag -- keep it as `false` per research recommendation (the two methods are called from server-side code that runs regardless of this flag)
