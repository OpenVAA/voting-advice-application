Create `voter-matching.spec.ts` in `tests/tests/specs/voter/`. This spec independently computes matching results using the `@openvaa/matching` package and compares them against the UI's displayed rankings.

**CRITICAL:** The `@openvaa/matching` package is a workspace package. It needs to be built before it can be imported in test files. Ensure `packages/matching` is built (it should be as part of the normal dev setup). Import from `@openvaa/matching` directly -- the workspace resolution should handle it.

If importing from `@openvaa/matching` doesn't work in the Playwright test environment (Playwright uses Node.js, not the browser, for test code), try importing from the built output: `import { MatchingAlgorithm, DISTANCE_METRIC, MISSING_VALUE_METHOD, OrdinalQuestion } from '@openvaa/matching'`. If that fails, import from the relative path: `import { MatchingAlgorithm } from '../../../../packages/matching/src/algorithms/matchingAlgorithm'` etc. The preferred approach is the workspace import.

Also import the voter dataset JSON for candidate answer data: `import voterDataset from '../../data/voter-dataset.json' assert { type: 'json' }`.

**Structure:**

```typescript
import { test, expect } from '../../fixtures';
import { buildRoute } from '../../utils/buildRoute';
import { testIds } from '../../utils/testIds';
import voterDataset from '../../data/voter-dataset.json' assert { type: 'json' };

// Import matching algorithm (workspace package)
import { MatchingAlgorithm, DISTANCE_METRIC, MISSING_VALUE_METHOD, OrdinalQuestion } from '@openvaa/matching';

// Compute expected rankings before any tests run
const QUESTION_COUNT = 8;
const VOTER_ANSWER_VALUE = 5; // "Fully agree"

// Create questions matching the voter dataset's 8 Likert-5 questions
const questions = Array.from({ length: QUESTION_COUNT }, (_, i) =>
  OrdinalQuestion.fromLikert({ id: voterDataset.questions[i].externalId, scale: 5 })
);

// Create voter answers (all "Fully agree")
const voterAnswers: Record<string, { value: number }> = {};
for (const q of questions) {
  voterAnswers[q.id] = { value: VOTER_ANSWER_VALUE };
}

// Create candidate answer objects from dataset
// Filter to only candidates with termsOfUseAccepted (visible candidates)
const visibleCandidates = voterDataset.candidates.filter(
  (c) => c.termsOfUseAccepted
);

// Build HasAnswers objects for each visible candidate
const candidateEntities = visibleCandidates.map((c) => {
  const answers: Record<string, { value: number }> = {};
  if (c.answersByExternalId) {
    for (const [qId, ans] of Object.entries(c.answersByExternalId)) {
      // Only include Likert question answers (the 8 opinion questions)
      const isLikert = questions.some((q) => q.id === qId);
      if (isLikert && ans.value) {
        answers[qId] = { value: Number(ans.value) };
      }
    }
  }
  return {
    name: `${c.firstName} ${c.lastName}`,
    externalId: c.externalId,
    answers
  };
});

// Run the matching algorithm
const algorithm = new MatchingAlgorithm({
  distanceMetric: DISTANCE_METRIC.Manhattan,
  missingValueOptions: { method: MISSING_VALUE_METHOD.RelativeMaximum }
});

// Note: The match() API takes objects implementing HasAnswers interface.
// The exact API may differ -- check the actual MatchingAlgorithm.match signature.
// May need to wrap candidates in a class implementing HasAnswers.
```

**IMPORTANT:** The `match()` method signature from the research shows it takes `{ questions, reference, targets }` where reference and targets implement `HasAnswers`. The `HasAnswers` interface from `@openvaa/core` requires an `answers` property of type `AnswerDict`. Check the exact types and adapt.

If the matching algorithm requires class instances (not plain objects), create a simple wrapper:
```typescript
class TestEntity {

}
```

**Tests:**

The spec uses serial mode. First test navigates to results, subsequent tests verify rankings.

**Test: "setup - navigate to results with all questions answered"**
- Same journey flow: home -> election selection -> intro -> answer all 8 with "Fully agree" -> results
- This is the setup for subsequent matching verification tests

**Test: "should display candidates in correct match ranking order"**
- Compute expected ranking using the algorithm (done in module scope above)
- Sort by ascending distance (best match first): `matches.sort((a, b) => a.distance - b.distance)`
- On the results page, read the candidate names/identifiers from the entity cards in order
- Compare the displayed order against the computed order
- Expected: cand-agree first (distance=0, 100% match), cand-oppose last (max distance)
- Reading names from cards: use `page.getByTestId(testIds.voter.results.card).allTextContents()` or iterate cards and extract text
- The entity card likely shows the candidate name. Extract it from each card in order.
- Compare the name sequence: displayed[0] should match computed[0], displayed[1] should match computed[1], etc.

**Test: "should show perfect match candidate (all agree) as top result"**
- Verify the first entity card contains the "agree" candidate name (e.g., "Fully Agree" or the candidate's firstName + lastName from the dataset)
- The agree candidate answered all 5s, voter answered all 5s, so distance = 0

**Test: "should show worst match candidate (all oppose) as last result"**
- Verify the last entity card (among visible candidates) contains the "oppose" candidate name
- The oppose candidate answered all 1s, voter answered all 5s, so distance = maximum

**Test: "should show partial-answer candidate in results with valid score"**
- The partial candidate has 4 answers (out of 8). With `RelativeMaximum` missing value method, missing answers are imputed with the maximum possible distance.
- Verify the partial candidate appears in the results list (is visible)
- The partial candidate should NOT be at the top or bottom but somewhere in between

**Test: "should NOT show hidden candidate (no termsOfUseAccepted)"**
- The hidden candidate has no `termsOfUseAccepted` field
- Verify the hidden candidate's name does NOT appear anywhere in the results
- Use text assertion (verification, not targeting): `expect(page.getByTestId(testIds.voter.results.candidateSection)).not.toContainText('Hidden')` where "Hidden" is part of the hidden candidate's name

**Test: "should confirm category intros are disabled" (VOTE-05 minimal coverage)**
- This verifies that the app settings change from data.setup worked
- Navigate to the intro page (or verify during the journey that no category intro page appeared)
- The setup journey already navigated through intro -> questions without hitting a category intro page
- Assert: `page.getByTestId(testIds.voter.questions.categoryIntro)` was never visible during the journey
- This can be verified by checking that the URL never contained `/questions/category/` during navigation
- Simplest: add an assertion in the setup test that after clicking intro start, the URL contains `/questions/` but NOT `/questions/category/`

**Test: "should confirm results are accessible after answering all questions" (VOTE-07 minimal coverage)**
- The minimum answers threshold is 5 (default). The voter answered all 8. Results should be accessible.
- This is already verified by reaching the results page in the setup test
- Explicitly assert: the results list is visible, entity cards are present
- This provides minimal VOTE-07 coverage -- the threshold is not tested at the boundary (that's Phase 4)
