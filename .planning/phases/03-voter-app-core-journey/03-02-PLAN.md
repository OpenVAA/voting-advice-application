Structure the spec as a serial describe block with these tests:

**Test: "should load home page and display start button" (VOTE-01)**
- Navigate to `buildRoute({ route: 'Home', locale: 'en' })`
- Assert `voterHomePage.startButton` is visible

- Verify navigation away from home page

**Test: "should show election selection with multiple elections" (VOTE-02)**
- After clicking start on home, if multiple elections exist, the app redirects to elections page
- Assert `page.getByTestId(testIds.voter.elections.list)` is visible
- Assert at least 1 election card is visible: `page.getByTestId(testIds.voter.elections.card).first()` is visible
- Click the election card for the voter test election (identify by text content "Test Voter Election" using `getByText` inside the card, or simply click the card that navigates to the voter election)
- Actually, per convention: avoid text-based selectors. Since there may be 2 elections (default + voter), click the one that corresponds to the voter election. Since we can't use text, click the card at a known index or use a more specific locator. The safest approach: use `page.getByTestId(testIds.voter.elections.card)` and iterate to find the right one. Or, since election selection order may not be deterministic, use `getByText` for verification only (not targeting) and click each card to find the voter election. Per Phase 2 decision: "text checks OK for verification, not targeting."
- Simplest approach: click each election card and check if it leads to intro/questions (single constituency auto-implies). If it leads back to elections, try the next one. This is fragile.
- Better approach: the voter election has a known name. Use `page.getByTestId(testIds.voter.elections.card).filter({ hasText: 'Test Voter Election' }).click()`. The `filter` method is not a raw locator -- it's Playwright's built-in method on Locator. This should be ESLint-safe since it starts from a testId locator.
- Verify navigation proceeds (URL changes from /elections to either /constituencies or /intro or /questions)

**Test: "should auto-imply constituency for single-constituency election" (VOTE-03)**

- Assert that the constituency selection page is NOT shown (URL does not contain /constituencies)

- If constituency selection IS shown (fallback), click the first constituency and continue

**Test: "should show questions intro page with start button" (VOTE-04)**

- Assert `voterIntroPage.startButton` is visible
- Note: category intros are disabled via app settings in data.setup, so no category selection checkboxes should appear

**Test: "should answer all 8 Likert questions with navigation" (VOTE-06)**
- Should now be on the first question page
- For questions 1-7:
  - Assert answer options are visible: `page.getByTestId(testIds.voter.questions.answerOption).first()` is visible
  - Select answer "Fully agree" (index 4, 0-based): `voterQuestionsPage.selectAnswer(4)`
  - Wait for auto-advance to next question. The app auto-advances after `DELAY.md` timeout. Wait for URL to change to next question.
  - IMPORTANT: Do NOT click next button after selecting answer -- the auto-advance handles navigation. Clicking next after auto-advance would skip a question (double navigation). Instead, after `selectAnswer`, wait for the URL to change.
- For question 8 (last):
  - Select answer "Fully agree" (index 4)
  - After auto-advance, the next button label changes to show "Results"
  - Click the next/results button: `voterQuestionsPage.goNext()`
  - Wait for results page: assert URL contains `/results`

**Test: "should navigate back and forth between questions" (VOTE-06 navigation)**

- Navigate to the voter home page and start again (fresh localStorage per fullyParallel)
- Wait -- in serial mode, tests share the same page. The previous tests already navigated to results. To test navigation, we need a fresh start.
- Actually, since `fullyParallel: true` is at config level but we use `test.describe.configure({ mode: 'serial' })`, tests within this serial block share the same worker. But each test gets its own browser context? No -- serial mode means they share the context.
- The simplest approach: make this a separate test that starts fresh. Or include navigation testing within the main question answering test.
- REVISED: Include previous/skip testing within the "answer all questions" test. After answering question 1, use previous to go back, verify we're on question 1 again, then answer and proceed. After question 3, skip question 4 (click next without answering), verify skip works, then go back and answer question 4.

Actually, the CONTEXT says "Answer all 8 Likert questions to exercise full question navigation (next, previous, skip, last-question behavior)." So navigation testing should be embedded in the journey flow.

REVISED structure for the questions test:
1. Answer question 1 (select "Fully agree", wait for auto-advance to Q2)
2. Go back to Q1 (click previous), verify Q1 is displayed again
3. Answer Q1 again (or verify answer is preserved), advance to Q2
4. Answer Q2, auto-advance to Q3
5. Skip Q3 (click next/skip without answering), advance to Q4
6. Go back to Q3, answer Q3, advance to Q4
7. Answer Q4-Q7 normally
8. Answer Q8, click "Results" button, verify results page

NOTE on auto-advance behavior: Research says `handleAnswer` calls `setTimeout(handleJump, DELAY.md)`. The `DELAY.md` is likely a constant. After selecting an answer, the page auto-advances. If the voter clicks "next" before auto-advance fires, it also navigates. The safest pattern: after `selectAnswer`, wait for URL change (next question). Don't click next explicitly for non-last questions.

For skip: click the next/skip button WITHOUT selecting an answer first. The button label shows "Skip" when no answer is selected.

For previous: click the previous button at any time.
