**1. bulk_import(data jsonb) RETURNS jsonb**

SECURITY INVOKER (uses caller's RLS context -- must be admin).
Language: plpgsql.

Structure:
```sql
CREATE OR REPLACE FUNCTION bulk_import(data jsonb)
RETURNS jsonb
LANGUAGE plpgsql
SECURITY INVOKER
AS $$
DECLARE
  -- Variables for processing
  collection_name text;
  collection_data jsonb;
  item jsonb;
  result jsonb := '{}'::jsonb;
  created_count integer;
  updated_count integer;
  -- Supported collections in dependency order
  processing_order text[] := ARRAY[
    'elections', 'constituency_groups', 'constituencies',
    'organizations', 'alliances', 'factions', 'candidates',
    'question_templates', 'question_categories', 'questions',
    'nominations', 'app_settings'
  ];
  col_name text;
BEGIN
  -- Process collections in dependency order
  FOREACH col_name IN ARRAY processing_order
  LOOP
    IF NOT data ? col_name THEN CONTINUE; END IF;
    collection_data := data -> col_name;
    created_count := 0;
    updated_count := 0;

    FOR item IN SELECT * FROM jsonb_array_elements(collection_data)
    LOOP
      -- For each item:
      -- 1. Extract external_id from item
      -- 2. Resolve any relationship references (e.g., {"external_id": "..."}) to UUIDs
      -- 3. Upsert by (project_id, external_id) using INSERT ON CONFLICT UPDATE
      -- 4. Count created vs updated
    END LOOP;

    result := result || jsonb_build_object(
      col_name, jsonb_build_object('created', created_count, 'updated', updated_count)
    );
  END LOOP;

  -- Validate no unknown collections were passed
  FOR collection_name IN SELECT * FROM jsonb_object_keys(data)
  LOOP
    IF NOT collection_name = ANY(processing_order) THEN
      RAISE EXCEPTION 'Unknown collection: %', collection_name;
    END IF;
  END LOOP;

  RETURN result;
END;
$$;
```

Key implementation details for the item processing loop:

a) **Relationship resolution:** Create a helper function `resolve_external_ref(ref jsonb, target_table text, project_id uuid) RETURNS uuid` that:
   - If ref is a JSON object with "external_id" key, looks up the target table for a row matching (project_id, external_id) and returns its UUID
   - If ref is a string (direct UUID), casts and returns it
   - If ref is null, returns null
   - Raises exception if external_id not found

   Use dynamic SQL (EXECUTE ... INTO) for the lookup since the target table name varies.

b) **Upsert logic:** For each item:
   - Extract `external_id` from item (required for import)
   - Extract `project_id` from item (or use a default if the function accepts it as parameter -- actually, the caller must include project_id in each item since the function is SECURITY INVOKER and needs project_id for RLS)
   - Build column-value pairs from the item JSON, mapping JSON keys to column names
   - For relationship fields (organization, election, constituency, category, template, parent, etc.), resolve via `resolve_external_ref()`
   - Use `INSERT INTO {table} (...) VALUES (...) ON CONFLICT (project_id, external_id) DO UPDATE SET ...`
   - Track whether the row was created or updated (use xmax = 0 to detect insert vs update)

   Since each table has different columns, use a collection-specific approach. Create an internal helper per collection OR use a mapping approach. The most maintainable approach: a single internal helper `_upsert_record(table_name text, item jsonb, project_id uuid)` that uses dynamic SQL.

   The relationship field mapping per collection:
   - candidates: `organization` -> organizations.external_id -> organization_id
   - nominations: `candidate` -> candidates.external_id -> candidate_id, `organization` -> organizations.external_id -> organization_id, `faction` -> factions, `alliance` -> alliances, `election` -> elections, `constituency` -> constituencies, `parent_nomination` -> nominations
   - questions: `category` -> question_categories.external_id -> category_id, `template` -> question_templates.external_id -> template_id
   - constituencies: `parent` -> constituencies.external_id -> parent_id

   Define the relationship mapping as a lookup within the function. For each collection, list which JSON keys map to which FK columns and which target tables.

c) **project_id handling:** Each item in the import data MUST include a `project_id` field. This is enforced by the function (raise exception if missing). The RLS policies will verify the caller has access to that project.

**2. bulk_delete(data jsonb) RETURNS jsonb**

SECURITY INVOKER.
Language: plpgsql.

Supports two modes based on the input format:
```json
// Mode 1: Prefix-based (delete all with matching external_id prefix)
{"elections": {"prefix": "import-2024-"}, "candidates": {"prefix": "import-2024-"}}

// Mode 2: Explicit ID list (delete by specific UUIDs or external_ids)
{"elections": {"ids": ["uuid-1", "uuid-2"]}, "candidates": {"external_ids": ["ext-1", "ext-2"]}}
```

For each collection:
- If `prefix` key: `DELETE FROM {table} WHERE external_id LIKE {prefix} || '%' AND project_id = {project_id}`
- If `ids` key: `DELETE FROM {table} WHERE id = ANY({ids}::uuid[]) AND project_id = {project_id}`
- If `external_ids` key: `DELETE FROM {table} WHERE external_id = ANY({external_ids}::text[]) AND project_id = {project_id}`

The `project_id` must be provided as a top-level key in the data JSON (not per-collection, since delete affects one project at a time).

Input format:
```json
{
  "project_id": "uuid",
  "collections": {
    "elections": {"prefix": "import-2024-"},
    "candidates": {"ids": ["uuid-1"]},
    "nominations": {"external_ids": ["nom-1", "nom-2"]}
  }
}
```

Process in REVERSE dependency order for delete (nominations first, then entities, then elections/groups -- to avoid FK violations):
```
nominations, questions, question_categories, question_templates,
candidates, factions, alliances, organizations,
constituencies, constituency_groups, elections, app_settings
```

Return result with deleted counts per collection.

Use dynamic SQL via EXECUTE for the table names. Validate collection names against the allowed list (same 12 tables).

**Grant execute** on both functions to authenticated role:
```sql
GRANT EXECUTE ON FUNCTION bulk_import(jsonb) TO authenticated;
GRANT EXECUTE ON FUNCTION bulk_delete(jsonb) TO authenticated;
```

Note: The functions are SECURITY INVOKER, so even though authenticated can call them, the actual operations inside are subject to RLS. Only admins with can_access_project() will succeed.
