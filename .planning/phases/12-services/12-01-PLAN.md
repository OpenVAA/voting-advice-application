2. Create `014-storage.sql` with the following sections:

   **a) Enable pg_net extension** (needed for cleanup triggers):
   ```sql
   CREATE EXTENSION IF NOT EXISTS pg_net WITH SCHEMA extensions;
   ```

   **b) Helper function for published-entity check on storage paths:**

   Create `is_storage_entity_published(entity_type_segment text, entity_id_segment text) RETURNS boolean` as a helper used by the anon SELECT policy. This function maps the entity_type path segment to the correct table and checks `published = true` on the owning entity.

   ```sql
   CREATE OR REPLACE FUNCTION is_storage_entity_published(entity_type_segment text, entity_id_segment text)
   RETURNS boolean
   LANGUAGE plpgsql SECURITY DEFINER
   SET search_path = ''
   AS $$
   DECLARE
     is_published boolean;
   BEGIN
     -- Project-level files (path: {project_id}/project/filename.ext) are always accessible
     IF entity_type_segment = 'project' THEN
       RETURN true;
     END IF;

     -- question_templates have no published flag (admin-only content, always accessible in public bucket)
     IF entity_type_segment = 'question_templates' THEN
       RETURN true;
     END IF;

     -- For all other entity types, check published = true on the owning entity
     -- Use dynamic SQL to query the correct table by entity_type path segment
     EXECUTE format(
       'SELECT published FROM public.%I WHERE id = $1',
       entity_type_segment
     ) INTO is_published USING entity_id_segment::uuid;

     -- If entity not found, deny access
     RETURN COALESCE(is_published, false);
   EXCEPTION
     WHEN OTHERS THEN
       RETURN false;
   END;
   $$;
   ```

   Note: The entity_type path segment directly matches the table name (e.g., 'candidates' -> candidates table), which is the convention established by the path format `{project_id}/{entity_type}/{entity_id}/filename.ext`.

   **c) Storage RLS policies on `storage.objects`:**

   Path format: `{project_id}/{entity_type}/{entity_id}/filename.ext`
   Use `(storage.foldername(name))[1]` for project_id, `[2]` for entity_type, `[3]` for entity_id.

   SELECT policies:
   - `anon_select_public_assets`: anon can SELECT from public-assets bucket **only for files belonging to published entities**. Per user decision: "Public bucket only serves files for published entities -- unpublished entity files stay private until published."
     ```sql
     CREATE POLICY "anon_select_public_assets" ON storage.objects FOR SELECT TO anon
       USING (
         bucket_id = 'public-assets'
         AND is_storage_entity_published(
           (storage.foldername(name))[2],
           (storage.foldername(name))[3]
         )
       );
     ```
     This joins to the owning entity table via the helper function and checks `published = true`. Project-level files (`{project_id}/project/...`) and question_templates always pass the check.

   - `authenticated_select_public_assets`: authenticated users can SELECT from public-assets. Two conditions (OR):
     (1) The entity is published (same `is_storage_entity_published` check), OR
     (2) The user is an admin with `can_access_project()` on the project_id from the path, OR
     (3) The user owns the entity (for candidates/organizations: entity's auth_user_id matches auth.uid())
     ```sql
     CREATE POLICY "authenticated_select_public_assets" ON storage.objects FOR SELECT TO authenticated
       USING (
         bucket_id = 'public-assets'
         AND (
           -- Published entities are readable by all
           is_storage_entity_published(
             (storage.foldername(name))[2],
             (storage.foldername(name))[3]
           )
           -- Admins can always see project files
           OR (SELECT can_access_project((storage.foldername(name))[1]::uuid))
           -- Entity owners can see their own files (candidates and organizations have auth_user_id)
           OR (
             (storage.foldername(name))[2] IN ('candidates', 'organizations')
             AND EXISTS (
               SELECT 1 FROM candidates c
               WHERE c.id = (storage.foldername(name))[3]::uuid
                 AND c.auth_user_id = (SELECT auth.uid())
               UNION ALL
               SELECT 1 FROM organizations o
               WHERE o.id = (storage.foldername(name))[3]::uuid
                 AND o.auth_user_id = (SELECT auth.uid())
             )
           )
         )
       );
     ```

   - `authenticated_select_private_assets`: authenticated users can SELECT their own files in private-assets (check via entity ownership or admin role)

   INSERT policies:
   - `candidate_insert_own_assets`: candidates can INSERT to public-assets under their own entity path. Check: bucket_id = 'public-assets' AND entity_type path segment = 'candidates' AND EXISTS subquery on candidates table matching entity_id to auth_user_id = auth.uid()
   - `candidate_insert_private_assets`: same pattern for private-assets bucket
   - `admin_insert_public_assets`: admins (can_access_project on project_id from path) can INSERT to public-assets
   - `admin_insert_private_assets`: admins can INSERT to private-assets

   UPDATE policies:
   - Same patterns as INSERT (candidates own files, admins project files)

   DELETE policies:
   - Candidates can DELETE their own entity files
   - Admins can DELETE any file in their project

   All policies use `(SELECT auth.uid())` and `(SELECT auth.jwt())` for optimizer caching, consistent with the 79 existing content table RLS policies.

   **IMPORTANT:** Cast path segments from `storage.foldername()` to uuid where needed. The foldername function returns text[]. Use `(storage.foldername(name))[1]::uuid` for project_id and `(storage.foldername(name))[3]::uuid` for entity_id.

   **d) Helper function for storage file deletion:**
   Create `delete_storage_object(bucket text, file_path text)` as SECURITY DEFINER with SET search_path = ''.
   Uses PostgreSQL custom settings `app.supabase_url` and `app.service_role_key` (set via seed.sql or ALTER DATABASE).
   Calls `net.http_delete()` to the Storage API endpoint: `{base_url}/storage/v1/object/{bucket}/{file_path}`.
   If settings are NULL, log a WARNING and return (graceful degradation for environments without pg_net config).

   **e) Entity deletion cleanup trigger:**
   Create `cleanup_entity_storage_files()` trigger function. On AFTER DELETE, calls `delete_storage_object` for each entity type table (candidates, organizations, factions, alliances, elections, constituencies, constituency_groups, nominations, question_templates, question_categories, questions).
   The trigger extracts project_id, derives entity_type from TG_TABLE_NAME, and uses the entity id to construct the path prefix.
   Note: pg_net can only delete individual files. The trigger should call the Storage API's bulk delete endpoint (POST to `/storage/v1/object/{bucket}` with JSON body `{"prefixes": ["{project_id}/{entity_type}/{entity_id}/"]}`). Use `net.http_post` instead of `net.http_delete` for the list-then-delete approach. Actually, the simplest approach: use the delete endpoint with the prefix. If the Storage API doesn't support prefix delete in a single call, use `net.http_post` to the `/storage/v1/object/list/{bucket}` endpoint first is too complex for a trigger. Instead, keep it simple: store the path prefix and use a single `net.http_post` to `/storage/v1/object/{bucket}` with body `{"prefixes":["{path_prefix}"]}`. This is the Storage API's batch delete endpoint that accepts prefixes.

   Attach this trigger to ALL entity tables (candidates, organizations, factions, alliances, elections, constituencies, constituency_groups, nominations, question_templates, question_categories, questions). Each table gets its own AFTER DELETE trigger calling the same function.

   **f) Image column update cleanup trigger:**
   Create `cleanup_old_image_file()` trigger function. On BEFORE UPDATE, if the `image` column changed (OLD.image IS DISTINCT FROM NEW.image) and OLD.image had a 'path' key, call `delete_storage_object` for the old path. Also check 'pathDark' and delete that too if it existed.
   Attach to all entity tables that have an `image` column: candidates, organizations, factions, alliances, elections, constituencies, constituency_groups, nominations, question_templates, question_categories, questions.

   **g) Seed data for pg_net configuration:**
   Add a comment noting that `app.supabase_url` and `app.service_role_key` custom settings must be set in seed.sql for local dev. Do NOT add the actual settings in this schema file -- they go in seed.sql (handled in seed update below).
