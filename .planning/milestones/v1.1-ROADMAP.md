## Phase Details

### Phase 8: Build Orchestration
**Goal**: Developers get fast, cached, dependency-aware builds across all workspace packages
**Depends on**: Nothing (first phase of v1.1)
**Requirements**: BUILD-01, BUILD-02, BUILD-03, BUILD-04, FIX-01
**Success Criteria** (what must be TRUE):
  1. Running `turbo run build` builds all packages in correct dependency order without manual sequencing
  2. Running `turbo run build` a second time with no changes completes in under 5 seconds (cache hit)
  3. Changing one package and rebuilding only rebuilds that package and its dependents
  4. A document exists evaluating Turborepo's impact on future Deno compatibility
  5. The app-shared ESM build produces correct output (no `packagec.json` typo in build artifacts) — NOTE: this dual CJS/ESM build is only needed for Strapi compatibility; can be removed after Strapi→Supabase migration
**Plans**: 2 plans

Plans:
- [x] 08-01-PLAN.md — Install Turborepo, create turbo.json, fix FIX-01 typo, write Deno evaluation
- [x] 08-02-PLAN.md — Migrate root scripts to turbo commands, verify caching, update documentation

### Phase 9: Directory Restructure
**Goal**: The monorepo follows apps/ + packages/ convention with all tooling updated for new paths
**Depends on**: Phase 8
**Requirements**: DIR-01, DIR-02, DIR-03, DIR-04, DIR-05
**Success Criteria** (what must be TRUE):
  1. Frontend, backend, and docs live under `apps/` and the dev Docker stack starts and runs correctly
  2. CI workflows execute successfully with the new directory layout
  3. All existing E2E tests pass without modification to test logic (only path configs if needed)
  4. TypeScript IDE resolution and builds work correctly from the new paths

Plans:

### Phase 10: Version Management
**Goal**: The project has automated version management with changelogs and release PRs
**Depends on**: Phase 9
**Requirements**: VER-01, VER-02, VER-03, VER-04, VER-05
**Success Criteria** (what must be TRUE):
  1. A developer can run `changeset` to declare a version bump intent and it records correctly
  2. Merging a PR with changesets triggers an automated version PR that updates package versions and changelogs
  3. PRs without changesets receive a bot comment reminding the author
  4. The release workflow includes an npm publish step that is present but disabled by default
  5. Per-package CHANGELOG.md files are generated with human-readable descriptions
**Plans**: 2 plans

Plans:
- [x] 10-01-PLAN.md — Install Changesets CLI, configure independent versioning with changelog-github plugin
- [x] 10-02-PLAN.md — Create release workflow, install changeset-bot GitHub App

### Phase 11: Package Publishing
**Goal**: Core library packages are publishable to npm with correct metadata, builds, and dependency resolution
**Depends on**: Phase 10
**Requirements**: PUB-01, PUB-02, PUB-03, PUB-04, PUB-05, PUB-06
**Success Criteria** (what must be TRUE):
  1. The `@openvaa` npm org exists and is configured for scoped package publishing
  2. Running `yarn pack` on core, data, matching, and filters produces tarballs with no `workspace:^` strings
  3. Each publishable package has complete npm metadata (license, description, repository, files, publishConfig)
  4. A fresh Node.js project can `npm install` a packed tarball and import its exports without errors
  5. Publishable packages produce ESM-only output via tsup (per user decision; app-shared retains temporary CJS for Strapi compatibility)
**Plans**: 3 plans

Plans:
- [x] 11-01-PLAN.md — Migrate all packages from tsc+tsc-esm-fix to tsup, output build/ to dist/
- [x] 11-02-PLAN.md — Add npm metadata, LICENSE files, Changesets public access, release workflow publishing
- [x] 11-03-PLAN.md — Full build verification and fresh install test for publishable packages

  2. Shared dependency versions are managed through Yarn catalogs (single source of truth for common deps)

- [x] 12-03-PLAN.md — Gap closure: add remote cache env vars and update Yarn version in main.yaml
