Create a reusable smoke test script at `scripts/deno-serve-test.sh`:
```bash
#!/usr/bin/env bash
set -euo pipefail

# Smoke test: SvelteKit production build served by Deno
# Usage: ./scripts/deno-serve-test.sh [port]

PORT="${1:-3456}"
BUILD_DIR="apps/frontend/build"

if [ ! -f "$BUILD_DIR/index.js" ]; then
  echo "ERROR: Build not found. Run 'yarn workspace @openvaa/frontend build' first."
  exit 1
fi

echo "Starting Deno server on port $PORT..."
PORT=$PORT deno run --allow-env --allow-read --allow-net "$BUILD_DIR/index.js" &
SERVER_PID=$!

# Wait for server startup
sleep 4

# Verify server is responding
HTTP_CODE=$(curl -f -s -o /dev/null -w "%{http_code}" "http://localhost:$PORT" 2>/dev/null || echo "000")

if [ "$HTTP_CODE" = "000" ]; then
  echo "FAIL: Server did not respond on port $PORT"
  kill $SERVER_PID 2>/dev/null || true
  exit 1
fi

echo "Server responded with HTTP $HTTP_CODE"

# Verify HTML content
BODY=$(curl -s "http://localhost:$PORT" 2>/dev/null | head -1)
if echo "$BODY" | grep -qi "html\|doctype\|redirect"; then
  echo "PASS: Deno-served frontend returns HTML content"
else
  echo "WARN: Response may not be HTML: $BODY"
fi

# Cleanup
kill $SERVER_PID 2>/dev/null || true
wait $SERVER_PID 2>/dev/null || true

echo "VAL-01: SvelteKit production build serves under Deno - PASS (HTTP $HTTP_CODE)"
```

Make the script executable: `chmod +x scripts/deno-serve-test.sh`

NOTE: If `--allow-env --allow-read --allow-net` is insufficient, update the script with the exact permissions that work. Add a comment documenting each additional permission and why it is needed.

**Step 3: Validate auth flow (VAL-05)**
Run the auth-related Playwright specs against the Deno-served frontend:
```bash

  --project=data-setup \
  --project=auth-setup \
  --project=candidate-app
```

The `candidate-app` project runs `candidate-auth.spec.ts` and `candidate-questions.spec.ts` which test:
- Login with email/password (PKCE flow through hooks.server.ts)
- Session persistence across navigation
- Protected route access
- Cookie-based session management

If auth specs pass, VAL-05 is validated. The PKCE auth flow goes through:
1. Client-side login form -> Supabase Auth API
2. Session cookie set via `@supabase/ssr` in hooks.server.ts
3. Subsequent requests read cookie for session validation
4. Protected routes check session in layout load functions
