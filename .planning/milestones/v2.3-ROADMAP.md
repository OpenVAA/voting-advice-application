## Phase Details

### Phase 45: Provider Abstraction and Configuration
**Goal**: Deployments can configure which identity provider (Signicat or Idura) to use, with a typed interface that both providers implement
**Depends on**: Nothing (foundation for v2.3)
**Requirements**: PROV-01, PROV-02, CONF-01, CONF-02, CONF-03
**Success Criteria** (what must be TRUE):
  1. A `PUBLIC_IDENTITY_PROVIDER_TYPE` env var selects between `signicat` and `idura` at startup
  2. Provider interface with `getAuthorizeUrl()`, `exchangeCodeForToken()`, and `getIdTokenClaims()` exists and both providers implement it
  3. All Idura-specific env vars are documented in `.env.example` with descriptions
  4. Key generation process (signing + encryption RSA key pairs) is documented
  5. Server constants file exports typed Idura env vars alongside existing Signicat vars
**Plans**: 2 plans

Plans:
- [x] 45-01-PLAN.md — Provider interface types, auth config, constants updates, env docs, key generation docs
- [x] 45-02-PLAN.md — Signicat provider implementation, Idura provider stub, provider factory

### Phase 46: Idura Authorization and Token Exchange
**Goal**: Candidates can authenticate via Idura's FTN bank authentication using server-side JAR and private_key_jwt
**Depends on**: Phase 45
**Requirements**: AUTH-01, AUTH-02, AUTH-03, TOKN-01, TOKN-02
**Success Criteria** (what must be TRUE):
  1. Server-side `/api/oidc/authorize` endpoint constructs a signed JWT Authorization Request and returns the Idura redirect URL
  2. Frontend preregister page calls the server-side authorize endpoint (instead of building PKCE redirect client-side) when provider is Idura
  3. Callback route works at a provider-agnostic path (not hardcoded to `signicat/oidc/callback`)
  4. Token exchange sends `private_key_jwt` client assertion (not `client_secret`) when provider is Idura
  5. JWE decryption handles both RSA-OAEP (Signicat) and RSA-OAEP-256 (Idura) encryption algorithms
**Plans**: 2 plans

Plans:
- [x] 46-01-PLAN.md — Idura provider JAR + private_key_jwt implementation, authorize endpoint, updated token route
- [x] 46-02-PLAN.md — Provider-agnostic callback route, updated preregister page, route map update, old callback deleted

### Phase 47: Edge Function Provider Support
**Goal**: The identity callback Edge Function correctly processes identity verification for both Signicat and Idura providers
**Depends on**: Phase 45
**Requirements**: EDGE-01, EDGE-02, EDGE-03
**Success Criteria** (what must be TRUE):
  1. Edge Function reads provider type from configuration and dispatches to the correct identity matching logic
  2. Idura identity matching uses `sub` claim (persistent pseudonym) instead of `birthdate` for user lookup
  3. Provider type (`signicat` or `idura`) is stored in user `app_metadata` so auth method is auditable
**Plans**: 1 plan

Plans:
- [x] 47-01-PLAN.md — Provider-agnostic identity-callback Edge Function, frontend caller update, skill docs

### Phase 48: Backward Compatibility and Testing
**Goal**: Both Signicat and Idura auth paths are verified working, with test coverage ensuring neither provider regresses
**Depends on**: Phase 45, Phase 46, Phase 47
**Requirements**: PROV-03, TEST-01, TEST-02
**Success Criteria** (what must be TRUE):
  1. Existing Signicat flow works identically when `PUBLIC_IDENTITY_PROVIDER_TYPE=signicat` (no behavioral change)
  2. Unit tests cover both Signicat and Idura code paths through the provider abstraction layer
  3. E2E preregistration tests pass using the provider-agnostic callback route
**Plans**: 3 plans

Plans:
- [x] 48-01-PLAN.md — Shared test fixtures (RSA keys, JWE/JWT builders), provider interface compliance tests, JWE decryption tests
- [x] 48-02-PLAN.md — JAR construction tests, private_key_jwt client assertion tests, Edge Function claim extraction tests
- [ ] 48-03-PLAN.md — E2E regression validation (candidate registration) and human verification checkpoint
