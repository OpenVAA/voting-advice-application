### v2.2 Deno Feasibility Study (Paused)

**Milestone Goal:** Evaluate Deno 2.x as a runtime replacement for Node.js across the OpenVAA monorepo, producing a go/no-go recommendation backed by a working proof of concept and performance benchmarks.

**Status:** Phase 42 validated that Deno 2.7.8 can run the full stack (SvelteKit, 38/38 E2E, Supabase auth, shared package tests). Code changes were rolled back to avoid maintenance burden without a forcing function (no CI enforcement, no production deploy path). Research artifacts preserved in `.planning/phases/42-runtime-validation-and-poc/`.

**Key findings from Phase 42:**
- SvelteKit adapter-node runs on Deno with zero code changes (HTTP 200, static assets, CSRF)
- `--unstable-bare-node-builtins` required for Paraglide JS (`async_hooks` without `node:` prefix)
- Supabase PKCE auth works identically (no cookie serialization issues)
- Turborepo/Changesets/tsup unaffected by hybrid deno.json+package.json workspace
- A vitest compatibility shim enables existing tests to run on both runtimes without duplication

**To resume:** Reinstall Deno, re-apply deno.json configs, and continue with Phase 43. See `42-RESEARCH.md` for technical details and the vitest shim approach.

- [x] **Phase 42: Runtime Validation and PoC** - Prove Deno runs the existing stack (completed 2026-03-26, code rolled back)
- [ ] **Phase 43: Evaluation and Benchmarking** - Quantify benefits and assess toolchain/security tradeoffs
- [ ] **Phase 44: Findings Report** - Synthesize evidence into go/no-go recommendation and migration plan

## Phase Details

### Phase 42: Runtime Validation and PoC
**Goal**: Deno 2.x can run the OpenVAA monorepo's critical paths without breaking the existing Yarn/Turborepo/tsup build pipeline
**Depends on**: Nothing (first phase of milestone)
**Requirements**: VAL-01, VAL-02, VAL-03, VAL-04, VAL-05, POC-01, POC-02, POC-03
**Success Criteria** (what must be TRUE):
  1. SvelteKit production build (adapter-node) starts and serves pages under `deno run`
  2. At least one package (@openvaa/core or @openvaa/matching) passes its full test suite via `deno test`
  3. Turborepo orchestrates builds across a workspace that contains both Deno and npm package members
  4. Supabase cookie-based PKCE auth flow completes successfully (login, session persistence, logout) when the SvelteKit server runs on Deno
  5. Playwright E2E tests execute against the Deno-served frontend (via npx or native, at least 10 specs passing)
**Plans:** 2/2 plans complete
Plans:
- [x] 42-01-PLAN.md — Deno install, hybrid workspace config, @openvaa/core PoC tests, coexistence smoke tests
- [x] 42-02-PLAN.md — SvelteKit on Deno, Supabase auth validation, Playwright E2E against Deno frontend

### Phase 43: Evaluation and Benchmarking
**Goal**: Quantified understanding of what Deno actually improves, what stays the same, and what the security model means for election software
**Depends on**: Phase 42 (need working runtime to benchmark)
**Requirements**: EVAL-01, EVAL-02, EVAL-03
**Success Criteria** (what must be TRUE):
  1. Side-by-side comparison document exists showing which tools Deno replaces, which must stay, and why (with evidence from Phase 42)
  2. Reproducible benchmark results exist for: cold install, full build, dev server startup, and production server startup (Node vs Deno)
  3. Security permissions model is assessed for election deployment (what permissions are needed, what attack surface changes)
**Plans**: TBD

### Phase 44: Findings Report
**Goal**: A clear go/no-go recommendation that the project can act on, with a concrete next-steps plan regardless of the decision
**Depends on**: Phase 43 (need benchmarks and evaluation data)
**Requirements**: RPT-01, RPT-02, RPT-03
**Success Criteria** (what must be TRUE):
  1. Written report exists with go/no-go recommendation and per-blocker evidence (each VAL/POC requirement has pass/fail/workaround status)
  2. If go: migration plan with prioritized adoption targets (which parts of the stack move to Deno first, timeline, rollback strategy)
  3. If no-go: cherry-pick recommendation identifying specific Deno features worth adopting without full migration (e.g., Edge Function alignment only)
  4. Performance benchmark summary with concrete numbers (not blog claims) from Phase 43
**Plans**: TBD
