## Phase Details

### Phase 22: Leaf Component Migration
**Goal**: Shared and voter-app leaf components are fully in Svelte 5 runes mode with no legacy globals, unblocking all downstream migration
**Depends on**: Phase 21 (v1.2 infrastructure complete)
**Requirements**: COMP-01, COMP-02, COMP-03, COMP-06, COMP-07, COMP-08, COMP-09
**Success Criteria** (what must be TRUE):
  1. Every shared and voter-app leaf component uses `$props()` instead of `export let`, with no `$$restProps`, `$$slots`, or `$$Props` remaining in any migrated file
  2. Leaf components that forwarded events via `on:click` (notably Button) now accept callback props (`onclick`), and all call sites across both apps pass handlers correctly
  3. Props that are bound via `bind:` have `$bindable()` annotations and binding works correctly in all leaf component consumers
  4. `svelte:self` in EntityTag and `svelte:component` deprecations (7 occurrences) are resolved with standard imports
  5. Event modifiers (`|once`, `|preventDefault`, `|capture`) are replaced with inline JavaScript equivalents in all leaf components
**Plans**: 7 plans

Plans:
- [ ] 22-01-PLAN.md — Simple shared leaf components (26 files: Avatar, Icon, EntityTag svelte:self, Image event forwarding, etc.)
- [ ] 22-02-PLAN.md — Complex shared leaf components (18 files: Button event forwarding, Input/Select/Toggle $bindable, EntityFilters svelte:component) + Button call site updates
- [ ] 22-03-PLAN.md — Question + controller sub-components (12 files) + Video.svelte (event modifiers, accessors, $bindable)
- [ ] 22-04-PLAN.md — Standard dynamic components (18 files: mechanical migration of entityDetails, entityList, feedback, footer, survey, etc.)
- [ ] 22-05-PLAN.md — High-attention dynamic components (12 files: EntityCard svelte:self, AppLogo svelte:component, NavItem event forwarding, createEventDispatcher boundary preservation)
- [ ] 22-06-PLAN.md — Candidate + admin components (11 files) + comprehensive verification sweep
- [ ] 22-07-PLAN.md — Gap closure: fix export function bind:this consumers, revert WithPolling slot, fix Select type

### Phase 23: Container Components and Layouts
**Goal**: Container components and root layouts deliver content via snippet props and callback props, with all consumers in both apps updated atomically
**Depends on**: Phase 22
**Requirements**: COMP-04, COMP-05, LAYOUT-01, LAYOUT-02
**Success Criteria** (what must be TRUE):
  1. All named `<slot>` elements in container components (Modal, Alert, EntityCard, HeadingGroup, Tabs, etc.) are replaced with `{@render}` snippet props, and every consumer passes content as snippets
  2. `createEventDispatcher` is removed from all 6 dispatching components (Alert, Expander, Navigation, SurveyButton, Feedback, DataConsent) and replaced with callback props
  3. MainContent.svelte's 6 named slots are converted to snippet props with all voter and candidate consumer call sites updated
  4. Root Layout.svelte uses `$bindable()` for `isDrawerOpen` and snippet-based content rendering, with both apps' root layouts working correctly
**Plans**: 4 plans

Plans:
- [ ] 23-01-PLAN.md — Dispatcher removal (5 components + DataConsent rename) + default-slot-only snippet conversion (11 components)
- [ ] 23-02-PLAN.md — Modal chain (ModalContainer, Modal, Drawer, ConfirmationModal, TimedModal) + Alert runes + snippets + all slot="actions"/slot="badge" consumer updates
- [ ] 23-03-PLAN.md — Layout.svelte + SingleCardContent.svelte runes + snippets + route layout consumer updates (3 +layout files)
- [ ] 23-04-PLAN.md — MainContent.svelte runes + snippets + all ~39 consumer route files (voter, candidate, admin, error)

### Phase 24: Voter Route Migration
**Goal**: All voter route pages and layouts use Svelte 5 runes for local state and reactivity while continuing to consume store-based contexts via `$storeName` syntax
**Depends on**: Phase 23
**Requirements**: ROUTE-01, ROUTE-02, ROUTE-03, ROUTE-04
**Success Criteria** (what must be TRUE):
  1. Every `$:` reactive statement in voter route files is converted to the correct rune: `$derived` for pure computations, `$derived.by()` for multi-statement computations, `$effect` for async/side-effect operations
  2. All `on:event` directives in voter route templates are replaced with native event attributes (`onclick`, `onsubmit`, etc.) or callback props passed to child components
  3. All `<slot>` usage in voter route files is converted to `{@render children?.()}` or appropriate snippet rendering
  4. The root `+layout.svelte` async data-loading pattern (election/constituency initialization) is correctly migrated from `$:` to `$effect` without SSR failures or infinite loops
**Plans**: 4 plans

Plans:
- [ ] 24-01-PLAN.md — Runes opt-in for 7 simple route pages + results layout slot conversion
- [ ] 24-02-PLAN.md — Elections, constituencies, questions pages ($: block splitting + $page store migration)
- [ ] 24-03-PLAN.md — Results page (complex $: blocks + $page.state shallow routing)
- [ ] 24-04-PLAN.md — Voter layouts ($: async data-loading + slot conversion, ROUTE-04 critical path)
