# Phase 69 — UI Design Contract

> Visual and interaction contract for Alliance Card Lane A. Generated by gsd-ui-researcher; verified by gsd-ui-checker. The contract is **alliance-specific** — it locks the new visual surfaces only. The Organization card / drawer surfaces stay as-is and are NOT re-documented here.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | none (project predates shadcn era; Tailwind v4 + DaisyUI 5 is the established system) |
| Preset | not applicable |
| Component library | DaisyUI 5 (native @plugin integration in `apps/frontend/src/app.css:3-7`) |
| Icon library | project-internal (`$lib/components/icon`); no new icons introduced this phase |
| Font | Inter (`--font-base` in `apps/frontend/src/tailwind-theme.css` via `app.css:226-228`) |

This phase introduces **no new visual primitives**. Every alliance surface reuses pre-existing components (`EntityCard`, `EntityDetails`, `EntityChildren`, `Avatar`, `MatchScore`, `EntityList`, `Tabs`, `Button`). Per ROADMAP SC-3: "no new bespoke components for the drawer body."

## Spacing Scale

The project uses an established Tailwind v4 spacing token set defined in `apps/frontend/src/app.css:154-179`. Phase 69 reuses these tokens unchanged — no new spacing primitives.

| Token | Value | Usage in Phase 69 |
|-------|-------|-------------------|
| `xs` | 4px (0.25rem) | (not directly used) |
| `sm` | 8px (0.5rem) | Inline gaps inside the alliance card header (existing `gap-sm` on title row) |
| `md` | 10px (0.625rem) | Alliance card grid `gap-md` (existing `EntityCard` `gridClasses`); summary line vertical breathing |
| `lg` | 20px (1.25rem) | Subcards stack `gap-lg`; drawer children-tab `py-lg`, `mb-lg` (existing `EntityChildren` surface) |
| `xl` | 40px (2.5rem) | Container max-width breakpoint (`max-w-xl`); not a spacing application |
| `16` | 16px (1rem) | Card vertical padding `py-16` (existing `EntityCard` baseline) |

**Phase 69 spacing rules (additive):**

- The new "X candidates across N parties" summary line sits in the existing `gap-md` card grid, immediately after the `<header>` block. **No new spacing token introduced** — it inherits the parent `grid gap-md` rhythm.
- Drawer-header summary line sits inside the existing `EntityCard variant="details"` slot at line 112 of `EntityDetails.svelte`. **No new spacing.**
- Member-org subcard list uses the existing `gap-lg` grid (`EntityCard.svelte:286`). **No `maxSubcards` truncation visual** — alliance branch sets `maxSubcards = Infinity`, so the expand-button slot is suppressed; the list renders flat.

**Exceptions:** none. All alliance spacing is delegated to the existing card grid.

## Typography

| Card title (list variant) | `h3 → text-lg + bold` | 1.0625rem (17px) | 700 | 1.21 | Alliance name on the card |

**Phase 69 typography rules:**

- Alliance card title: identical to Organization card title (no visual differentiation by font; `<h3>` for `variant="list"`, `<h4>` for `variant="subcard"`). The hierarchy is established by avatar + name + summary, not by oversize type.
- Summary line ("X candidates across N parties"): use `text-sm text-secondary` to visually subordinate it under the bold `<h3>` name. This matches the existing pattern in `EntityCard.svelte:225-241` (subtitle row uses `gap-sm` + secondary content).
- **No new font sizes. No new font weights.** This is mandatory — the project ships exactly two font weights (400 and 700, declared at `app.css:222-223`).

## Color

The project uses a DaisyUI semantic color token system defined in `apps/frontend/src/app.css:9-69` (light + dark themes). Phase 69 reuses existing tokens unchanged.

| Role | Token | Light Hex | Dark Hex | Usage in Phase 69 |
|------|-------|-----------|----------|-------------------|
| Dominant (60%) | `--color-base-100` | `#ffffff` | `#000000` | Card surface (alliance card, drawer body) |
| Secondary surface (30%) | `--color-base-200` / `--color-base-300` | `#e8f5f6` / `#d1ebee` | `#101212` / `#1f2324` | Page background under results list (`bg-base-300` from `+layout.svelte:351`); subcard offset-border (existing `--line-color`) |
| Accent (10%) | `--color-accent` | `#0a716b` | `#11a8a0` | **Reserved-for list below** |
| Match score callout | (computed by `MatchScore` component from score 0-100) | per-score gradient | per-score gradient | Top-right of card (existing `MatchScore` placement; alliance card surfaces this when match ≥ minimumAnswers) |
| Per-alliance accent | `entity.color` from `Avatar` (alliance color set by Phase 67 planner-picked seed) | per-alliance | per-alliance | Avatar tile background ONLY (existing `Avatar.svelte:65-71`); used as identifying chip, not as borders/links |
| Destructive | `--color-error` / `--color-warning` | `#a82525` | `#e16060` | Not used by alliance card or drawer (no destructive action on read-only results page) |

**Accent reserved for** (explicit allowlist — never "all interactive elements"):

1. Inline links (`a:link:not(.btn)` → `text-primary` per `app.css:312-318`).
2. `Button variant="main"` and `Button color="accent"` instances (none introduced this phase).

**`entity.color` (per-alliance avatar tile):**

- The Phase 67 default seed picks one color per alliance via `colorPalette[allianceIndex % palette.length]` (deterministic). This already flows through `Avatar.svelte:65-71` which reads `unwrapEntity(entity).entity.color` and applies `--bg-color`/`--bg-color-dark` to the avatar tile background.
- **Phase 69 does NOT introduce additional surfaces colored by `alliance.color`.** Specifically: no full-card color-stripe, no border-tinting, no per-card background tint. The avatar tile is the sole surface that distinguishes alliances visually by color — identical to how Organization cards are distinguished today.
- Rationale: introducing additional color surfaces would (a) require defining contrast guarantees against `--color-base-100` for arbitrary planner-picked palettes, and (b) diverge from the org-card precedent. Lane A is "uniform extension of an existing pattern" (RESEARCH §"Don't Hand-Roll" key insight).

**Phase 69 color rules:**

- The alliance card's visual distinguishability from organization cards comes from THREE existing channels:
  1. **Avatar tile color** (`entity.color` planner-picked at seed time).
  2. **Subcard list content** (member orgs, not candidates — visible at-a-glance from the subcard avatar shape and party-name text).
  3. **Summary line copy** ("X candidates across N parties" vs no equivalent on org cards).
- **No new color tokens. No new gradient/stripe surfaces. No new border-tint utilities.**

- Locales without translator availability at planning time: ship English `Members` as fallback (matches the existing locale-ratchet pattern for new keys).

**Destructive confirmations:** N/A — alliance card and drawer are read-only views on the voter results page. No destructive actions introduced.

### Alliance Card (variant="subcard")

**Not used in Phase 69.** Alliances render only as top-level list cards inside the "Alliances" results tab. They are never subcards of another card. (Alliance is the topmost parent type — Alliance → Organization → Candidate.) The `variant === 'subcard'` branch in `EntityCard.svelte` ignores subentities (per recursion guard at line 134), so an alliance card cannot recursively spawn alliance subcards.

## Frontmatter / Implementation Notes for Planner

These are NOT visual contracts — they are notes the planner consumes alongside the contract above.

1. **Type-rename gate:** the `'candidates' → 'children'` rename across cardContents/details types is a SINGLE-COMMIT operation per Risk #2. The visual contract above assumes the rename is atomic; the planner enforces atomicity at task-split time.
2. **Route-matcher widening (Risk #4):** widening `entityTypePlural` and `entityTypeSingular` is a UI prerequisite for SC-3 click-to-drawer. The visual contract assumes the matcher is widened — without it, "click → drawer opens" cannot be observed and no card-render fix can be visually verified.
3. **`results/+layout.svelte` tab handling (Risk #5):** `ENTITY_PLURALS`, `_urlPlural`, `handleEntityTabChange`, `_pluralForActiveType` extension are also UI prerequisites. The visual contract assumes the alliance tab is fully wired into the existing tab strip.
4. **i18n regen:** after JSON edits, regenerate `apps/frontend/src/lib/types/generated/translationKey.ts` via `tsx apps/frontend/tools/translationKey/generateTranslationKeyType.ts` per VALIDATION.md Wave 0.
5. **Summary helper:** RESEARCH Finding 6 recommends a tiny `getAllianceSummary(allianceNom): { numCandidates: number; numParties: number }` utility shared between `EntityCard` (list variant summary) and `EntityDetails` (drawer header summary) to avoid drift. Visual contract above assumes the same string is rendered in both places — the helper is the implementation device that guarantees this.
