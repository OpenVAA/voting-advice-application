<!-- SES email flow architecture (from 28-RESEARCH.md) -->

Email send chain:
1. Test calls strapiAdminClient.sendEmail({ candidateId, subject, content, requireRegistrationKey })
2. Strapi email service finds candidate, checks registrationKey
3. sendToAll() calls sendRegistrationEmail() for each candidate
4. sendRegistrationEmail() calls strapi.plugins['email'].services.email.send({ to, subject, text })
5. @strapi/provider-email-nodemailer uses SES transport -> sendRawEmail to LocalStack
6. LocalStack stores email in memory, accessible via /_aws/ses

Email read chain:
1. Test calls emailHelper.countEmailsForRecipient(email) to get stale count
2. Test sends email via step 1
3. Test polls emailHelper.getLatestEmailHtml(email, skipCount) until truthy
4. getLatestEmailHtml -> fetchEmails() -> GET http://localhost:4566/_aws/ses
5. Response: { messages: [{ Id, Region, Source, RawData, Timestamp }] }
6. Filters by recipient in RawData, skips old emails, parses latest with mailparser
7. Returns parsed.textAsHtml || parsed.html
8. Test extracts link with extractLinkFromHtml() using cheerio

Failing tests:
- candidate-registration.spec.ts:73 "should complete registration via email link"
- candidate-registration.spec.ts:133 "should complete forgot-password and reset flow via SES email"
- candidate-profile.spec.ts:59 "should register the fresh candidate via email link"

Failure mode: registrationLink is undefined -> page.goto(undefined) fails

SES Config (plugins.ts):
- Provider: 'nodemailer' with SES transport
- SES client: new aws.SES({ endpoint: env('LOCALSTACK_ENDPOINT'), region: env('AWS_SES_REGION'), credentials: { accessKeyId, secretAccessKey } })
- Inside Docker container: LOCALSTACK_ENDPOINT should be http://awslocal:4566
- Outside Docker (tests): LOCALSTACK_ENDPOINT is http://localhost:4566

Diagnostic commands (run in order):
1. docker ps --filter name=awslocal --format "{{.Status}}"
2. docker exec $(docker ps -q --filter name=awslocal) awslocal ses list-identities --region us-east-1
3. curl -s http://localhost:4566/_aws/ses | python3 -m json.tool
4. docker logs $(docker ps -q --filter name=strapi) 2>&1 | grep -i "email\|ses\|error" | tail -20
5. docker exec $(docker ps -q --filter name=strapi) printenv LOCALSTACK_ENDPOINT

Most likely root causes (from research):
A. LocalStack SES not receiving emails (Strapi transport error, silent catch in sendToAll)
B. Email identity not verified (LocalStack restarted without init script re-running)
C. Nodemailer SES transport misconfiguration (LOCALSTACK_ENDPOINT wrong inside container)
D. SDK version mismatch (aws-sdk v2 vs v3 API)

    Per D-10, the 3 failing tests are caused by SES email infrastructure issues. Per D-11, Docker stack is assumed running. Diagnose the root cause by following the investigation priority list, then apply the fix.

    **Step 1: Verify Docker stack is accessible**

    ```bash
    curl -s -o /dev/null -w "%{http_code}" http://localhost:5173  # frontend
    curl -s -o /dev/null -w "%{http_code}" http://localhost:1337  # strapi
    curl -s -o /dev/null -w "%{http_code}" http://localhost:4566/_localstack/health  # localstack health
    ```

    If any service is not responding, STOP and report. The plan assumes `yarn dev` is already running.

    **Step 2: Check LocalStack container health and SES state**

    ```bash
    # Container status
    docker ps --filter name=awslocal --format "{{.Status}}"

    # Verified email identities (must include no-reply@openvaa.org per MAIL_FROM)
    docker exec $(docker ps -q --filter name=awslocal) awslocal ses list-identities --region us-east-1

    # Current SES inbox contents
    curl -s http://localhost:4566/_aws/ses | python3 -m json.tool
    ```

    If `no-reply@openvaa.org` is NOT in the identity list, the init script did not run. Fix:
    ```bash
    docker exec $(docker ps -q --filter name=awslocal) awslocal ses verify-email-identity --email-address no-reply@openvaa.org --region us-east-1
    ```

    **Step 3: Check Strapi's LOCALSTACK_ENDPOINT inside the container**

    ```bash
    docker exec $(docker ps -q --filter name=strapi) printenv LOCALSTACK_ENDPOINT
    ```

    Expected: `http://awslocal:4566` (Docker internal hostname). If it shows `http://localhost:4566` or `http://127.0.0.1:4566`, the Strapi container cannot reach LocalStack because localhost inside a container refers to the container itself. This would be the root cause.

    Fix: Check `apps/strapi/docker-compose.dev.yml` or the root `docker-compose.dev.yml` for the `LOCALSTACK_ENDPOINT` environment variable override. It should be set to `http://awslocal:4566`.

    **Step 4: Check Strapi logs for email send errors**

    ```bash
    docker logs $(docker ps -q --filter name=strapi) 2>&1 | grep -i "email\|ses\|error\|nodemailer" | tail -30
    ```

    Look for:
    - `ECONNREFUSED` -- transport cannot reach LocalStack
    - `MessageRejected` -- SES rejected the email (identity not verified)
    - `InvalidParameterValue` -- SDK API mismatch
    - Any error from `sendRegistrationEmail` or `sendForgotPasswordEmail`

    **Step 5: Send a test email to verify the full chain**

    Attempt to send a test email directly via LocalStack SES CLI:
    ```bash
    docker exec $(docker ps -q --filter name=awslocal) awslocal ses send-email \
      --from "no-reply@openvaa.org" \
      --destination "ToAddresses=test@test.com" \
      --message 'Subject={Data=Test},Body={Text={Data=Test body}}' \
      --region us-east-1
    ```

    Then check if it appears:
    ```bash
    curl -s http://localhost:4566/_aws/ses | python3 -m json.tool
    ```

    If the CLI email appears but Strapi emails don't, the issue is in Strapi's transport configuration. If even the CLI email doesn't appear, LocalStack SES itself is broken.

    **Step 6: If Strapi transport is the issue -- test email sending via Strapi API**

    Use curl to trigger a registration email through Strapi's Admin Tools endpoint and check the response:

    ```bash
    # Get admin JWT
    TOKEN=$(curl -s -X POST http://localhost:1337/admin/login \
      -H "Content-Type: application/json" \
      -d '{"email":"mock.admin@openvaa.org","password":"admin"}' | \
      python3 -c "import sys,json; print(json.load(sys.stdin)['data']['token'])")

    # Find an unregistered candidate
    CANDIDATE_RESULT=$(curl -s -X POST http://localhost:1337/openvaa-admin-tools/find-data \
      -H "Authorization: Bearer $TOKEN" \
      -H "Content-Type: application/json" \
      -d '"{ \"collection\": \"candidates\", \"filters\": { \"email\": { \"\\$eq\": \"test.unregistered@openvaa.org\" } } }"')
    echo "Find result: $CANDIDATE_RESULT"
    ```

    This helps determine if the Strapi Admin Tools API is responding correctly and the candidate exists with a valid registrationKey.

    **Step 7: Apply the fix based on diagnosis**

    The fix depends on the root cause found in steps 2-6. Common fixes:

    A. **Identity not verified**: Run `awslocal ses verify-email-identity` (step 2 fix). This is transient -- would need to be re-verified on each LocalStack restart. The init script should handle this, so also verify the init script runs correctly.

    B. **LOCALSTACK_ENDPOINT wrong inside Strapi container**: Fix the docker-compose override to set `LOCALSTACK_ENDPOINT: http://awslocal:4566`. Restart the stack for changes to take effect (ask user to run `yarn dev:down && yarn dev` if needed).

    C. **Nodemailer SES transport SDK mismatch**: The plugins.ts uses `require('@aws-sdk/client-ses')` and passes `{ ses: new aws.SES({...}), aws }` to nodemailer. If nodemailer expects a different shape, update the transport configuration per nodemailer SES transport docs.

    D. **Silent error in sendToAll**: The `sendToAll` function catches errors per-candidate. If all sends fail, it returns `{ type: 'success', sent: 0, errors: [...] }`. The test's `strapiAdminClient.sendEmail()` checks `body.type === 'failure'` but not `sent === 0`. If this is the issue, the fix is either in the test (check `sent > 0`) or in the backend (return failure if no emails sent successfully).

    E. **LocalStack API response format change**: If `fetchEmails()` returns an empty array but emails exist, the response format may have changed. Check the raw curl output from step 5 against what emailHelper.ts expects (`{ messages: [{ RawData: ... }] }`).

    After applying the fix, verify by running just the registration email test to confirm the email chain works:
    ```bash
    npx playwright test -c tests/playwright.config.ts --project=candidate-app-mutation -g "should send registration email"
    ```
