**## Package Purpose** (~5 lines)

**## Conventions** (~50 lines)
Write each convention as an imperative rule (DO/NEVER/ALWAYS). Include these 10 conventions:

1. **internal.ts barrel**: ALL intra-package imports use `../internal` (or deeper). NEVER import from `index.ts` or directly between source files. New exports go in `internal.ts` following the ordering: core -> root -> utils -> i18n -> questions base -> questions category -> questions variants -> elections -> constituencies -> entities base -> entities variants -> nominations base -> nominations variants.

2. **objectType discriminator**: Every concrete class has `readonly objectType = OBJECT_TYPE.ClassName`. When adding a new class, add it to `OBJECT_TYPE` in `core/objectTypes.ts`, add to `ObjectTypeMap`, and update `utils/typeGuards.ts`.

3. **Type guards over instanceof**: NEVER use `instanceof` for DataObject type checking. Use `isEntity()`, `isQuestion()`, `isChoiceQuestion()`, `isNomination()`, `isObjectType(obj, type)` from `utils/typeGuards.ts`. Historical instanceof bugs (commit 87efe19a) caused cross-module-boundary failures.

4. **Smart default values**: Property getters return empty defaults for missing data: `''` for strings, `null` for optional objects, `{}` for records, `Infinity` for order, `false` for boolean flags. Pattern: `get prop() { return this.data.prop ?? defaultValue; }`. NEVER use `MISSING_VALUE` in data objects.

5. **MISSING_VALUE is for matching only**: `MISSING_VALUE` from `@openvaa/core` is a matching-context sentinel. Data objects use `undefined` or empty literals. Question `normalizeValue()` returns `MISSING_VALUE` for values that should be imputed by the matching algorithm.

6. **DataRoot provision methods**: Objects are NEVER created directly with `new`. Always use DataRoot provision methods: `provideElectionData()`, `provideConstituencyData()`, `provideQuestionData()`, `provideEntityData()`, `provideNominationData()`. Data provision order matters: elections first, then constituencies, then entities+nominations / question categories+questions.

7. **Updatable transaction system**: Changes wrapped in `update(transaction)` track nesting and only fire `onUpdate()` at the outermost boundary. Failed transactions do NOT trigger updates. Use `subscribe(handler)` to listen for changes.

8. **Formatter system**: Use `DataRoot.setFormatter()` to customize string output for answers and entity names. Frontend overrides formatters in `dataContext.ts` for locale-aware formatting. Never hardcode name formatting -- use `root.formatCandidateName()`.

9. **Co-located tests**: Every `entity.ts` has a `entity.test.ts` in the same directory. Tests import from `../internal` (same as source files) plus `../../testUtils`. Test data comes from `getTestData()` and `getTestDataRoot()` in `testUtils/`.

10. **Localization via translate()**: Use `translate({ value, locale })` with `isLocalizedValue()` check. The function provides 3-tier fallback: requested locale -> first available key -> undefined. `LocalizedValue<T>` wraps any string-valued property. `TRANSLATIONS_KEY` marks translation objects.

**## Reviewing Data Package Changes** (~25 lines)
Write as a numbered checklist. Include these 8 review items:

1. No `instanceof` checks -- use type guards or `objectType` property
2. All imports within package use `../internal` (never `./index`, never direct file imports)
3. New exports added to `internal.ts` in correct position (base types before dependent types)
4. New concrete classes registered in `OBJECT_TYPE`, `ObjectTypeMap`, and `typeGuards.ts`
5. Smart defaults used for property getters (no bare `undefined` returns for optional string/record/array properties)
6. `MISSING_VALUE` not stored in data objects (only in matching/normalization context)
7. Tests co-located as `*.test.ts` next to source files, importing from `../internal`
8. Data provision order respected (elections -> constituencies -> entities+nominations / categories+questions)
