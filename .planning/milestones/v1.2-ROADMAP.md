## Phase Details

### Phase 15: Scaffold and Build
**Goal**: Frontend runs on Svelte 5 with a clean, modern build configuration
**Depends on**: Nothing (first phase of v1.2)
**Requirements**: SCAF-01, SCAF-02, SCAF-03, SCAF-04, SCAF-05, SCAF-06
**Success Criteria** (what must be TRUE):
  1. Running `yarn workspace @openvaa/frontend dev` starts a Svelte 5 dev server without errors
  2. Existing Svelte 4 component files compile and render correctly under the Svelte 5 compiler
  3. Path aliases ($types, $voter, $candidate) resolve correctly in imports
  4. Production build with adapter-node completes successfully
  5. No svelte-preprocess or PostCSS pipeline remains in the config
**Plans:** 2/2 plans complete
Plans:
- [ ] 15-01-PLAN.md — Replace config files with Svelte 5 scaffold, migrate CSS pipeline to @tailwindcss/vite
- [ ] 15-02-PLAN.md — Build verification, Svelte 4 compat fixes, vitest validation

### Phase 16: CSS Architecture
**Goal**: The frontend styling system uses Tailwind 4 CSS-first configuration with DaisyUI 5
**Depends on**: Phase 15
**Requirements**: CSS-01, CSS-02, CSS-03, CSS-04, CSS-05
**Success Criteria** (what must be TRUE):
  1. Tailwind utility classes generate from CSS `@theme` directives (no JS config file)
  2. DaisyUI 5 component classes render correctly with updated class names
  3. Custom theme values (spacing, fonts, colors, radii) produce the same visual output as before
  4. Dynamic colors from `staticSettings` apply correctly at runtime via CSS custom properties
  5. No silent color failures from renamed OKLCH variables
**Plans:** 2/2 plans complete
Plans:
- [ ] 16-01-PLAN.md — Rewrite app.css with DaisyUI 5 @plugin + @theme directives, delete JS config files
- [ ] 16-02-PLAN.md — Update all component OKLCH vars, DaisyUI class renames, opacity and viewport unit migrations

### Phase 17: Internationalization
**Goal**: Migrate from unmaintained sveltekit-i18n to Paraglide JS (inlang) with compile-time type-safe messages and a runtime override wrapper for backend translationOverrides
**Depends on**: Phase 15
**Requirements**: I18N-01, I18N-02, I18N-03, I18N-04, I18N-05, I18N-06, I18N-07
**Success Criteria** (what must be TRUE):
  1. Paraglide JS installed and configured with Vite plugin, compiling all message files
  2. All 740 translation call sites migrated from $t() to t() wrapper (Paraglide + override fallback)
  3. Switching language in the UI updates all visible text (page reload acceptable)
  4. Routes without an explicit locale parameter detect and apply the user's language via Paraglide strategy
  5. API routes handle locale correctly via Paraglide's reroute hook and middleware
**Plans:** 4/4 plans complete
Plans:
- [x] 17-01-PLAN.md — Merge fr/lb locales, hardcode DEFAULT_PAYLOAD_KEYS, convert translations to inlang format
- [x] 17-02-PLAN.md — Install Paraglide, create override wrapper, rewrite hooks/routing, update layout loaders
- [x] 17-03-PLAN.md — Migrate 740 call sites, update contexts/language switcher, tests, build verification
- [ ] 17-04-PLAN.md — Gap closure: sync fr/lb locale key structures to match en

### Phase 18: Dependency Modernization

**Depends on**: Phase 15
**Requirements**: DEP-01, DEP-02, DEP-03, DEP-04

  2. Unused dependencies (Capacitor packages, etc.) are removed from package.json files
  3. Pre-existing build failures (ai package, Strapi TS errors) are either resolved or explicitly documented with workarounds
  4. Yarn catalog entries reflect the updated dependency versions
**Plans:** 2/2 plans complete
Plans:
- [ ] 18-01-PLAN.md — Remove unused deps, expand Yarn catalog, bump all non-Strapi versions
- [ ] 18-02-PLAN.md — Build verification and fix-up after version bumps

**Depends on**: Phase 15, Phase 16, Phase 17, Phase 18

  3. CI pipeline (GitHub Actions) passes all checks on the migration branch

### Phase 20: OXC Toolchain Exploration
**Goal**: Evaluate whether oxlint and oxc-based formatting can replace ESLint + Prettier with equivalent or better lint coverage and faster performance
**Depends on**: Phase 19
**Requirements**: OXC-01, OXC-02, OXC-03
**Success Criteria** (what must be TRUE):
  1. oxlint rule coverage compared against current ESLint config — gaps and equivalences documented
  2. Performance benchmarks show lint time comparison (ESLint vs oxlint) on the monorepo
  3. A clear recommendation (migrate, defer, or reject) with rationale is documented
  4. If recommending migration: a concrete migration plan with breaking changes and workarounds is drafted
**Plans:** 1/1 plans complete
Plans:
- [ ] 20-01-PLAN.md — Benchmark ESLint vs oxlint, write evaluation report with rule coverage, recommendation
