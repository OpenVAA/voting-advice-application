# Requirements: OpenVAA v1.2 Svelte 5 Migration (Infrastructure)

**Defined:** 2026-03-15
**Core Value:** A reliable, well-tested VAA framework that developers can confidently extend, customize, and deploy for real elections.

## v1.2 Requirements

Requirements for the Svelte 5 infrastructure milestone. Each maps to roadmap phases.

### Scaffold & Build

- [x] **SCAF-01**: Frontend uses Svelte 5 + SvelteKit 2 (latest) with fresh scaffold configuration
- [x] **SCAF-02**: `svelte.config.js` uses native TypeScript support (no svelte-preprocess)
- [x] **SCAF-03**: `vite.config.ts` uses `@tailwindcss/vite` plugin (no PostCSS pipeline)
- [x] **SCAF-04**: Path aliases ($types, $voter, $candidate) preserved in new config
- [x] **SCAF-05**: adapter-node configured for production builds
- [x] **SCAF-06**: Existing Svelte 4 component syntax compiles and runs without changes

### CSS Architecture

- [x] **CSS-01**: Tailwind CSS 4 with CSS-first `@theme` configuration replaces JS config
- [x] **CSS-02**: DaisyUI 5 integrated with updated class names across all templates
- [x] **CSS-03**: Custom theme (spacing, fonts, colors, radii) migrated from JS to CSS `@theme` directives
- [x] **CSS-04**: Dynamic color system (`staticSettings`) works with CSS-first Tailwind 4
- [x] **CSS-05**: DaisyUI OKLCH variable renames applied (no silent color failures)

### Internationalization

- [x] **I18N-01**: i18n alternatives evaluated against OpenVAA's runtime translation loading pattern
- [x] **I18N-02**: If better alternative found, i18n library migrated with all translation call sites updated
- [x] **I18N-03**: Server hooks.ts cleaned of unnecessary i18n middleware if migrating
- [x] **I18N-04**: ICU/intl message format variables migrated to new library format
- [x] **I18N-05**: Language change in UI works (page reload acceptable if necessary)
- [x] **I18N-06**: Routes without locale param support automatic language detection
- [x] **I18N-07**: API routes work correctly with locale parameter handling

### Dependencies

- [x] **DEP-01**: All monorepo workspace dependencies updated to latest compatible versions
- [x] **DEP-02**: Unused dependencies removed (Capacitor packages, etc.)
- [x] **DEP-03**: Pre-existing build failures resolved or documented (ai package, Strapi TS errors)
- [x] **DEP-04**: Yarn catalog updated with new dependency versions

### OXC Toolchain

- [x] **OXC-01**: oxlint rule coverage compared against current ESLint config with gaps documented
- [x] **OXC-02**: Performance benchmarks comparing ESLint vs oxlint on the monorepo
- [x] **OXC-03**: Migration recommendation (migrate, defer, or reject) with rationale and plan

### Validation

- [x] **VAL-01**: Frontend builds successfully with `yarn build`
- [x] **VAL-02**: Docker dev stack starts and serves the frontend correctly
- [x] **VAL-03**: CI pipeline passes with updated dependencies
- [x] **VAL-04**: Existing E2E tests pass (or failures documented as pre-existing)

### Migration Cleanup

- [x] **CLEAN-01**: Dead code from i18n migration removed (paramStore('lang') block in dataContext.ts)
- [x] **CLEAN-02**: Migration-introduced TypeScript errors in non-Strapi workspaces resolved

## Future Requirements

Deferred to follow-up milestone (v1.3 Svelte 5 Content Migration).

### Content Migration

- **CONT-01**: All `export let` props migrated to `$props()` runes
- **CONT-02**: All `$:` reactive declarations migrated to `$derived()` / `$effect()`
- **CONT-03**: All `<slot>` usages migrated to `{@render}` snippets
- **CONT-04**: All `on:event` directives migrated to callback props
- **CONT-05**: All `svelte/store` usages evaluated for runes migration
- **CONT-06**: i18n translation package separation (if beneficial)

## Out of Scope

Explicitly excluded. Documented to prevent scope creep.

| Feature | Reason |
|---------|--------|
| Component content migration (runes, snippets) | Deferred to v1.3 — Svelte 4 syntax works in Svelte 5 |
| Route restructuring | Current route structure preserved |
| Backend changes | Strapi is unchanged in this milestone |
| New frontend features | Infrastructure only — no new user-facing features |
| SSR strategy changes | Preserve existing SSR/CSR behavior |
