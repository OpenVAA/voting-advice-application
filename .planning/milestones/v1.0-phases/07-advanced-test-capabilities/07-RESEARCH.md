# Phase 7: Advanced Test Capabilities - Research

**Researched:** 2026-03-11
**Domain:** Playwright visual regression testing and performance benchmarking
**Confidence:** HIGH

## Summary

Phase 7 adds two new testing capabilities to the existing Playwright E2E suite: visual regression screenshot comparison and page load performance budgets. Both are narrowly scoped -- visual covers 2 pages at 2 viewports (4 screenshots), performance covers 1 page -- and both use tag-based gating to stay excluded from the default `yarn test:e2e` run.

Playwright 1.58.2 (already installed) has mature built-in support for both capabilities. `toHaveScreenshot()` handles screenshot capture, pixel-diff comparison, baseline management, and `--update-snapshots` workflow out of the box. Performance measurement uses `page.evaluate()` with the Navigation Timing API and PerformanceObserver, requiring no additional dependencies.

**Primary recommendation:** Use Playwright's native `toHaveScreenshot()` with `fullPage: true`, `animations: 'disabled'`, and `mask` for dynamic content. Use Navigation Timing API (`domContentLoadedEventEnd`) for performance budgets. Exclude both capabilities from default runs via `grepInvert: [/@visual/, /@perf/]` in the root config.

### Pattern 1: Visual Regression with answeredVoterPage Fixture
**What:** Reuse the existing `answeredVoterPage` fixture for voter results screenshots, and navigate to candidate preview page with auth for that screenshot.
**When to use:** Both visual regression tests.
**Example:**
```typescript
// Source: Playwright docs + project patterns
import { voterTest } from '../../fixtures/voter.fixture';
import { expect } from '@playwright/test';
import { testIds } from '../../utils/testIds';

voterTest.describe('visual regression - voter results', { tag: ['@visual'] }, () => {
  voterTest('voter results page screenshot @visual', async ({ answeredVoterPage: page }) => {
    // Wait for stable rendering
    await page.waitForLoadState('networkidle');

    await expect(page).toHaveScreenshot('voter-results-desktop.png', {
      fullPage: true,
      animations: 'disabled',
      mask: [
        // Mask dynamic content like avatars/timestamps
      ],
      maxDiffPixelRatio: 0.01,
      threshold: 0.2
    });
  });
});
```

### Pattern 2: Performance Budget with Navigation Timing
**What:** Use `page.evaluate()` to extract Navigation Timing API data after the voter results page loads.
**When to use:** Performance budget test for results page.
**Example:**
```typescript
// Source: Checkly docs + Playwright docs
import { voterTest } from '../../fixtures/voter.fixture';
import { expect } from '@playwright/test';

voterTest.describe('performance budget - voter results', { tag: ['@perf'] }, () => {
  voterTest('results page loads within time budget @perf', async ({ answeredVoterPage: page }) => {
    // Extract navigation timing from the browser
    const timing = await page.evaluate(() => {
      const entries = performance.getEntriesByType('navigation') as PerformanceNavigationTiming[];
      const nav = entries[0];
      return {
        domContentLoaded: nav.domContentLoadedEventEnd,
        loadEvent: nav.loadEventEnd,
        domInteractive: nav.domInteractive,
        duration: nav.duration
      };
    });

    // Assert within budget (calibrated to Docker dev mode)
    expect(timing.domContentLoaded).toBeLessThan(5000); // 5s budget for dev mode
  });
});
```

### Pattern 3: Tag-Based Gating with grepInvert
**What:** Add `grepInvert` to root config so `@visual` and `@perf` tests are excluded by default.
**When to use:** Playwright config modification.
**Example:**
```typescript
// In playwright.config.ts root config:
export default defineConfig({
  // ... existing config
  grepInvert: [/@visual/, /@perf/],
  // ...
});
```

### Anti-Patterns to Avoid
- **Do NOT use separate Playwright projects for visual/perf tests.** These tests use the same data-setup dependency as existing tests. Adding new projects would complicate the already complex dependency chain. Instead, place spec files in `specs/visual/` and `specs/perf/` subdirectories and use tag-based inclusion/exclusion.
- **Do NOT use `devices['iPhone 14']` for mobile viewport.** iPhone 14 defaults to WebKit browser, but the entire test suite runs on Chromium. Use `devices['Pixel 7']` (Chromium, 412x839 viewport) instead, or define a custom viewport on a Chromium-based config to keep screenshots comparable.
- **Do NOT set tight performance budgets.** Docker dev mode has significant overhead compared to production. A 3s Time to Interactive budget will flake. Use generous budgets (5-8s range) and treat as smoke-level regression detection, not absolute performance guarantees.
- **Do NOT run performance tests multiple times and average.** Single-run with a generous budget is more practical than multi-run averaging for a dev environment test.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Screenshot comparison | Custom pixel diff logic | `toHaveScreenshot()` | Handles retries, threshold tuning, baseline management, OS-specific snapshots |
| Baseline update workflow | Custom CLI for updating baselines | `--update-snapshots` flag | Built into Playwright, no custom tooling needed |
| Performance timing | Custom timing wrappers | `performance.getEntriesByType('navigation')` | Browser-native API, no external dependencies |
| Dynamic content masking | Manual DOM manipulation before screenshot | `mask` option on `toHaveScreenshot()` | Overlay approach is simpler and doesn't modify DOM state |

**Key insight:** Playwright 1.58.2 has everything needed for both visual and performance testing built-in. Zero new dependencies.

## Common Pitfalls

### Pitfall 1: Platform-Specific Screenshot Baselines
**What goes wrong:** Screenshots generated on macOS (developer machine) differ from Linux (CI) due to font rendering, sub-pixel anti-aliasing, and system fonts.
**Why it happens:** `toHaveScreenshot()` generates platform-specific snapshot paths by default using `{platform}` token.
**How to avoid:** Use `snapshotPathTemplate` that includes `{platform}` in the path. Generate separate baselines for each platform. For this project, since visual tests are CI-optional (non-blocking), generate baselines on the developer's platform and accept that CI may differ. Alternatively, use a custom `stylePath` to normalize fonts.
**Warning signs:** Tests pass locally but fail on CI with pixel diff errors. The Playwright HTML report shows visual diff overlay.

### Pitfall 2: Mobile Viewport Browser Mismatch
**What goes wrong:** Using `devices['iPhone 14']` sets `defaultBrowserType: 'webkit'`, but the test infrastructure only installs Chromium.
**Why it happens:** Playwright device presets include the "correct" browser for that device. iPhone presets default to WebKit/Safari.
**How to avoid:** Use `devices['Pixel 7']` (Chromium, 412x839) for mobile viewport, or create a custom viewport config spreading over `devices['Desktop Chrome']` with overridden viewport dimensions: `{ ...devices['Desktop Chrome'], viewport: { width: 390, height: 844 } }`.
**Warning signs:** Test fails with "browser not installed" or produces WebKit-rendered screenshots that look different from Chromium ones.

### Pitfall 3: Animations and Transitions Causing Flaky Screenshots
**What goes wrong:** CSS transitions or loading animations are captured mid-frame, producing different screenshots on each run.
**Why it happens:** Even with `animations: 'disabled'`, some JS-driven animations or lazy-loaded images may not be caught.
**How to avoid:** Always combine `animations: 'disabled'` with `waitForLoadState('networkidle')`. Wait for specific elements to be visible before screenshot. Use `mask` for any content that has shimmer/skeleton loading states.
**Warning signs:** Screenshots differ by a few pixels in animation-heavy areas. `maxDiffPixelRatio` catches it but masks the root cause.

### Pitfall 4: Performance Metrics Variance in Docker Dev Mode
**What goes wrong:** Performance metrics fluctuate wildly between runs due to Docker resource contention, SvelteKit dev mode overhead (no code splitting, no minification), and hot-reloading watchers consuming CPU.
**Why it happens:** Dev mode is not production-representative. Docker adds virtualization overhead. Other containers (postgres, localstack) compete for resources.
**How to avoid:** Set generous budgets (5-8s for DOMContentLoaded, 10-15s for full load). Use the test as a regression detector (did it get 2x slower?) rather than an absolute benchmark. Consider a "warn" threshold below the "fail" threshold for early warning.
**Warning signs:** Perf test passes locally but flakes on CI. Budget values need frequent adjustment.

### Pitfall 5: answeredVoterPage Fixture Timing for Performance Measurement
**What goes wrong:** The `answeredVoterPage` fixture navigates through the entire voter journey (16 questions) and lands on results. Navigation Timing API gives the timing for the last navigation (results page load), but the fixture's own navigation to results may interleave with timing data.
**Why it happens:** `answeredVoterPage` uses `page.waitForURL(/\/results/)` which triggers on the auto-advance redirect. The navigation timing reflects this redirect, not a clean page load.
**How to avoid:** For performance tests, after the fixture lands on results, do a fresh `page.reload()` or navigate directly to the results URL (if bookmarkable). This gives clean navigation timing. The trade-off is slightly longer test time for an accurate measurement.
**Warning signs:** Performance metrics are unreliable or show sub-100ms DOMContentLoaded times (measuring a client-side route transition, not a page load).

### Pitfall 6: grepInvert Blocks Explicit --grep Execution
**What goes wrong:** `grepInvert: [/@visual/, /@perf/]` in config excludes `@visual` and `@perf` tags from all runs, including `--grep @visual`.
**Why it happens:** `grepInvert` and `--grep` work independently. If `grepInvert` excludes `@visual` in the config, `--grep @visual` selects them, but `grepInvert` then removes them. Net result: zero tests run.
**How to avoid:** The `--grep` CLI flag overrides `grepInvert` in Playwright 1.58+ when passed explicitly. VERIFY THIS BEHAVIOR before relying on it. Alternative approach: don't use `grepInvert` at all -- instead, place visual/perf specs in a separate `testDir` that isn't matched by any default project, and add dedicated projects that are only activated when explicitly targeted. Or simpler: add `testIgnore` patterns to existing projects to skip `specs/visual/` and `specs/perf/` directories.
**Warning signs:** `playwright test --grep @visual` reports "0 tests found."

## Code Examples

### Complete Visual Regression Spec Structure
```typescript
// tests/tests/specs/visual/visual-regression.spec.ts
// Source: Playwright docs + project convention patterns

import { voterTest } from '../../fixtures/voter.fixture';
import { test, expect } from '../../fixtures';
import { buildRoute } from '../../utils/buildRoute';
import { testIds } from '../../utils/testIds';
import { STORAGE_STATE } from '../../../playwright.config';
import { devices } from '@playwright/test';

// --- Voter Results Page: Desktop ---
voterTest.describe('visual regression - voter results desktop', { tag: ['@visual'] }, () => {
  voterTest('voter results page - desktop viewport', async ({ answeredVoterPage: page }) => {
    await page.waitForLoadState('networkidle');
    // Wait for results list to be fully rendered
    await page.getByTestId(testIds.voter.results.list).waitFor({ state: 'visible' });

    await expect(page).toHaveScreenshot('voter-results-desktop.png', {
      fullPage: true,
      animations: 'disabled',
      maxDiffPixelRatio: 0.01,
      threshold: 0.2
    });
  });
});
```

### Performance Budget Test
```typescript
// tests/tests/specs/perf/performance-budget.spec.ts
import { voterTest } from '../../fixtures/voter.fixture';
import { expect } from '@playwright/test';

voterTest.describe('performance budget - voter results', { tag: ['@perf'] }, () => {
  voterTest.setTimeout(60000); // Generous timeout for fixture + reload

  voterTest('voter results page loads within time budget', async ({ answeredVoterPage: page }) => {
    // Reload to get clean navigation timing (fixture's auto-advance is not a fresh navigation)
    await page.reload({ waitUntil: 'load' });

    const timing = await page.evaluate(() => {
      const entries = performance.getEntriesByType('navigation') as PerformanceNavigationTiming[];
      const nav = entries[0];
      return {
        domContentLoaded: Math.round(nav.domContentLoadedEventEnd),
        loadComplete: Math.round(nav.loadEventEnd),
        domInteractive: Math.round(nav.domInteractive),
        duration: Math.round(nav.duration)
      };
    });

    // Budget: Docker dev mode with 11 candidates + parties
    // These are regression detection thresholds, not production targets
    expect(timing.domContentLoaded).toBeLessThan(8000);  // 8s hard fail
    expect(timing.loadComplete).toBeLessThan(15000);       // 15s hard fail
  });
});
```

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|

| `globalSetup` for data | Project dependencies pattern | Playwright 1.35+ (used since Phase 1) | Already established in this project |

**Deprecated/outdated:**
- `toMatchSnapshot()` for screenshots: Still works but `toHaveScreenshot()` is recommended (auto-retry for rendering stability)
- `snapshotDir` config property: Deprecated in favor of `snapshotPathTemplate`

## Open Questions

1. **grepInvert + --grep interaction in Playwright 1.58.2**
   - What we know: Playwright docs say CLI `--grep` should work alongside `grepInvert`, but interaction semantics are not explicit
   - What's unclear: Whether `--grep @visual` correctly overrides `grepInvert: [/@visual/]` in all cases
   - Recommendation: Test this during implementation. If it does NOT work, use `testIgnore` patterns on existing projects instead of `grepInvert`. Add `specs/visual/` and `specs/perf/` directories to every existing project's `testIgnore`.

2. **Performance budget calibration**
   - What we know: Docker dev mode is significantly slower than production (no bundling, no code splitting, hot-reload watchers)
   - What's unclear: Actual DOMContentLoaded timing for the voter results page in Docker dev mode
   - Recommendation: Run an initial calibration during Plan 07-02 implementation. Measure 3-5 runs, take the P90, add 50% margin as the budget. Document the calibration run results in the spec file comments.

3. **Cross-platform screenshot baselines**
   - What we know: Screenshots differ between macOS and Linux due to font rendering
   - What's unclear: Whether the visual tests will run on CI (Linux) at all, or only locally
   - Recommendation: Since CI runs are non-blocking/informational, start with developer-generated baselines (macOS). If CI visual test failures are too noisy, add platform-specific paths using `{platform}` token in `snapshotPathTemplate`.
