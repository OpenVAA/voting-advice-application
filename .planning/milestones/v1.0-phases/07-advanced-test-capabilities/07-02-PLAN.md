**Test: Voter results page load timing**
- Tag: `{ tag: ['@perf'] }` on `voterTest.describe()`
- Set generous timeout: `voterTest.setTimeout(60000)` (fixture navigation + reload + measurement)
- Use `answeredVoterPage` fixture to navigate through voter journey to results
- CRITICAL (Pitfall 5): Do a fresh `await page.reload({ waitUntil: 'load' })` AFTER the fixture lands on results. This gives clean Navigation Timing data for a full page load, not the client-side route transition that the fixture's auto-advance produces.
- Extract timing via `page.evaluate()`:
  ```typescript
  const timing = await page.evaluate(() => {
    const entries = performance.getEntriesByType('navigation') as PerformanceNavigationTiming[];
    const nav = entries[0];
    return {
      domContentLoaded: Math.round(nav.domContentLoadedEventEnd),
      loadComplete: Math.round(nav.loadEventEnd),
      domInteractive: Math.round(nav.domInteractive),
      duration: Math.round(nav.duration),
      ttfb: Math.round(nav.responseStart - nav.requestStart)
    };
  });
  ```
- Assert budgets calibrated to Docker dev mode (generous -- regression detection, not absolute targets):
  - `expect(timing.domContentLoaded).toBeLessThan(8000)` -- 8s DOMContentLoaded
  - `expect(timing.loadComplete).toBeLessThan(15000)` -- 15s full load
- Add JSDoc comment block at top of file documenting:
  - What these budgets measure (Docker dev mode, not production)
  - How to update budgets (run 3-5 times, take P90, add 50% margin)
  - That these are regression detectors (if results page suddenly takes 30s, something broke)
- Log timing values to console for calibration visibility: `console.log('Performance timing:', timing)`
- The test is a hard fail (not warn) because the budgets are generous enough that exceeding them indicates a real regression, not normal variance.

The spec should be concise -- a single test.describe with a single test case. No multi-run averaging (per research: "single-run with a generous budget is more practical").

  <name>Task 2: Add optional visual/perf CI job to GitHub Actions workflow</name>
  <files>.github/workflows/main.yaml</files>
  <action>
Add an optional (non-blocking) CI job for visual and performance tests to `.github/workflows/main.yaml`.

Add a new job `e2e-visual-perf` after the existing `e2e-tests` job:

1. **Job configuration:**
   - `runs-on: ubuntu-latest`
   - `continue-on-error: true` -- this makes the job non-blocking (failures don't fail the PR check)
   - `needs: []` -- runs independently (does NOT depend on `e2e-tests` job; it spins up its own Docker stack)
   - Or alternatively, to avoid running two Docker stacks: `needs: [e2e-tests]` -- but this means visual/perf only runs after the full E2E suite passes, which is acceptable since it's informational

   Decision: Use `needs: [e2e-tests]` to avoid duplicate Docker stack setup. The visual/perf job reuses the same stack lifecycle pattern but runs after e2e-tests.

   Actually, looking at the workflow more carefully: the `e2e-tests` job does `yarn dev:down` at the end. So a dependent job cannot reuse the stack. Each job gets its own runner.

   Final decision: Make `e2e-visual-perf` an independent job (no `needs`) with its own Docker stack setup, and `continue-on-error: true` so it never blocks merging. This allows it to run in parallel with the main e2e-tests job.

2. **Job steps** (mirror the `e2e-tests` job setup, then run visual/perf):
   - Checkout, setup Yarn 4.6, setup Node.js 20.18.1, install dependencies
   - Install Playwright: `yarn playwright install --with-deps`
   - Disable mock data generation (same as e2e-tests): `sed -i 's/GENERATE_MOCK_DATA_ON_INITIALISE=true/GENERATE_MOCK_DATA_ON_INITIALISE=false/' .env.example`
   - Start OpenVAA: `yarn dev:start`
   - Collect Docker logs on failure (same pattern as e2e-tests)
   - Run visual and perf tests: `yarn playwright test -c ./tests/playwright.config.ts --grep "@visual|@perf"`
   - Upload report artifact with distinct name: `playwright-report-visual-perf`
   - Stop OpenVAA: `yarn dev:down` (always)

3. **Important details:**
   - The visual tests need baseline screenshots. On the first CI run on Linux, baselines won't exist (they're generated on macOS by the developer). The test will fail with "missing baseline" which is expected and non-blocking due to `continue-on-error: true`. This is by design -- the developer generates baselines locally and commits them.
   - Alternatively, if `snapshotPathTemplate` includes `{platform}`, Linux baselines would need to be generated separately. Per the research, start without platform-specific paths and add if needed.
   - The `--grep "@visual|@perf"` flag works because no existing project's `testMatch` or `testDir` scoping catches these files, and the grep selects tests by their tag. Since we did NOT use `grepInvert` (per Plan 07-01), there's no conflict.

4. **Do NOT modify the existing `e2e-tests` job.** It continues to run `yarn test:e2e` which already excludes visual/perf specs by directory scoping.
