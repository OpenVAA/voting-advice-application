### Select.svelte type fix

The issue: `SelectProps` extends `SvelteHTMLElements['select']` which types restProps as select-element attributes. But when `autocomplete === 'on'`, restProps are spread onto an `<input>` element (line 261: `{...concatClass(restProps, ...)}`), and when there's only 1 option, onto a `<div>` (line 251). This creates type mismatches.

Fix: Change `SelectProps` base type from `SvelteHTMLElements['select']` to `SvelteHTMLElements['div']`. This is the most permissive common denominator since:
- div attributes are a subset that works on all three elements
- The component already handles select-specific attributes (`name`, `value`, `autocomplete`) as explicit named props
- `SvelteHTMLElements['div']` covers class, style, id, aria-*, data-*, and other universal HTML attributes

In `Select.type.ts`:
- Change `export type SelectProps = SvelteHTMLElements['select'] & {` to `export type SelectProps = SvelteHTMLElements['div'] & {`

In `Select.svelte`:
- No template changes needed. The restProps will now be typed as div attributes which is compatible with div, input, and select elements.
- Verify the `<select>` element at line 310 still works -- it receives restProps via `concatClass(restProps, ...)` which spreads class and standard HTML attributes, which is fine.
