```typescript

// Definitions match Svelte's internal types exactly.
```

**Fix 3: i18nContext.type.ts (1 error)**

**Fix 4: buildRoute.ts (2 errors)**
Two type mismatches from SvelteKit/Paraglide generated types being stricter than the dynamic route-building pattern needs.

- Line 57: `resolveRoute` expects a specific route pattern union, but receives a dynamic string from `ROUTE[route]`.
  Change FROM: `let url = resolveRoute(routeId, flattenParams(routeParams));`
  Change TO: `let url = resolveRoute(routeId as string, flattenParams(routeParams));`
  Rationale: The `ROUTE` constant values are valid SvelteKit route IDs, but the `typeof ROUTE[keyof typeof ROUTE]` union doesn't match the generated `RouteId` union type exactly (different string literal widths). The cast is safe because all values in `ROUTE` are valid route patterns.

  Rationale: The `locale` property in `RouteOptions` is typed as `string` (any locale), but the Paraglide runtime may generate a more specific type. Since `localizeHref`'s published type accepts `locale?: string | undefined`, this assertion aligns the types. The `as string` is technically a no-op but satisfies the compiler when the generated runtime types are stricter.
