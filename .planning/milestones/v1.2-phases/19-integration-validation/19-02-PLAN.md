  <name>Task 1: Docker stack verification with Node 22</name>
  <files>docker-compose.dev.yml</files>
  <read_first>
    docker-compose.dev.yml
    apps/frontend/docker-compose.dev.yml
    apps/strapi/docker-compose.dev.yml
    apps/frontend/Dockerfile
    apps/strapi/Dockerfile
  </read_first>
  <action>
    **Clean start Docker stack with rebuilt images (no cache to pick up Node 22 base):**

    1. Run `yarn dev:down` to clean up any existing containers/volumes
    2. Run `docker compose -f docker-compose.dev.yml build --no-cache` to force rebuild with the new node:22-alpine base image. This is necessary because Docker may cache the old node:20.18.1-alpine layers.
    3. Run `yarn dev:start` which runs `turbo run build --filter='./packages/*' && docker compose -f docker-compose.dev.yml up -d --force-recreate --build --wait`
    4. Wait for all services to become healthy (the --wait flag handles this)

    **Verify all 4 services are running:**
    - `docker compose -f docker-compose.dev.yml ps` -- all 4 services should show "Up" and "healthy"
    - Frontend at http://localhost:5173 -- `curl -s -o /dev/null -w "%{http_code}" http://localhost:5173` should return 200
    - Strapi admin at http://localhost:1337/admin -- `curl -s -o /dev/null -w "%{http_code}" http://localhost:1337/admin` should return 200
    - Postgres -- `docker compose -f docker-compose.dev.yml exec postgres pg_isready` should succeed
    - LocalStack -- `curl -s -o /dev/null -w "%{http_code}" http://localhost:4566/_localstack/health` should return 200

    **If corepack fails during Docker build** (see Research pitfall 2):
    - Try changing `corepack prepare yarn@${YARN_VERSION}` to `corepack install` in the failing Dockerfile
    - Or add `--activate` flag: `corepack prepare yarn@${YARN_VERSION} --activate`
    - Re-run the build

    **If any service fails to start:**
    - Check logs: `docker compose -f docker-compose.dev.yml logs <service-name>`
    - Fix the issue (update Dockerfile, docker-compose config, etc.)
    - Re-run the start command

    Record results for VALIDATION-REPORT.md: which services started, any fixes applied, any issues encountered.
  </action>
  <verify>
    <automated>docker compose -f docker-compose.dev.yml ps --format "table {{.Service}}\t{{.Status}}" 2>/dev/null | head -10 && curl -s -o /dev/null -w "Frontend: %{http_code}\n" http://localhost:5173 && curl -s -o /dev/null -w "Strapi: %{http_code}\n" http://localhost:1337/admin</automated>
  </verify>
  <acceptance_criteria>
    - `docker compose -f docker-compose.dev.yml ps` shows 4 services running (frontend, strapi, postgres, awslocal)
    - `curl http://localhost:5173` returns HTTP 200
    - `curl http://localhost:1337/admin` returns HTTP 200
    - No services in "Exit" or "Restarting" state
  </acceptance_criteria>

    **Prerequisites:** Docker stack must be running and healthy from Task 1.

    **Step 1: Run core E2E test suite**
    Run `yarn test:e2e` which executes `playwright test -c ./tests/playwright.config.ts ./tests`.

    This runs all 18 spec files across all Playwright projects (data-setup, auth-setup, candidate-app, voter-app, variants, etc.).

    **If tests fail:**
    - Read the Playwright test output to identify failing tests
    - For each failure, determine if it is migration-caused or pre-existing:
      - Migration-caused: Fix the underlying issue (likely related to Paraglide i18n changes, CSS class renames, or import path changes)
      - Pre-existing: Fix if the fix takes < 30 minutes; otherwise document the failure for VALIDATION-REPORT.md
    - Re-run `yarn test:e2e` after fixes until all tests pass (or all remaining failures are documented pre-existing)

    **Step 2: Save "before" visual baselines**

    - voter-results-mobile.png

    Run: `PLAYWRIGHT_VISUAL=1 npx playwright test -c tests/playwright.config.ts --grep @visual --update-snapshots`
    This generates new baselines reflecting the CSS architecture changes from Phase 16 (DaisyUI 5 + TW4).

    **Step 4: Create before/after comparison**
    Copy the new baselines alongside the old ones for easy comparison:

    Run: `PLAYWRIGHT_PERF=1 npx playwright test -c tests/playwright.config.ts --grep @perf`
    If performance tests fail due to threshold violations from CSS migration changes, adjust the thresholds in the perf spec file. The current budgets are generous regression detectors (8s DOMContentLoaded, 15s full load), not targets.

    **Note on visual baseline platform dependency (Research pitfall 3):**
