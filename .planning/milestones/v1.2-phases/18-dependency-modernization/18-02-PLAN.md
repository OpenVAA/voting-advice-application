**Step 1: Run initial build**

Run `yarn build` from the repo root. This builds all workspaces via Turborepo. Capture the output to identify which workspaces fail and why.

**Step 2: Fix Vite 6 issues in frontend (if any)**

Known Vite 6 changes that may affect this project:
- `commonjsOptions.strictRequires` now defaults to `true` -- may affect CJS interop. If builds fail with CJS-related errors, add `build: { commonjsOptions: { strictRequires: false } }` to `apps/frontend/vite.config.ts`
- Range braces in globs no longer supported in `import.meta.glob` -- search for `import.meta.glob` patterns in `apps/frontend/src/` that use `{` brace ranges (NOT brace alternatives which are fine). If found, rewrite to array syntax
- CSS minification enabled by default for SSR builds -- should not cause issues

The frontend `vite.config.ts` uses standard SvelteKit + Tailwind Vite plugins and should work as-is with Vite 6. If no errors, no changes needed.

**Step 3: Fix @sveltejs/vite-plugin-svelte v5 issues (if any)**

vite-plugin-svelte v5 has these known changes:
- Requires `vite >= 6.0.0` (already done in Plan 01)
- Requires `svelte >= 5.0.0` (already at ^5.53.12)
- If v5 changes any config APIs, update `vite.config.ts` accordingly

**Step 4: Fix jsdom v26 issues (if any)**

jsdom 26 may have breaking changes affecting vitest test environment. If test:unit fails with jsdom errors, check if the vitest config specifies `environment: 'jsdom'` and update accordingly.

**Step 5: Fix intl-messageformat v11 issues (if any)**

intl-messageformat v11 changed from `new IntlMessageFormat(msg)` to potentially different API. Search `apps/frontend/src/` for `IntlMessageFormat` or `intl-messageformat` imports. If the API changed, update call sites. The main usage is in the Paraglide override wrapper for runtime translations.

**Step 6: Fix isomorphic-dompurify v3 issues (if any)**

isomorphic-dompurify v3 may have changed exports. Search `apps/frontend/src/` for `isomorphic-dompurify` imports. If the default export changed to named export (or vice versa), update import statements.

**Step 7: Fix jose v6 issues (if any)**

jose v6 is a major bump. Search `apps/frontend/src/` for `jose` imports. If APIs like `jwtVerify`, `SignJWT`, etc. changed signatures, update call sites. jose v6 dropped Node.js web crypto polyfill (not needed in Node 20+).

**Step 8: Fix svelte-eslint-parser v1 issues (if any)**

svelte-eslint-parser jumped from ^0.43.0 to ^1.6.0. This should be transparent since it's used by eslint-plugin-svelte internally. If lint scripts fail, check compatibility.

**Step 9: Handle LLM package build (ai stays at v5)**

The LLM package uses `ai@^5.0.0` which was NOT bumped. It should continue to build successfully. If the build fails for OTHER reasons (e.g., zod v4 catalog bump), fix the type issues. Per research, the LLM package currently builds successfully on ai@5 -- this step is a verification, not an expected fix.

If the LLM package build fails and the fix is non-trivial (more than ~10 minutes), add `"//build-issue"` comment to `packages/llm/package.json` documenting the issue, and exclude it from the Turborepo build by adding to turbo.json if needed. Document in the SUMMARY.

**Step 10: Handle Strapi TS errors in generateMockData.ts**

Pre-existing TS errors exist in `apps/strapi/src/util/generateMockData.ts`. Fix ONLY to the extent needed for `yarn workspace @openvaa/strapi build` to succeed. Strategy:
- If errors are type-narrowing issues: add `as` type assertions
- If errors are missing property issues: add `// @ts-expect-error` with explanation comment
- Do NOT refactor the mock data generation logic
- Do NOT fix TS errors in other Strapi files unless they also block build

**Step 11: Run yarn build again after all fixes**

Run `yarn build` to confirm all workspaces build successfully. All packages (core, data, matching, filters, app-shared, llm, argument-condensation, question-info) and all apps (frontend, strapi, docs, strapi-admin-tools) must complete.
