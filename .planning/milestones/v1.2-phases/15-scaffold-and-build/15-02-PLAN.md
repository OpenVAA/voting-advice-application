**Step 1: Build all workspace dependencies first**

Run `yarn build` from the repo root to ensure all @openvaa/* packages are built (the frontend imports from them and needs their .js output). This uses Turborepo caching so it will be fast if packages are already built.

**Step 2: Attempt dev server startup**

Run `cd apps/frontend && npx vite dev --port 5199` in background. Wait for startup (watch for "ready in" or "Local:" output). Check for:
- Module resolution errors for `@openvaa/*` imports -- if these appear, re-add `vite-tsconfig-paths` to `vite.config.ts` (`import viteTsConfigPaths from 'vite-tsconfig-paths'` and add `viteTsConfigPaths()` after `sveltekit()` in plugins array) and add `"vite-tsconfig-paths": "^4.3.2"` back to devDependencies, then `yarn install` and retry
- Svelte compilation errors (CompileError) -- these are the hard failures to fix
- Tailwind/CSS errors -- check @config resolution, @source inline() syntax

Kill the dev server after verifying startup succeeds (or after diagnosing failures).

**Step 3: Attempt production build**

Run `yarn workspace @openvaa/frontend build`. This executes `svelte-kit sync && vite build && ([ -d "data" ] && mkdir -p build/data && cp -r data/ build/ || true)`.

Analyze the output:
- **Deprecation warnings** (export let, on:event, $:, createEventDispatcher, slot): Expected and OK. Leave visible per user decision.
- **CompileError / hard failures**: These must be fixed. Count them.
- **Pre-existing failures** (ai package LanguageModelUsage, etc.): These are known from STATE.md and are NOT caused by this migration. Document but do not fix.

**Step 4: Fix compilation blockers (if any, max 5 files)**

For each hard CompileError:
1. Read the file and the error message
2. Apply the simplest Svelte 5-compatible fix
3. Add a `// TODO[Svelte 5]: {description of what was changed and why}` comment
4. Common fixes:
   - Invalid HTML nesting (e.g., `<div>` inside `<p>`): Change the outer element to a valid container
   - `<svelte:component>` with `on:` handler: Convert to callback prop pattern if needed
   - Any other hard error: Use the simplest fix that makes it compile

If MORE than 5 files need fixing: STOP. Document the list of files and errors. Flag this as a risk that requires replanning. Do not attempt to fix them all.

**Step 5: Re-verify build after fixes**

If any fixes were applied in Step 4, run `yarn workspace @openvaa/frontend build` again to confirm all CompileErrors are resolved. Deprecation warnings are expected and acceptable.

**Step 6: Handle DaisyUI 4 via @config**

If the build succeeds but you notice warnings about DaisyUI not loading (e.g., missing theme colors, unstyled components), the @config bridge may not be loading plugins correctly. In that case:
- Check if `@plugin "daisyui"` in CSS works as a fallback (despite being a DaisyUI 5 pattern, DaisyUI 4 may support it)
- If neither works, document the issue for Phase 16 but do NOT change tailwind.config.mjs
