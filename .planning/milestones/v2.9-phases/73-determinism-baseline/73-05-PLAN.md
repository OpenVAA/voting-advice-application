### Setup-hook race patterns (specific to Plan 5)

Setup hooks (`tests/tests/setup/*.setup.ts`) run BEFORE tests and create the deterministic precondition. Race patterns here are particularly leveraged:

**auth-cookie race (auth.setup.ts):**
```typescript
// BEFORE — likely shape (from Plan 1 inventory's auth.setup.ts 2-cond classification)
await page.fill('input[name="email"]', email);
await page.click('button[type="submit"]');
if (await page.getByText('Logged in').isVisible()) {
  await context.storageState({ path: '.auth/user.json' });
}
// AFTER — wait for the auth API response, not the visual confirmation
await page.fill('input[name="email"]', email);
await page.click('button[type="submit"]');
await page.waitForResponse(resp => resp.url().includes('/auth/v1/token') && resp.status() === 200);
await page.waitForURL(/\/(home|dashboard|results)/, { timeout: 10000 });
await context.storageState({ path: '.auth/user.json' });
```

The race here is between (a) the visual "Logged in" text and (b) the actual cookie being set in the response. Use `waitForResponse` for the API contract; cookie-set is implicit in a 200.

**data-seeding race (data.setup.ts):**
The 5 conditionals likely guard "if data already seeded, skip" or "if seed data N rows, then assert M rows on second seed". Make seeds idempotent: the setup hook should set up state regardless of prior runs. Replace `if (existsAlready) return;` with explicit cleanup-before-seed.

**variant-setup race (variant-*.setup.ts × 3):**
Each variant has 1 conditional. Likely: `if (variant === 'X') applyXFixture();` — hoist to module-level (Pattern 4 canonical 3) since the variant identity is determined by the project name, not by per-test logic.
