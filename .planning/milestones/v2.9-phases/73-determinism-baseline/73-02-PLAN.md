### Pattern: networkidle → element-state wait (RESEARCH §"Pattern 2")

```typescript
// BEFORE — voter-static-pages.spec.ts:99-100 (current)
await page.goto(buildRoute({ route: 'Nominations', locale: 'en' }));
await page.waitForLoadState('networkidle');
await expect(page.getByTestId(testIds.voter.nominations.container)).toBeVisible({ timeout: 15000 });

// AFTER — drop networkidle; rely on the element-state wait that follows
await page.goto(buildRoute({ route: 'Nominations', locale: 'en' }));
await expect(page.getByTestId(testIds.voter.nominations.container)).toBeVisible({ timeout: 15000 });
```

When the line that follows the networkidle wait already has a `toBeVisible({ timeout })` or a `waitFor({ state, timeout })` against the asserted element, simply DELETE the networkidle line. When there is no follow-up element wait, ADD one against the dominant page element (use the testId registry: `testIds.voter.results.list`, `testIds.voter.nominations.container`, `testIds.candidate.profile.container`, etc.).

### Pattern: raw-locator → semantic locator (RESEARCH §"Pattern 3")

Priority order:
1. `getByRole({ name: ... })` — accessibility-first
2. `getByText(...)` — what users see
3. `getByLabel(...)` — form-associated
4. `getByPlaceholder(...)`
5. `getByAltText(...)`
6. `getByTitle(...)`
7. `getByTestId(...)` — last resort, ONLY when (a) a registered testId exists in `tests/tests/utils/testIds.ts` AND (b) no role-based alternative is meaningful

```typescript
// BEFORE — voter-results.spec.ts:181 (current)
const firstCheckbox = page.locator('dialog input[type="checkbox"]').first();

// AFTER — combine getByRole('dialog') + getByRole('checkbox')
const firstCheckbox = page.getByRole('dialog').getByRole('checkbox').first();
```

```typescript
// BEFORE — voter-results.spec.ts:221 (current)
const dialog = page.locator('dialog[open]');

// AFTER — implicit "open" via role
const dialog = page.getByRole('dialog');
await expect(dialog).toBeVisible(); // role-based assertion subsumes "open"
```

```typescript
// BEFORE — voter-results.spec.ts:291 (current — filter chip color check)
.filter({ has: page.locator('.btn-warning, [color="warning"]') });

// AFTER — semantic role with name regex
.filter({ has: page.getByRole('button', { name: /warning/i }) });
```

### When to add a new testId vs use a role-based locator

- Default to role-based. Only add a new testId to `testIds.ts` when:
  - The element has no semantic role (e.g., a presentational `<div>` wrapper)
  - The role-based locator is non-unique on the page and there's no clean way to scope (e.g., 5 buttons with the same name)
  - The text is locale-dependent and using a regex would be fragile

- If a new testId IS required: add it to `tests/tests/utils/testIds.ts` AND add `data-testid={testIds.X.Y}` to the corresponding component in `apps/frontend/src/lib/**`. This is the contract per RESEARCH.md "Anti-pattern: Adding ad-hoc testids without registering."

- For raw locators that have NO clean rewrite (e.g., `[color="warning"]` looking at a CSS attribute set by component logic that has no role/text/test-id), inline-justify with `// reason: ...` + `// eslint-disable-next-line playwright/no-raw-locators` per the v2.8 P70 Cat A "Option A inline ignore-with-rationale preamble" pattern. Use this sparingly — most sites should rewrite cleanly.
