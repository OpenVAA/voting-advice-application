# Requirements: OpenVAA Idura FTN Auth

**Defined:** 2026-03-27
**Core Value:** A reliable, well-tested VAA framework that developers can confidently extend, customize, and deploy for real elections.

## v2.3 Requirements

Requirements for Idura FTN auth handler with provider abstraction. Each maps to roadmap phases.

### Provider Abstraction

- [x] **PROV-01**: Codebase supports switching between Signicat and Idura identity providers via environment variable
- [x] **PROV-02**: Provider interface defines `getAuthorizeUrl()`, `exchangeCodeForToken()`, and `getIdTokenClaims()` operations
- [x] **PROV-03**: Existing Signicat flow continues to work unchanged when selected as provider

### Authorization Flow

- [x] **AUTH-01**: Server-side endpoint constructs signed JWT Authorization Requests (JAR) for Idura
- [x] **AUTH-02**: Frontend redirects to identity provider via server-side authorize endpoint instead of client-side PKCE when using Idura
- [x] **AUTH-03**: Callback route is provider-agnostic (not hardcoded to `signicat/oidc/callback`)

### Token Exchange

- [x] **TOKN-01**: Token exchange uses `private_key_jwt` client assertion when provider is Idura
- [x] **TOKN-02**: JWE decryption supports RSA-OAEP-256 (Idura) alongside existing RSA-OAEP (Signicat)

### Edge Function

- [x] **EDGE-01**: Identity callback Edge Function supports both providers via configuration
- [x] **EDGE-02**: Idura identity matching uses `sub` claim (persistent pseudonym) instead of `birthdate`
- [x] **EDGE-03**: Provider type is stored in user `app_metadata` for audit trail

### Configuration

- [x] **CONF-01**: All Idura-specific environment variables documented in `.env.example`
- [x] **CONF-02**: Key generation process documented (signing + encryption RSA key pairs)
- [x] **CONF-03**: Server constants updated with new Idura env vars

### Testing

- [x] **TEST-01**: Unit tests cover provider abstraction layer (both Signicat and Idura paths)
- [ ] **TEST-02**: E2E preregistration tests work with provider-agnostic callback route

## Future Requirements

### Key Management

- **KEYS-01**: Key rotation strategy for signing and encryption keys
- **KEYS-02**: Automated key generation tooling

### Monitoring

- **MNTR-01**: Auth flow error tracking and alerting
- **MNTR-02**: Provider-specific metrics dashboard

## Out of Scope

| Feature | Reason |
|---------|--------|
| Signicat removal | Keep for backward compatibility — deployments may still use Signicat |
| Multi-provider simultaneous auth | Only one provider active at a time per deployment |
| Non-FTN identity schemes | Idura supports others but FTN is the only Finnish requirement |
| Production Idura contract | Business/legal step, not technical |
| Key rotation automation | Manual key management sufficient for v2.3 |
