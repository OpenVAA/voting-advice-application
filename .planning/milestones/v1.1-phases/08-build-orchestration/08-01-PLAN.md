1. **Fix FIX-01** in `packages/app-shared/package.json` line 7: Change `packagec.json` to `package.json` in the `package:esm` script. The corrected line should be:
   ```
   "package:esm": "mkdir -p ./build/esm/ && echo '{ \"type\": \"module\" }' > ./build/esm/package.json"
   ```

2. **Install Turborepo** as a root devDependency:
   ```bash
   yarn add -D turbo
   ```
   This should install turbo ^2.8.x. Verify with `yarn turbo --version`.

3. **Create `turbo.json`** at project root with this exact configuration:
   ```json
   {
     "$schema": "https://turbo.build/schema.json",
     "tasks": {
       "build": {
         "dependsOn": ["^build"],
         "outputs": ["build/**", "dist/**"],
         "inputs": [
           "src/**",
           "tsconfig.json",
           "tsconfig.*.json",
           "package.json"
         ]
       },
       "test:unit": {
         "dependsOn": ["build"],
         "cache": false
       }
     }
   }
   ```
   Key design decisions:
   - `outputs: ["build/**", "dist/**"]` covers all packages (most use build/, strapi-admin-tools uses dist/)
   - `inputs` specified to prevent cache invalidation from README edits, test file changes, etc.
   - `test:unit` cache disabled so tests always re-run
   - `dependsOn: ["^build"]` ensures topological ordering

4. **Add `.turbo` to `.gitignore`** by appending:
   ```

   # Turborepo
   .turbo
   ```

5. **Verify turbo discovers all workspaces** by running:
   ```bash
   yarn turbo run build --dry
   ```
   This should show the task graph with all workspace packages in topological order. Verify that:
   - core builds before data, matching
   - data builds before filters
   - filters builds before app-shared (via data dependency chain)
   - app-shared builds before frontend, strapi, docs, llm

6. **Verify FIX-01** by building app-shared:

   Then verify:
   ```bash
   ls packages/app-shared/build/esm/package.json
   cat packages/app-shared/build/esm/package.json
   ```
   The file should exist and contain `{ "type": "module" }`.

**Title:** Deno Compatibility Evaluation: Turborepo Impact

**Sections:**

1. **Summary** (2-3 sentences): Turborepo has no native Deno support and this is unlikely to change. The impact on a future Deno migration is minimal because Turborepo would be replaced by Deno's native task runner, not adapted to work with it.

2. **Current State** (3-4 bullets):
   - Turborepo requires Node.js package managers (reads package.json and lockfiles)
   - Deno 2.x can read package.json but it's not the native workflow
   - Deno 2.1+ is building its own monorepo task dependencies with caching (inspired by Wireit)
   - Turborepo maintainers have stated Deno support is "not immediately on the roadmap"

3. **Impact on OpenVAA** (3-4 bullets):
   - If/when migrating to Deno, Turborepo would be replaced with Deno's native task runner
   - turbo.json task definitions map cleanly to deno.json task definitions
   - Build caching concepts transfer directly
   - Investment in proper dependency graphs and caching patterns is not wasted

4. **Recommendation** (2-3 sentences): Turborepo is the right choice for now. A future Deno migration would replace Turborepo entirely rather than trying to make them coexist. The patterns established (dependency-aware builds, content-hashed caching, topological ordering) will transfer to any future build tool.

**Source references:** Cite Turborepo GitHub Discussion #3057, Issue #6368, Discussion #7454. Add a "Last reviewed" date of 2026-03-12.
