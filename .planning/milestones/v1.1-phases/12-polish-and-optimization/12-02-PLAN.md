1. Update turbo.json -- add `lint` and `typecheck` task definitions alongside existing `build` and `test:unit`:

   ```json
   "lint": {
     "dependsOn": ["^lint"],
     "outputs": [],
     "inputs": [
       "$TURBO_DEFAULT$",
       "eslint.config.*"
     ]
   },
   "typecheck": {
     "dependsOn": ["^build"],
     "outputs": [],
     "inputs": [
       "$TURBO_DEFAULT$",
       "tsconfig.json",
       "tsconfig.*.json"
     ]
   }
   ```

   Key design: `lint.dependsOn: ["^lint"]` so shared-config ESLint changes invalidate downstream. `typecheck.dependsOn: ["^build"]` because type checking needs .d.ts from built dependencies.

2. Add per-workspace scripts to each workspace package.json:

   For standard packages (core, data, matching, filters, app-shared, llm, argument-condensation, question-info):
   ```json
   "lint": "eslint --flag v10_config_lookup_from_file src/",
   "typecheck": "tsc --noEmit"
   ```

   For shared-config: Only add `"lint": "eslint --flag v10_config_lookup_from_file src/"`. For typecheck, check if shared-config has a tsconfig.json and TypeScript source -- if not, skip typecheck for this workspace (it may only export config files). If it does have TS source, add `"typecheck": "tsc --noEmit"`.

   For apps/frontend:
   ```json
   "lint": "eslint --flag v10_config_lookup_from_file src/",
   "typecheck": "svelte-kit sync && svelte-check --tsconfig ./tsconfig.json"
   ```
   Note: frontend already has a `check` script doing `svelte-kit sync && svelte-check --tsconfig ./tsconfig.json`. The new `typecheck` script should use the same command. Keep `check` as-is for backward compatibility.

   For apps/strapi:
   ```json
   "lint": "eslint --flag v10_config_lookup_from_file src/",
   "typecheck": "tsc --noEmit"
   ```
   Note: Strapi has pre-existing TS errors in generateMockData.ts. If `tsc --noEmit` fails, use `tsc --noEmit || true` temporarily and add a comment: `// TODO: Fix pre-existing TS errors in generateMockData.ts`. Alternatively, check if strapi has a tsconfig that excludes problematic files.

   For apps/docs:
   ```json
   "lint": "eslint .",
   "typecheck": "svelte-kit sync && svelte-check --tsconfig ./tsconfig.json"
   ```
   Note: docs already has `lint: "prettier --check . && eslint ."` and `check: "svelte-kit sync && svelte-check ..."`. Simplify `lint` to just `eslint .` (no --flag needed for docs if it uses flat config directly; check its eslint.config.*). Prettier is a separate concern. Keep existing `check` script. If docs has its own eslint.config that handles flat config lookup differently, use `eslint .` without the flag.

   For apps/strapi/src/plugins/openvaa-admin-tools:
   ```json
   "lint": "eslint --flag v10_config_lookup_from_file src/",
   "typecheck": "tsc -p admin/tsconfig.json --noEmit && tsc -p server/tsconfig.json --noEmit"
   ```
   Note: Per research, this plugin has separate admin and server tsconfigs. Check the actual file structure. If `admin/tsconfig.json` and `server/tsconfig.json` exist, use the compound command. If different, adapt accordingly. The plugin already has `test:ts:front` and `test:ts:back` scripts -- check what they do and mirror that pattern.

3. Do NOT modify the Claude workflows (claude.yml, claude-code-review.yml, claude-solve-issue.yml) -- they don't run turbo directly and don't need remote cache secrets. The `claude.yml` allows `yarn build:*` but the actual turbo commands execute within the action runner; remote cache is a nice-to-have there and would require exposing TURBO_TOKEN to the action, which is a security consideration for a workflow triggered by PR comments.

Note: TURBO_TOKEN and TURBO_TEAM must be configured manually by the user in GitHub repository settings:
- TURBO_TOKEN: Repository secret (Settings > Secrets > Actions > New repository secret)
- TURBO_TEAM: Repository variable (Settings > Variables > Actions > New repository variable)
The token is created at https://vercel.com/account/tokens (Scoped Access Token with appropriate team access).
These are documented as user_setup but the plan proceeds without them -- the workflows will simply skip remote caching when the vars are not set.
