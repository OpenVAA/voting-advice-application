Create the new callback server route at `apps/frontend/src/routes/api/oidc/callback/+server.ts` (per D-05, D-06, D-07).

This is an API-style server route (GET handler). The identity provider redirects the browser here after authentication. The handler exchanges the authorization code for a token, sets the id_token cookie, and redirects the browser to the preregister page. No client-side JavaScript involved.

```typescript
import { redirect } from '@sveltejs/kit';
import { getActiveProvider } from '$lib/api/utils/auth/providers';
import type { RequestEvent } from '@sveltejs/kit';
```

Implement `GET` handler with `({ url, cookies }: RequestEvent)`:

1. Extract query parameters:
   ```typescript
   const code = url.searchParams.get('code');
   const returnedState = url.searchParams.get('state');
   const errorParam = url.searchParams.get('error');
   ```

2. Handle IdP errors (user canceled, etc.):
   ```typescript
   if (errorParam) {
     throw redirect(303, '/candidate/preregister?error=' + encodeURIComponent(errorParam));
   }
   ```

3. Validate authorization code:
   ```typescript
   if (!code) {
     throw redirect(303, '/candidate/preregister?error=missing_code');
   }
   ```

4. Verify state parameter (CSRF protection). Read the stored state from the cookie:
   ```typescript
   const storedState = cookies.get('oidc_state');
   if (storedState) {
     // State was stored (Idura flow, or Signicat if we stored it)
     if (!returnedState || returnedState !== storedState) {
       cookies.delete('oidc_state', { path: '/' });
       cookies.delete('oidc_nonce', { path: '/' });
       throw redirect(303, '/candidate/preregister?error=invalid_state');
     }
     // Clean up state cookie
     cookies.delete('oidc_state', { path: '/' });
   }
   // If no stored state, skip verification (backward compat with Signicat PKCE which stored code_verifier in localStorage)

5. Read the code_verifier from cookie (for Signicat PKCE flow). The preregister page will store it in a cookie instead of localStorage when using the new callback path:
   ```typescript
   const codeVerifier = cookies.get('oidc_code_verifier');
   if (codeVerifier) {
     cookies.delete('oidc_code_verifier', { path: '/' });
   }
   ```

6. Get the active provider and build redirect URI:
   ```typescript
   const provider = getActiveProvider();
   const redirectUri = url.origin + url.pathname;
   ```

7. Exchange code for token:
   ```typescript
   const { idToken } = await provider.exchangeCodeForToken({
     authorizationCode: code,
     redirectUri,
     codeVerifier
   });
   ```

8. Verify the token:
   ```typescript
   const claims = await provider.getIdTokenClaims(idToken);
   if (!claims.success) {
     throw redirect(303, '/candidate/preregister?error=invalid_token');
   }
   ```

9. Optionally verify nonce (if stored):
   ```typescript
   const storedNonce = cookies.get('oidc_nonce');
   if (storedNonce) {
     cookies.delete('oidc_nonce', { path: '/' });
     // Nonce verification is an additional check -- for now we store it
     // and it will be verified against the id_token in a future enhancement
   }
   ```

10. Set the id_token cookie (same pattern as the token route):
    ```typescript
    cookies.set('id_token', idToken, {
      httpOnly: true,
      secure: true,
      sameSite: 'strict',
      path: '/'
    });
    ```

11. Redirect to the preregister page:
    ```typescript
    throw redirect(303, '/candidate/preregister');
    ```

Wrap steps 6-10 in try/catch. On error, redirect to preregister with error:
```typescript
throw redirect(303, '/candidate/preregister?error=token_exchange_failed');
```

IMPORTANT: In SvelteKit server routes, you must `throw redirect()`, not `return redirect()`. The redirect is thrown as an exception that SvelteKit catches and converts to an HTTP 303 response.

   ```typescript
   async function redirectToIdentityProvider() {
     if (!browser) return;

     const redirectUri = `${window.location.origin}${$getRoute('CandAppPreregisterIdentityProviderCallback')}`;

     if (constants.PUBLIC_IDENTITY_PROVIDER_TYPE === 'idura') {
       // Idura: call server-side authorize endpoint for JAR construction (per D-12)
       const response = await fetch('/api/oidc/authorize', {
         method: 'POST',
         headers: { 'Content-Type': 'application/json' },
         body: JSON.stringify({ redirectUri })
       });

       if (!response.ok) {
         console.error('Failed to get authorization URL');
         return;
       }

       const { authorizeUrl } = await response.json();
       window.location.href = authorizeUrl;
     } else {
       // Signicat: client-side PKCE redirect (per D-12)
       const { codeVerifier, codeChallenge } = await generateChallenge(window.crypto);

       // Store code_verifier in a cookie via the authorize endpoint so the
       // callback server route can access it (localStorage is client-only).
       // We use the authorize endpoint to set the cookie server-side.
       const response = await fetch('/api/oidc/authorize', {
         method: 'POST',
         headers: { 'Content-Type': 'application/json' },
         body: JSON.stringify({ redirectUri, codeChallenge })
       });

       if (!response.ok) {
         console.error('Failed to get authorization URL');
         return;
       }

       const { authorizeUrl } = await response.json();

       // Store code_verifier in cookie for the callback route to read
       document.cookie = `oidc_code_verifier=${codeVerifier}; path=/; max-age=600; secure; samesite=lax`;

       window.location.href = authorizeUrl;
     }
   }
   ```

5. The `constants` import from `$lib/utils/constants` is already present and already includes `PUBLIC_IDENTITY_PROVIDER_TYPE` (added in Phase 45).

NOTE on Signicat flow change: The Signicat flow now also calls `/api/oidc/authorize` to get the authorize URL from the provider abstraction (which returns the PKCE URL with `clientSideRedirect: true`). The authorize endpoint also stores the state cookie if the provider returns one. This means both Idura and Signicat use `/api/oidc/authorize` -> redirect -> `/api/oidc/callback`, making the flow truly provider-agnostic.
