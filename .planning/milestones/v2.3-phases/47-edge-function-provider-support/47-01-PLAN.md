<objective>
Create a provider-agnostic identity callback Edge Function that replaces the Signicat-specific
`signicat-callback` function. The new `identity-callback` function uses a configuration-driven
approach to support both Signicat and Idura providers: a `PROVIDER_CONFIGS` object maps provider
type to claim property names (identity match, first name, last name, extra claims), eliminating
provider-specific branching throughout the function.

Purpose: Enable deployments to use either Signicat or Idura for Finnish Trust Network bank
authentication by switching an env var, with full audit trail in user app_metadata.

Output: `identity-callback/index.ts` Edge Function, deleted `signicat-callback/` directory,
updated frontend caller.
</objective>

**1. Provider config object at the top (per D-03, D-04, D-07, D-08):**

```typescript
interface ProviderConfig {
  identityMatchProp: string;
  firstNameProp: string;
  lastNameProp: string;
  extractClaims: string[];
}

const PROVIDER_CONFIGS: Record<string, ProviderConfig> = {
  signicat: {
    identityMatchProp: 'birthdate',
    firstNameProp: 'given_name',
    lastNameProp: 'family_name',
    extractClaims: []
  },
  idura: {
    identityMatchProp: 'sub',
    firstNameProp: 'given_name',
    lastNameProp: 'family_name',
    extractClaims: ['birthdate', 'hetu']
  }
};
```

Read provider type early in the request handler:
```typescript
const providerType = Deno.env.get('IDENTITY_PROVIDER_TYPE') ?? 'signicat';
const config = PROVIDER_CONFIGS[providerType];
if (!config) {
  return new Response(JSON.stringify({ error: `Unknown identity provider type: ${providerType}` }), {
    status: 500, headers: { ...corsHeaders, 'Content-Type': 'application/json' }
  });
}
```

**2. Rename env vars (per D-11, D-12):**
- `SIGNICAT_DECRYPTION_JWKS` -> `IDENTITY_PROVIDER_DECRYPTION_JWKS` (in `decryptJweToken`)
- `SIGNICAT_JWKS_URI` -> `IDENTITY_PROVIDER_JWKS_URI` (in `verifyJwt`)
- `SIGNICAT_CLIENT_ID` -> `IDENTITY_PROVIDER_CLIENT_ID` (in `verifyJwt`)

**3. Generalize `extractIdentityClaims` (per D-04, D-07, D-08):**

Replace the current hardcoded function with a config-driven version. The function signature becomes:
```typescript
function extractIdentityClaims(
  payload: jose.JWTPayload,
  config: ProviderConfig
): {
  firstName: string;
  lastName: string;
  identityMatchValue: string;
  extractedClaims: Record<string, unknown>;
}
```

It reads `payload[config.identityMatchProp]`, `payload[config.firstNameProp]`,
`payload[config.lastNameProp]`. Throws with a descriptive error listing which props are
present/missing (include the prop names from config in the error message). Loops over
`config.extractClaims` to build the `extractedClaims` object.

**4. Generalize `findUserByBirthdateId` -> `findUserByIdentityMatch` (per D-05):**

Rename the function. Instead of searching `u.app_metadata?.birthdate_id === birthdateId`,
search `u.app_metadata?.identity_match_value === identityMatchValue`. Same pagination logic.
Do NOT also search for `birthdate_id` -- per D-06, this is a clean break.

**5. Update user creation app_metadata (per D-09, D-10):**

Replace the current:
```typescript

```

With:
```typescript
app_metadata: {
  identity_provider: providerType,

  identity_match_value: identityMatchValue,
  ...extractedClaims
}
```

The `user_metadata` stays the same shape but uses the config-driven names:
```typescript
user_metadata: { given_name: firstName, family_name: lastName }
```
