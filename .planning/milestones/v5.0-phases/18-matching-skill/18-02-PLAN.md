**# Matching Algorithm Reference** (title)

Opening line: "Detailed reference for the `@openvaa/matching` algorithm internals. For paradigm and process overview, read `packages/matching/README.md`."

**## Matching Pipeline** (~30 lines)
Document the full pipeline that `MatchingAlgorithm.match()` follows:

1. Receive questions, reference (voter), targets (candidates), and optional MatchingOptions
2. Filter questions to only those the reference has answered (non-MISSING_VALUE in reference.answers)
3. Create MatchingSpace via `MatchingSpace.fromQuestions(questions, questionWeights?)` -- derives shape from each question's `normalizedDimensions`, weights from `questionWeights` (default all equal)
4. Project all entities to Positions by calling `question.normalizeValue(answer)` for each question. Coordinates are flattened into the MatchingSpace shape. Result is a Position object per entity.
5. Optionally apply `MatchingSpaceProjector.project()` to transform Positions to a lower-dimensional space (e.g., 2D political compass)
6. For each target, call `measureDistance(reference, target, space, metric, missingValueOptions)`:
   a. `imputeMissingPosition()` replaces MISSING_VALUE coordinates in target based on reference coordinates and chosen method
   b. Distance metric function computes normalized global distance
   c. If questionGroups provided, also compute SubMatch distances via `createSubspace()` for each group
7. Wrap results in `Match<TTarget>` objects containing: entity reference, global distance, and optional `subMatches` array
8. Sort matches by ascending distance (best match = lowest distance = first)

Source: `algorithms/matchingAlgorithm.ts` match() and projectToNormalizedSpace() methods.

**## Distance Metrics** (~50 lines)
Document each built-in metric with its decomposition.

**Generic distance() function** (source: `distance/metric.ts`):
- Takes two Position objects plus kernel/sum/subdimWeight functions
- Iterates dimension-by-dimension (accounting for subdimensions via shape)
- For each dimension: `kernelResult = kernel(a_coord, b_coord)`, weighted by dimension weight
- For subdimensions within a dimension: weight each by `subdimWeight(numSubdims)`
- Aggregates via `sum()`, then normalizes: `(sum / maxPossibleSum) * COORDINATE.Extent`
- Result range: `[0, COORDINATE.Extent]` (i.e., `[0, 1]`)

**Manhattan** (`DISTANCE_METRIC.Manhattan`):
- kernel: `absoluteKernel(a, b)` = `Math.abs(a - b)` -- range `[0, Extent]`
- sum: `basicSum(values)` = simple addition
- subdimWeight: `basicDivision(n)` = `1/n`
- Behavior: Standard absolute difference. Two identical answers = 0, opposite extremes = Extent.

**Directional** (`DISTANCE_METRIC.Directional`):
- kernel: `directionalKernel(a, b)` = `0.5 * Extent - (2 * (a - Neutral) * (b - Neutral)) / Extent` (Mendez 2017)
- sum: `basicSum(values)` = simple addition
- subdimWeight: `basicDivision(n)` = `1/n`
- Behavior: Treats neutral (0) as "uncertain." Two neutral answers yield distance `0.5 * Extent` (50% disagreement), not 0. Two identical non-neutral answers yield 0.

**Euclidean** (`DISTANCE_METRIC.Euclidean`):
- kernel: `absoluteKernel(a, b)` = `Math.abs(a - b)` -- same as Manhattan
- sum: `euclideanSum(values)` = `Math.sqrt(sum of squares)`
- subdimWeight: `(n) => 1 / Math.sqrt(n)` -- ensures equal max distances across dimension shapes
- Behavior: Geometric distance. More sensitive to large single-dimension differences than Manhattan.

**## MatchingSpace and Position** (~25 lines)

**MatchingSpace** (source: `space/matchingSpace.ts`):
- `shape`: Array of numbers where each element is the number of subdimensions for that question. OrdinalQuestion contributes `1`, CategoricalQuestion with N>2 choices contributes `N`.
- `weights`: Array of numbers (same length as shape), default all `1`. Set via `questionWeights` option.
- `fromQuestions(questions, weights?)`: Factory that builds shape from `question.normalizedDimensions` and validates weights length.

**Position** (source: `space/position.ts`):
- Wraps coordinates (flat `CoordinateOrMissing[]` array) within a MatchingSpace
- Constructor validates that `coordinatesShape(coordinates)` equals `space.shape`
- Coordinates are accessed via `flatten()` (flat array) or `reshape()` (nested by dimension)
- MISSING_VALUE coordinates are valid -- imputed later during distance measurement

**Shape utilities** (source: `space/shape.ts`):
- `flatten(nested)`: Nested coordinate arrays to flat array
- `reshape(flat, shape)`: Flat array to nested based on shape
- `coordinatesShape(coordinates)`: Infer shape from coordinate array structure
- `equalShapes(a, b)`: Compare two shapes for equality

**## Missing Value Imputation** (~15 lines)

Source: `missingValue/impute.ts`

`MISSING_VALUE_METHOD.Neutral`:
- Imputes `COORDINATE.Neutral` (0) for missing coordinates
- Assumes the candidate has a moderate/middle position

`MISSING_VALUE_METHOD.RelativeMaximum`:
- Imputes the coordinate furthest from the reference's coordinate
- If reference is `COORDINATE.Min` (-0.5), impute `COORDINATE.Max` (0.5) and vice versa
- If reference is `COORDINATE.Neutral` (0), uses `MISSING_VALUE_BIAS` (Positive or Negative) to choose direction
- Worst-case assumption: maximizes distance for unanswered questions

`imputeMissingPosition(position, reference, space, method, bias?)`:
- Creates a NEW Position with MISSING_VALUE coordinates replaced
- Only imputes target coordinates -- reference missing values should already be excluded at the question filtering step

**## Match Object Structure** (~25 lines)

**MatchBase** (source: `match/matchBase.ts`):
- `distance`: Normalized distance `[0, COORDINATE.Extent]` (0 = perfect match)
- `matchFraction`: `(COORDINATE.Extent - distance) / COORDINATE.Extent` (0 = worst, 1 = best)
- `score`: `Math.round(matchFraction * MatchBase.multiplier)` (0 = worst, 100 = best with default multiplier of 100)
- `toString()`: Returns `score + '%'`
- `static multiplier`: Default 100, can be overridden for different score scales

**Match<TTarget, TGroup>** (source: `match/match.ts`):
- Extends `MatchBase`
- `entity`: Reference to the matched target entity (TTarget)
- `subMatches`: Array of `SubMatch<TGroup>` objects (empty if no questionGroups provided)

**SubMatch<TGroup>** (source: `match/subMatch.ts`):
- Extends `MatchBase`
- `questionGroup`: Reference to the `MatchableQuestionGroup` (TGroup) this submatch is for
- Computed via `createSubspace()` which zeros out weights for questions not in the group

**MATCH_TYPE discriminator** (source: `match/matchTypes.ts`):
- `MATCH_TYPE.Match` = `'match'`
- `MATCH_TYPE.SubMatch` = `'subMatch'`
- Type guards: `isMatch(obj)`, `isSubMatch(obj)`, `isMatchBase(obj)`, `isMatchType(obj, type)`

**## Question Types (Matching Side)** (~20 lines)

Document the standalone matching question implementations (not data package question classes):

**OrdinalQuestion** (source: `question/ordinalQuestion.ts`):
- For Likert-scale questions with ordered numeric choices
- `normalizedDimensions` = `1`
- `normalizeValue(value)`: Maps choice value to `[COORDINATE.Min, COORDINATE.Max]` using linear interpolation: `COORDINATE.Min + (value - min) / (max - min) * COORDINATE.Extent`
- Constructor takes: `{ id, values: {min, max, ...} }`

**CategoricalQuestion** (source: `question/categoricalQuestion.ts`):
- For nominal (unordered) questions with named choices
- Binary (2 choices): `normalizedDimensions` = `1`, normalizes first choice to `COORDINATE.Min`, second to `COORDINATE.Max`
- N>2 choices: `normalizedDimensions` = `N`, normalizes to one-hot array: selected choice = `COORDINATE.Max`, all others = `COORDINATE.Min`
- Constructor takes: `{ id, values: {choices: string[]} }`

**MatchableQuestionGroup** (source: `question/matchableQuestionGroup.ts`):
- Interface with `label` (string) and `matchableQuestions` (MatchableQuestion[])
- Used by `MatchingOptions.questionGroups` to define SubMatch categories (e.g., "Economy", "Environment")

**Note:** These are standalone implementations for testing and direct use. In production with @openvaa/data, question types come from `DataRoot` (e.g., `SingleChoiceOrdinalQuestion`, `SingleChoiceCategoricalQuestion`).

**# Extension Patterns for @openvaa/matching** (title)

Opening line: "Step-by-step guides for the most common matching package extensions. All file paths in steps are relative to `packages/matching/src/` unless stated otherwise."

**## Adding a New Distance Metric** (~50 lines)
Reference implementation: Manhattan metric in `distance/metric.ts`.

Steps:

1. **Implement kernel function** in `distance/metric.ts`
   - Signature: `(a: number, b: number) => number`
   - Computes per-coordinate distance between two normalized coordinates
   - Must return a value in `[0, COORDINATE.Extent]` range
   - Reference: `absoluteKernel` for standard absolute difference, `directionalKernel` for direction-aware

2. **Implement sum function** in `distance/metric.ts` (only if different from existing sums)
   - Signature: `(values: number[]) => number`
   - Aggregates per-dimension distances into a single value
   - Reference: `basicSum` for linear addition, `euclideanSum` for `Math.sqrt(sum of squares)`

3. **Implement subdimWeight function** in `distance/metric.ts` (only if different from existing)
   - Signature: `(n: number) => number`
   - Returns weight factor for subdimensions within a multi-subdimension question
   - Reference: `basicDivision` returns `1/n`, Euclidean uses `1/Math.sqrt(n)`

4. **Create named metric function** in `distance/metric.ts`
   - Follows the pattern of `manhattan()`, `directional()`, `euclidean()` functions
   - Calls the generic `distance()` function passing the three component functions:
     ```
     distance({ a, b, kernel: myKernel, sum: mySum, subdimWeight: mySubdimWeight, space, allowMissing })
     ```
   - Must accept the same parameters: `{ a, b, space, allowMissing? }`

5. **Add to DISTANCE_METRIC const** in `distance/metric.ts`
   - Add `{Name}: {nameFunction}` to the `DISTANCE_METRIC` const object
   - The key becomes the enum-like accessor: `DISTANCE_METRIC.{Name}`
   - The value is the metric function itself

6. **Add to module barrel** in `distance/index.ts`
   - Verify `export *` from `'./metric'` already covers the new export (it should since DISTANCE_METRIC is already exported)
   - No changes needed if the new metric is only added as a property of DISTANCE_METRIC

7. **Create tests** in `tests/distance.test.ts` (relative to `packages/matching/`)
   - Test with single-dimension positions (normalizedDimensions=1)
   - Test with multi-subdimension positions (CategoricalQuestion-style)
   - Test edge cases: all-same answers (distance=0), all-opposite answers (distance=COORDINATE.Extent), single-question matching
   - Test with weighted dimensions
   - Pattern: follow existing Manhattan/Directional/Euclidean test blocks

**## Adding a New MatchingSpaceProjector** (~25 lines)
The `MatchingSpaceProjector` interface enables projecting positions from the full matching space to a lower-dimensional space (e.g., 2D political compass maps).

Reference: `algorithms/matchingSpaceProjector.ts` defines the interface.

   - Output: a new (lower-dimensional) MatchingSpace and projected Positions
   - The projected MatchingSpace shape defines the target dimensions (e.g., `[1, 1]` for a 2D map)

   - Test round-trip properties if applicable
