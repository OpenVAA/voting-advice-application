# Translations

Internationalization is handled by [Paraglide JS](https://inlang.com/m/gerre34r/library-inlang-paraglideJs) (inlang). Paraglide compiles translation messages at build time into tree-shakable JavaScript functions, providing type safety and zero runtime overhead for static translations.

Translations are accessed using the `t()` wrapper function, e.g. `t('translation.key', {variable: value})`. Note: `t` is a plain function, **not** a Svelte store -- use `t(...)` directly, not `$t(...)`.

## Architecture

### Wrapper pattern

The `t()` function in `wrapper.ts` implements a two-tier lookup:

1. **Runtime overrides** (`overrides.ts`): Checks for backend `translationOverrides` loaded from `appCustomization`. This allows each VAA deployment to customize labels without rebuilding.
2. **Paraglide compiled messages**: Falls back to the statically compiled message functions generated by Paraglide's Vite plugin.

### Key files

- `wrapper.ts` -- The `t()` function: override check then Paraglide fallback
- `overrides.ts` -- Runtime override store (`setOverrides`, `getOverride`, `clearOverrides`)
- `init.ts` -- Re-exports `t`, `getLocale`, `setLocale`, `locales`, `translate`, `translateObject`
- `utils/assertTranslationKey.ts` -- Type assertion for dynamically constructed keys

### Generated output

The Paraglide Vite plugin compiles messages from `messages/` into `src/lib/paraglide/` on build/dev. This directory is gitignored and should not be committed.

## Translation sources

1. **Static message files** in `messages/` directory (see next section) -- compiled by Paraglide at build time
2. **Backend translation overrides** loaded from `appCustomization` via `DataProvider`
   - Applied via `setOverrides()` in the root layout loader
   - By default these include only translations listed in `dynamic.json` but arbitrary ones may also be provided
   - Overrides use ICU message format parsed by `intl-messageformat` at runtime

## Message files

Translations are stored in the `messages/` directory at the frontend root, organized by locale:

```
messages/
  en/
    about.json
    common.json
    dynamic.json
    ...
  fi/
    about.json
    common.json
    dynamic.json
    ...
```

Each locale directory must contain the same set of JSON files with matching keys. This is verified by the translation structure tests.

### Message format

Messages use the [inlang message format](https://inlang.com/m/reootnfj/plugin-inlang-messageFormat):

- **Simple strings**: `"Hello world"`
- **Variables**: `"Hello {name}"`
- **Plurals**: Use inlang variant syntax with `declarations`, `selectors`, and `match` arrays (NOT ICU inline syntax)

### File organization

- `dynamic.json` -- Default values for translations typically customized per deployment. These should be overridden by backend data.
- `common.json` -- Common terms used throughout the app and sometimes as part of other translations.
- `components.json` -- Translations used by static components, grouped by component name.
- One file each for each page and dynamic component. Files for the Candidate App are prefixed like `candidateApp.basicInfo.json`.
- `lang.json` -- Locale display names (e.g., `"en": "English"`, `"fi": "suomi"`).

### Special characters

Do not use special characters like `'\n'` in translations -- their backslashes will be escaped during compilation.

## Translation key assertions

The `assertTranslationKey()` utility provides runtime validation for dynamically constructed keys. For static keys, Paraglide provides compile-time type safety through its generated message functions.

## Managing translations

For bulk translation management, use the [`editTranslations`](/frontend/tools/editTranslations/editTranslations.ts) tool. It can export, import, and replace translation keys across the source code.

NB! Automatic key replacement only covers `.svelte` files in the frontend. Make sure to check that translations accessed by [e2e tests](/tests/) and other non-Svelte code are also updated accordingly.
